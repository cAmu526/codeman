# CodeMan Orchestrator — 主编排入口

## 宿主环境说明

CodeMan 支持 Cursor、Claude Code、OpenCode 和 Trae 四种宿主环境。**CODEMAN_HOME** 的路径规则：
- Cursor：`~/.cursor/skills/.codeman`
- Claude Code：`~/.claude/skills/.codeman`
- OpenCode：`~/.claude/skills/.codeman`（通过 Claude Code 兼容层读取）
- Trae：`~/.trae/skills/.codeman`

在执行任何需要读取模板文件的步骤时，请根据当前运行环境自动选择正确的 CODEMAN_HOME 路径。如无法判断，优先尝试 `~/.claude/skills/.codeman`，其次 `~/.cursor/skills/.codeman`，再次 `~/.trae/skills/.codeman`。

---

## 概述

你是 CodeMan 工作流框架的主编排器。用户唤起 CodeMan 后，你负责：
1. 读取项目状态
2. 识别任务类型和规模
3. 选择合适的工作流通道
4. 向用户确认后调度对应 Skill 链

## 执行步骤

### Step 1: 读取项目状态

```
读取 .codeman/docs/STATUS.md
  → 如果文件不存在：判定为"需要初始化"场景
  → 如果文件存在：解析当前 phase、current_module、待处理事项
```

```
读取 .codeman/docs/DIRECTIVES.md
  → 加载关键指令和约束
  → 如果文件不存在：跳过，后续初始化时创建
```

**外部代码变更检测（自动，仅对工作类命令触发）：**

当用户执行工作类命令（继续、开始开发、新需求）时，在进入正式流程前自动检测是否有外部代码变更需要同步文档：

```bash
# 从 STATUS.md 的 last_updated 时间获取上次更新时间
# 检查此时间之后是否有他人的提交
git log --since="{last_updated}" --not --author="{当前 git 用户}" --oneline
```

- 无新提交 → 跳过，正常继续
- 有新提交 → 自动触发「增量文档补全」（见下方），完成后继续原本的工作流
- 非 git 仓库或无 git 记录 → 跳过（用户可手动 `CodeMan 同步`）

> 查看状态（`CodeMan 状态`/`CodeMan 概览`）不触发补全，只提示："检测到 {N} 个外部提交尚未同步文档，建议执行 `CodeMan 同步`"。

### Step 2: 分析用户输入，判断场景

根据 STATUS.md 内容和用户输入，判断以下场景之一：

**场景零：项目初始化（最高优先级）**
- 触发条件：用户说"CodeMan 初始化"，或 .codeman/ 目录不存在
- 用户说：CodeMan 初始化、初始化项目、init
- 执行：见下方「场景零：项目初始化流程」

**场景一：全新项目开发**
- 触发条件：.codeman/ 已初始化，STATUS.md 的 phase = "init"，且项目目录中无业务代码
- 用户说：CodeMan 开始开发、新项目、从零开始

**场景二：版本迭代/需求变更**
- 触发条件：STATUS.md 存在且 phase = "deploy-ready" 或 "completed"，或用户明确提到新需求/迭代
- 用户说：新需求、版本迭代、增加功能、修改需求

**场景三：旧项目接入**
- 触发条件：.codeman/ 已初始化，但项目中有已有业务代码，且 STATUS.md 的 phase = "init"
- 用户说：接入现有项目、已有代码、老项目
- 也可由场景零自动触发（init.sh 检测到有代码后自动进入）

**场景四：轻量快速修改（快速通道）**
- 触发条件：满足下方"轻量任务判定规则"的全部条件
- 用户说：修复、小改动、样式调整、Bug 修复

**场景五：批量迭代（混合新功能 + Bug 修复 + 优化）**
- 触发条件：用户说"迭代"、"本周迭代"、"这周要做"，且输入包含多条工作项
- 用户说：CodeMan 迭代：[混合列表]
- 执行：见下方「场景五：批量迭代」

**场景六：外部交接接入（从其他工具/流程中途接手）**
- 触发条件：用户说"CodeMan 接手"、"接手测试"、"接手开发"、"从X阶段开始"，或用户明确表示已有外部文档和代码想让 CodeMan 从某个阶段继续
- 用户说：CodeMan 接手、接手、从测试开始、帮我测试、接手后续工作
- 执行：见下方「场景六：外部交接接入」

**特殊：断点续做**
- 触发条件：STATUS.md 存在且 phase 不是 "completed"，且用户说"继续"
- 优先级高于其他场景判断

**特殊：文档同步**
- 触发条件：用户说"同步"、"同步文档"、"CodeMan 同步"
- 执行：见下方「增量文档补全」流程
- 适用场景：同事直接给了代码文件、非 git 管理的代码变更、手动触发同步

**特殊：添加项目规则**
- 触发条件：用户说"添加规则"、"创建规范"、"写一条规则"、"CodeMan 添加规则：[描述]"
- 执行：调度自进化 Skill（Read `{CODEMAN_HOME}/skills/evolve/SKILL.md`），以「用户手动创建」模式执行
- 流程：根据用户描述生成 `.codeman/rules/proj-*.mdc` 文件 → 同步到 IDE 规则目录 → 更新 INDEX.md

**特殊：查看状态**
- 触发条件：用户说"状态"、"进度"、"查看"
- 直接输出 STATUS.md 摘要，不启动工作流

**特殊：项目概览**
- 触发条件：用户说"概览"、"项目概览"、"CodeMan 概览"、"项目总览"
- 执行：见下方「特殊：生成/更新项目概览」

**特殊：注册/管理外部 Skill**
- 触发条件：用户说"注册 Skill"、"挂载 Skill"、"添加外部工具"、"把 XX 挂到 XX 阶段"、"移除外部 Skill"、"查看已注册 Skill"
- 执行：见下方「外部 Skill 管理流程」

**特殊：框架升级**
- 触发条件：用户说"CodeMan 升级"、"升级 CodeMan"、"升级框架"、"更新 CodeMan"
- 执行：见下方「CodeMan 升级流程」

### Step 3: 轻量任务判定规则

**必须全部满足才判定为轻量任务：**

| # | 条件 | 说明 |
|---|------|------|
| 1 | **不涉及业务代码逻辑** | **首要硬性条件**。仅限纯文案、纯样式（CSS/className/Tailwind token）、纯注释、文档、依赖版本号、常量/变量重命名等非逻辑改动。任何会影响运行时行为的改动（条件分支、循环、计算、状态流转、校验规则、权限判断、数据处理流程、错误处理分支等）即视为业务逻辑改动，**必须走完整流程** |
| 2 | 预估影响文件数 ≤ 3 | 修改范围小 |
| 3 | 不涉及架构变更 | 不新增模块或重构模块边界 |
| 4 | 不新增 API 端点 | 不涉及接口契约变更 |
| 5 | 不涉及数据模型变更 | 不修改表结构或核心数据结构 |
| 6 | 不涉及多模块联动 | 局限在单一模块内 |
| 7 | 用户未明确标注为复杂任务 | 尊重用户判断 |

**✅ 可走快速通道的典型场景：**
- UI 文案调整（按钮文字、错误提示、空态文案）
- 纯样式微调（间距、颜色、字号、响应式断点）
- 注释/文档补全
- README、CHANGELOG 更新
- 依赖版本号 bump（不涉及 API 变更）
- 无逻辑影响的变量/文件/目录重命名

**❌ 必须走完整流程的典型场景（哪怕只改 1 行 1 个文件）：**
- 修复状态判断 Bug（条件写反、少写分支）
- 改金额/佣金/折扣/税率计算
- 修改权限/角色校验
- 修复数据过滤/排序/聚合条件
- 调整业务流程分支
- 修改 API 请求/响应处理
- 任何 try/catch 分支的新增或调整

**误判防护：**
- 快速通道执行过程中，一旦发现涉及业务逻辑，立即暂停并切换完整流程
- 判定不确定时必须问用户（不确定即停原则）
- 用户有最终决定权：任何时候用户说"切换完整流程"立即切换

### Step 4: 向用户确认

输出以下格式的确认信息：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CodeMan 场景识别
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
识别结果：[场景名称]
当前状态：[STATUS.md 摘要，如无则显示"新项目"]
即将执行：[工作流步骤概述]
预估轮次：[X-Y 轮]
预估 Token：[~XK]
是否需要生成本次迭代的版本文档？（回复版本名如"v1.2.0"，不需要则留空）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认继续？（直接回复即可，或说"切换到[其他场景]"）
```

**⚠️ 强制：仅当识别结果为「场景四：轻量快速修改」或场景五中任何工作项被判定为快速通道时，必须在同一条消息里追加以下二次确认块，未得到用户明确"确认非业务逻辑"答复前，禁止进入 Step 5：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 快速通道适用性确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
快速通道会跳过：需求分析 / 技术方案 / 测试用例设计 三个阶段，
只做：定位 → 改动 → 单测+快速回归。

AI 对本次改动的分析：
  - 预估影响文件：[文件清单]
  - 改动性质分类：[文案 / 样式 / 注释 / 文档 / 依赖号 / 重命名 / 其他]
  - 是否触及业务逻辑：[否 / 可能 / 是]
  - 判定依据：[一句话说明为什么不涉及业务逻辑]

请明确回答：本次改动是否【仅限非业务逻辑】改动？
  ✅ 回答"是"或"确认非业务逻辑" → 执行快速通道
  ❌ 回答"否"或"涉及业务逻辑"     → 切换到完整流程
  ❓ 回答"不确定"                  → 切换到完整流程（稳妥优先）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

若 AI 自己分析「是否触及业务逻辑」结果为 **"是"** 或 **"可能"**，应当直接推荐切换到完整流程，不再询问走快速通道。

**版本文档处理：**

如果用户回复了版本名：
1. 创建目录：`.codeman/docs/iterations/{版本名}/`
2. 创建 `SUMMARY.md` 骨架（版本名、日期、场景类型）
3. 创建空的 `PRD.md` 和 `DESIGN.md`（后续 Skill 写碎片文件时同步写入）
4. 更新 `.codeman/docs/iterations/INDEX.md`（追加行）
5. 在 STATUS.md 中写入 `iteration_version: {版本名}`，后续 Skill 据此判断是否同步写入版本文档

如果用户未提供版本名，不创建版本文档目录，STATUS.md 中不写入 `iteration_version`。

### Step 5: 调度对应 Skill 链

用户确认后，按以下顺序调度：

**Skill 调度机制（跨 IDE 通用，必须遵守）：**

调度任何 Skill 时，必须通过 **Read 文件** 的方式加载并执行，不依赖斜杠命令或其他 IDE 特有机制。

具体操作：读取 `{CODEMAN_HOME}/skills/{skill-name}/SKILL.md`，然后按其指令执行。

Skill 名称与路径映射表：

| Skill 名称 | 目录名 | 路径 |
|------------|--------|------|
| 需求分析 Skill | requirements | `{CODEMAN_HOME}/skills/requirements/SKILL.md` |
| 技术方案 Skill | design | `{CODEMAN_HOME}/skills/design/SKILL.md` |
| 开发实现 Skill | development | `{CODEMAN_HOME}/skills/development/SKILL.md` |
| 测试验证 Skill | testing | `{CODEMAN_HOME}/skills/testing/SKILL.md` |
| Review Skill | review | `{CODEMAN_HOME}/skills/review/SKILL.md` |
| 修复闭环 Skill | fix | `{CODEMAN_HOME}/skills/fix/SKILL.md` |
| 部署清单 Skill | deploy | `{CODEMAN_HOME}/skills/deploy/SKILL.md` |
| 自进化引擎 Skill | evolve | `{CODEMAN_HOME}/skills/evolve/SKILL.md` |

> 示例：调度"需求分析 Skill"时，实际执行 `Read {CODEMAN_HOME}/skills/requirements/SKILL.md` 并按其内容执行。
> CODEMAN_HOME 的值见本文件顶部「宿主环境说明」。

**外部 Skill 调度机制：**

读取 `.codeman/config.yaml` 的 `external_skills` 配置。如果配置不为空，在调度内部 Skill 时遵循以下规则：

1. **进入任何阶段的任何 Step 前后**，检查是否有外部 Skill 挂载在当前位置
   - 匹配规则：`hook` 字段格式为 `{阶段名}.before_step{N}`、`{阶段名}.after_step{N}` 或 `{阶段名}.execution`（替换该阶段的执行逻辑，由阶段 Skill 自行检测和处理）
   - 阶段名对应内部 Skill 目录名：requirements / design / development / testing / review / fix / deploy
2. **匹配到时**，按 `order` 升序依次执行：
   - 检查 `path` 文件是否存在。不存在则跳过并提示用户（"外部 Skill {name} 未找到，已跳过"）
   - 检查 `confirm` 字段（默认 `true`）：
     - `confirm: true`：询问用户是否执行（"检测到外部 Skill「{name}」— {description}，是否执行？（y/跳过）"），用户跳过则不执行，继续下一个
     - `confirm: false`：静默执行，不询问
   - Read 该 Skill 文件并按其指令执行
   - 执行完毕后，产出写入 `output` 指定的路径（外部 Skill 自行写入，CodeMan 不干预写入过程）
3. **产出对接**：后续内部 Skill 的前置条件中，如需外部 Skill 产出，从 `output` 路径读取
4. **无配置时**：`external_skills` 为空或不存在时，行为与当前完全一致，无任何影响
5. **execution 类型 hook**：与 before/after 不同，`{阶段名}.execution` hook 不由 orchestrator 在 Step 前后自动检查。它由对应阶段的 Skill 在执行策略选择时主动读取 config.yaml 并检测。当匹配到 execution hook 时，该 Skill 将执行逻辑整体托管给外部 Skill，完成后跳过被替代的内部步骤。

**所有场景通用规则：**

每次阶段转换（S{N} → S{N+1}）时，重新读取 DIRECTIVES.md，
执行"周期性上下文回顾协议"，输出回顾摘要后再调度下一个 Skill。

**场景零（项目初始化）：**
```
执行 init.sh → 读取输出摘要 → 判断项目类型
  → 新项目（无业务代码）：直接进入场景一
  → 旧项目（有业务代码）：直接进入场景三
```

**场景一（全新项目）：**
```
S0: 初始化 → S1: 需求分析 Skill → S2: 技术方案 Skill
→ S3: 测试验证 Skill 阶段一（生成测试用例，编码前锁定验收基准）
→ S4: 创建 feat/ 分支 → 开发实现 Skill（checkpoint commit）
→ S5: 测试验证 Skill 阶段二（编写脚本 + 执行）→ 通过后 squash merge 回主分支
→ S6: 修复闭环 Skill（如有失败）→ S7: 部署清单 Skill
```

**场景二（版本迭代 / 新需求）：**
```
S1: 变更范围识别
→ S1.5: 骨架模块前置补全（如有骨架模块被波及）
→ S2: 增量需求分析 Skill
→ S3: 增量技术方案 Skill
→ S4: 测试验证 Skill 阶段一（生成测试用例 + 受影响老功能回归用例，编码前锁定）
→ S5: 创建 feat/ 分支 → 增量开发实现 Skill（checkpoint commit）
→ S6: 测试验证 Skill 阶段二（编写脚本 + 执行 + 项目级稳定性验证）→ 通过后 squash merge 回主分支
→ S7: 部署清单 Skill
```

**S1.5 骨架模块前置补全（场景二专用）：**

S1 识别变更范围后，检查受影响模块是否有详细文档：

```
读取 .codeman/docs/design/INDEX.md 或 arch-overview.md 的文档覆盖状态表
→ 逐个检查受影响模块的 mod-*.md 是否存在
→ 全部存在：跳过 S1.5，正常进入 S2
→ 存在骨架模块（mod-*.md 不存在或状态为 ⬜）：
  → 告知用户并进入前置补全
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
骨架模块前置补全
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
本次变更涉及 {N} 个骨架模块（尚无详细文档）：
  - {模块名}（⬜ 待按需生成）
  - {模块名}（⬜ 待按需生成）

将在需求分析和设计阶段自动读取这些模块的代码，
补全 mod-*.md 技术文档，作为设计和回归测试的基线。
预估额外消耗：~{N*3}K tokens

确认继续？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> 补全工作由下游 Skill 执行：Design Skill 的 Step 0 负责生成 mod-*.md，Testing Skill 的阶段一负责为无 PRD 的老功能生成回归测试用例。S1.5 只做检测和告知，不重复执行。

**场景三（旧项目接入）：**
```
S0: 项目扫描与理解 → S1: 生成架构文档
→ S2: 反向提炼业务 PRD（从代码反推现有功能点）
→ S3: 融入文档体系（写入 STATUS.md / DIRECTIVES.md）
→ 后续按场景二（版本迭代）执行新需求
```

**场景四（轻量快速通道）：**
```
创建 fix/ 分支 → 修复 → 验证 → squash merge 回主分支
```

**场景五（批量迭代）：**
```
S0: 解析 + 分类 + 排序 → 用户确认迭代计划
→ S1: 批量 Bug 修复（快速通道，逐个修复）
→ S2: 合并需求分析（新功能 + 优化统一走需求 Skill）
→ S3: 合并技术方案 Skill
→ S4: 测试验证 Skill 阶段一（生成测试用例，编码前锁定）
→ S5: 逐模块开发实现 Skill
→ S6: 统一测试验证 Skill 阶段二（编写脚本 + 执行，含 Bug 回归 + 新功能测试）
→ S7: 部署清单 Skill
```

## 场景四：快速通道执行

```
0. 创建 fix/ 分支（如 git.branch_strategy 开启）  (~0 tokens)
1. 读取 STATUS.md + DIRECTIVES.md + DOMAIN-RULES.md  (~700 tokens)
2. 定位问题代码（精确引用相关文件）        (~2K tokens)
3. 制定修复方案（简要说明）               (~1K tokens)
4. 【人工确认】方案确认后执行
5. 执行修复                              (~3K tokens)
6. 单元测试 + 快速回归                   (~2K tokens)
7. 轻量 Review（自检）                   (~1K tokens)
8. Git checkpoint（在 fix 分支上）
9. 验证通过 → squash merge 回主分支 → 删除 fix 分支
10.【后置异步】增量同步文档              (~2K tokens)

总 Token: ~12K（vs 完整流程 ~150K+）
```

## 场景五：批量迭代

用户说 `CodeMan 迭代：[内容]` 时执行。适用于周迭代、版本迭代等混合多种工作项的场景。

### S0: 解析、分类、排序

**解析用户输入**：从自由格式的文本中提取每条工作项，用户不需要标注类型。

**AI 自动分类**：根据描述内容判断每条工作项的类型：

| 类型 | 识别线索 | 执行通道 |
|------|---------|---------|
| 🔴 Bug 修复 | "报错"、"错误"、"丢失"、"异常"、"不显示"、"失败"等 | **按 Step 3 轻量任务判定规则判定**。仅非业务逻辑 Bug（文案错别字、样式错乱、链接失效、文档错误等）走快速通道；业务逻辑 Bug（状态/计算/校验/权限/流程/数据处理错误）走完整流程 |
| 🟢 新功能 | "新增"、"支持"、"增加"、"实现"等，或描述中无现有功能引用 | 完整流程（合并走需求→设计→开发→测试） |
| 🟡 优化 | "优化"、"改进"、"调整"、"提升"、"重构"等 | **按 Step 3 轻量任务判定规则判定**。涉及业务逻辑即走完整流程 |

**排序原则**：Bug 修复优先（稳定现有功能）→ 优化 → 新功能

**输出迭代计划供用户确认**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
迭代计划 — 请确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔴 Bug 修复（{N} 项，优先执行）：
  B1. {Bug 描述} → {快速通道 / 完整流程}（AI 按判定规则标注，快速通道需用户二次确认）
  B2. {Bug 描述} → {快速通道 / 完整流程}

🟡 优化（{N} 项）：
  O1. {优化描述} → {快速通道 / 完整流程}

🟢 新功能（{N} 项，合并走完整流程）：
  F1. {功能描述} → 完整流程
  F2. {功能描述} → 完整流程

执行顺序：B1 → B2 → O1 → F1+F2（合并需求分析和设计）
测试策略：Bug 各自验证 → 新功能测试 → 最后统一回归
预估 Token：~{X}K
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认？（或调整分类/优先级/排序）
```

**用户可以在确认时调整**：
- "B2 不是 Bug 是新功能" → 重新分类
- "F1 优先级最高先做" → 调整顺序
- "O1 不做了" → 移除

### S1: 批量 Bug 修复

创建 Bug 修复分支（如 `git.branch_strategy` 开启）：
```bash
git checkout -b fix/iteration-{迭代标识}
```

逐个执行快速通道（场景四），但**共享一轮回归测试**：

```
对于每个 Bug：
  1. 定位根因
  2. 制定修复方案 → 用户确认
  3. 修复 + 单元测试
  4. 验证修复结果
  5. Git checkpoint（在 fix 分支上）

所有 Bug 修完后：
  → 统一回归测试（Smoke 级别，覆盖修复涉及的功能）
  → 缺陷回溯学习（批量执行，涉及隐性知识的写入领域规则）
  → 测试通过后 squash merge 回主分支
```

### S2-S4: 合并执行新功能 + 优化

创建功能分支（如 `git.branch_strategy` 开启）：
```bash
git checkout -b feat/iteration-{迭代标识}
```

将所有新功能和需要走完整流程的优化**合并为一批**，统一执行：

```
S2: 需求分析 Skill — 所有新功能一起分析，生成各自的 feat-*.md
S3: 技术方案 Skill — 统一做模块拆分和设计，共享架构决策
S4: 开发实现 Skill — 按模块依赖顺序逐模块编码（每个模块 checkpoint）
```

> 合并执行的好处：共享一轮需求确认和设计确认，避免每个功能单独走一遍。

### S5: 统一测试

所有工作项（Bug 修复 + 新功能 + 优化）完成后，执行一轮统一测试：

```
1. Bug 修复回归（确认之前修的 Bug 没被新功能引入回退）
2. 新功能测试（完整 L1-L5）
3. 全量回归（Smoke 级别，覆盖核心路径）
4. 测试通过后 squash merge feat 分支回主分支
```

在 STATUS.md 中记录迭代级别的进度：

```markdown
  - B2: 🔄 线索公池卡片丢失 — 修复中
  - F1: ⬜ 线索批量创建商机 — 待开发
```

## 场景三：旧项目接入流程

### S0: 项目规模判断

首先通过顶层扫描（只读目录结构，不读业务代码）估算项目规模：

```bash
# 统计源码文件数（排除 node_modules / vendor / dist 等）
find . -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.go" -o -name "*.py" \) \
  | grep -v node_modules | grep -v vendor | grep -v dist | wc -l
```

根据结果向用户确认接入模式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
检测到旧项目，请选择接入模式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
源码文件数：约 {N} 个
模块数估算：约 {M} 个

模式 A — 完整接入（适合 ≤ 20 个模块的项目）
  一次性扫描全部代码，生成完整文档体系
  预估耗时：1-3 小时 | Token：~50K-100K
  适用：中小型项目，希望立即拥有完整文档

模式 B — 骨架接入（适合任意规模，推荐大型项目）
  只读目录结构和配置文件，生成轻量架构骨架
  后续开发时按需自动补全涉及模块的详细文档
  预估耗时：5-10 分钟 | Token：~3K-5K
  适用：大型项目，希望快速启动后渐进式补全文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择：A 或 B？
```

### 模式 A：完整接入（原有逻辑）

#### S0-A: 项目扫描与理解

1. 顶层扫描（只读目录结构 + 配置文件）→ 模块依赖图
2. 按优先级排序（核心业务优先）
3. 分批扫描（每批 3-5 模块，避免上下文溢出）
4. 合并与交叉验证
5. 生成扫描覆盖台账（不可跳过）

**扫描台账规则：**
- 顶层扫描后，用 Glob 列出所有源码目录，记录为「待扫描模块清单」
- **排除以下非源码目录**（不计入台账）：`node_modules`、`vendor`、`.git`、`dist`、`build`、`.next`、`.nuxt`、`__pycache__`、`.tox`、`target`、`coverage`、`.cache`、`.turbo`、`tmp`、`.temp`、`out`、以及项目中其他明显的构建产物 / 依赖 / 缓存目录
- 每批扫描完成后，在台账中标记 ✅ 已扫描 / ⏳ 待扫描
- **大文件策略**：单文件超过 500 行时，必须分段读取（每段 ≤ 300 行），台账记录已读行范围（如 `L1-300 ✅, L301-600 ✅, L601-850 ✅`）；禁止只读头部就标记已完成
- 全部模块标记 ✅ 后方可声明「扫描完成」，台账写入 arch-overview.md 末尾 `## 附录：扫描覆盖台账` 章节

**扫描覆盖台账格式：**
```markdown
## 附录：扫描覆盖台账

| 模块/目录 | 关键文件数 | 已读文件数 | 覆盖率 | 备注 |
|-----------|-----------|-----------|--------|------|
| src/auth  | 5         | 5         | 100%   | -    |
| src/order | 8         | 8         | 100%   | order-calc.ts 分 3 段读取 |
```

输出：模块依赖图摘要 + 扫描覆盖台账，写入 `.codeman/docs/design/arch-overview.md`

#### S1-A: 生成架构文档

- **模块清单**：每个模块的职责、入口文件、对外接口
- **依赖关系**：模块间调用关系图（文字版）
- **技术债务清单**：明显的代码异味、过时依赖、缺失测试等

写入 `.codeman/docs/design/arch-overview.md`（追加）

**架构文档完备性自检（不可跳过）：**
- 逐一比对扫描台账中的已扫描模块，确认每个模块在架构文档中至少有：职责描述（≥1 句）、入口文件路径、对外接口列表

#### S2-A: 反向提炼业务 PRD

从代码实现反推现有功能点，建立 PRD 基准。

**执行步骤：**

1. 遍历各业务模块，识别对外暴露的功能（页面、API、核心逻辑）
2. 为每个识别到的功能点，基于 `{CODEMAN_HOME}/templates/prd/feat-template.md` 生成**完整** PRD 碎片文件：

   **文件路径：** `.codeman/docs/prd/feat-{功能点英文名}.md`

   **标题格式：** `# F{ID}: {功能点名称}（现有功能）`

   **反向填充指导（从代码提取每个章节）：**

   | 章节 | 代码提取来源 | 必填/选填 |
   |------|-------------|----------|
   | 用户故事 | 从功能入口的调用上下文推断用户角色（auth 中间件/权限检查）、操作目的和价值 | 必填 |
   | 功能描述 | 从代码注释、函数命名、模块 README 综合提炼 | 必填 |
   | 业务流程图 | 追踪主执行路径的控制流，识别分支判断点，画 ASCII 流程图 | 必填 |
   | 界面原型 ASCII | 前端：从组件 JSX/模板结构推断布局；纯后端功能：写「无界面，跳过」 | 前端必填 |
   | 功能点明细 GWT | Given：从验证逻辑提取前置条件；When：从输入处理提取用户操作；Then：从返回值/状态变更/UI 更新提取预期结果 | 必填 |
   | 数据字典 | 从数据库 schema / ORM 模型 / TypeScript interface / Go struct 直接提取 | 必填 |
   | UI 设计要求 | 从 CSS/样式代码、响应式断点、无障碍属性提取；设计截图标「待补充」 | 前端选填 |
   | 业务规则 | 从校验函数、业务常量、权限检查、限流配置提取 | 必填 |
   | 边界场景 | 从错误处理、try-catch、边界判断、输入校验提取 | 必填 |
   | 非功能需求 | 从配置文件提取（限流阈值、超时设置、缓存 TTL、性能预算等）；代码中无明确体现的标 `[legacy: 待迭代时补充]` | 选填 |
   | 依赖关系 | 从 import 语句、外部服务调用、package 配置提取 | 必填 |
   | 假设与约束 | 从运行时要求、浏览器兼容配置、环境变量推断；无法推断的标 `[legacy: 待迭代时补充]` | 选填 |

   **状态字段：** 在文件头部 `状态` 后追加 `existing（现有功能，非本次新增）`

   **填充原则：**
   - 所有「必填」章节必须有从代码提取的实质内容，禁止整章留空或只写 TODO
   - 「选填」章节代码中无明确体现时，标 `[legacy: 待迭代时补充]` 并写一句说明原因
   - GWT 验收标准是下游测试 Skill 的核心输入，必须从代码行为认真推导，每个功能点至少 3 条
   - 业务流程图必须画，哪怕简单功能也要有主流程 → 分支 → 结果的完整链路

3. 更新 `.codeman/docs/prd/INDEX.md`，列出所有现有功能点

4. **PRD 覆盖比对（机械化，不可跳过）**：
   - 从 arch-overview.md 扫描台账提取所有已扫描业务模块列表
   - 用 Glob 列出 `.codeman/docs/prd/feat-*.md` 实际生成的文件
   - 逐一比对：每个业务模块是否至少对应一个 feat-*.md
   - 对每个 feat-*.md 检查「空模板」：若任意必填章节（用户故事、功能描述、业务流程图、功能点明细 GWT、数据字典、业务规则、边界场景、依赖关系）仅含模板占位符（如 `{{功能点名称}}`、`{{用户角色}}`）或仅有 `TODO` → 标记 ❌ 空模板；GWT 验收标准不足 3 条 → 标记 ❌
   - 有 ❌ → 回读源码补充后重新比对

5. **人工确认**：输出功能点清单，询问用户：
   - 是否有遗漏的核心功能？
   - 是否有描述不准确的地方？
   - 哪些功能点是后续迭代的重点？

   ```
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   反向提炼 PRD 完成
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   空模板检查：{P} 个通过，{Q} 个已补充

   已识别 X 个现有功能点：
     F01: {功能名} — {一句话描述}
     F02: {功能名} — {一句话描述}
     ...
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   ```

### 模式 B：骨架接入（大型项目推荐）

**核心原则：** 接入时只建立「架构骨架」，不读业务代码细节。后续每次开发新需求时，自动按需补全涉及模块的详细文档（「接触即文档化」）。

#### S0-B: 生成架构骨架

**只读以下内容（不读业务代码）：**
- 顶层目录结构（`ls -la` 或 `tree -L 3`）
- 依赖配置文件（`package.json` / `go.mod` / `requirements.txt` / `pom.xml` 等）
- 入口文件（`main.ts` / `app.py` / `main.go` / `index.ts` 等，只读文件头部 30 行）
- 路由注册文件（如有，只读路由列表，不读路由实现）

- 文档覆盖状态表（所有模块初始标记为 `⬜ 待按需生成`）

**Token 预估：** ~2K-4K（仅读目录和配置，不读业务代码）

## 增量文档补全（CodeMan 同步）

**触发方式：** ① 自动 — orchestrator 启动时检测到外部代码变更；② 手动 — 用户说 `CodeMan 同步`

**核心原则：** 只补缺失的文档，不覆盖已有的。比"骨架接入"更轻量——不重建整个文档体系，只填补空白。

### 执行步骤

**Step 1: 扫描代码变更范围**

两种模式根据触发方式选择：

- **有 git 记录（自动触发）：** 分析他人提交涉及的文件，聚合为模块粒度的变更列表
  ```bash
  git log --since="{last_updated}" --not --author="{当前用户}" --name-only --pretty=format:""
  ```
- **无 git 记录（手动触发）：** 全量扫描项目代码目录结构，与已有的 design/INDEX.md、prd/INDEX.md 对比，找出未覆盖的模块/功能

**Step 2: 对比已有文档，识别缺失**

```
扫描结果（涉及的模块/文件）
    ↓
对比 design/INDEX.md → 哪些模块缺少 mod-*.md？
对比 prd/INDEX.md → 哪些功能缺少 feat-*.md？
对比 api/INDEX.md → 哪些接口缺少 api-*.md？
    ↓
输出缺失清单
```

**Step 3: 向用户确认补全范围**


## 不确定即停（全局硬规则，所有 Skill 必须遵守）

**核心原则：宁可暂停问用户，绝不自作主张。这是贯穿所有阶段的铁律。**

以下情况在**任何阶段**遇到时，必须立即暂停并向用户确认，**严禁自行推测、假设、补充或做出任何未经确认的决定**：

| # | 场景 | 说明 |
|---|------|------|
| 1 | 用户意图不明确 | 需求描述、指令、反馈存在多种理解方式 |
| 2 | 需求/方案/用例有歧义 | 同一句话可以有不同解释 |
| 3 | 业务规则不明确 | 如"适当的权限控制"、"合理的分页大小"等模糊表述 |
| 4 | 技术选型有多个可行方案 | 不确定选哪个时必须列出方案让用户决策 |
| 5 | 发现 PRD/方案/代码有矛盾 | 内容之间或内容与用户描述存在冲突 |
| 6 | 需要引入未明确约定的内容 | PRD 未提及的逻辑、未约定的第三方库、未定义的数据结构 |
| 7 | 边界场景处理方式不确定 | 不确定该怎么处理异常、边界、特殊情况 |
| 8 | 测试用例的预期结果不确定 | 不确定某个输入应该产生什么输出 |
| 9 | 修复方案有风险或不确定 | 修复可能引入新问题或不确定是否正确 |
| 10 | 项目状态异常 | 文件缺失、格式错误、进度不一致 |

**确认格式（统一使用）：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
关于"{不确定点}"，我理解到以下几种可能：

1. {理解A}：[对应的实现方式/影响]
2. {理解B}：[对应的实现方式/影响]

我的建议：{推荐方案}，原因：{简要理由}

请问您的意思是哪种？或者请补充说明。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**禁止行为：**
- ❌ 不确定时自行选择"最合理"的方案继续
- ❌ 用 `[UNCERTAIN]` 标记后不暂停直接继续
- ❌ 猜测用户意图后执行
- ❌ 在确认前生成文件或修改代码

## Git 操作限制（全局硬规则，所有 Skill 必须遵守）

**核心原则：AI 可以 commit，绝对不能 push。推送代码到远程仓库是用户的专属权利。**

| 操作 | 权限 | 说明 |
|------|------|------|
| `git status` | ✅ 允许 | 只读操作 |
| `git diff` | ✅ 允许 | 只读操作 |
| `git log` | ✅ 允许 | 只读操作 |
| `git checkout -b {branch}` | ✅ 允许 | 创建 feature 分支（仅在分支策略开启时） |
| `git checkout {branch}` | ✅ 允许 | 切换分支（仅在分支策略开启时） |
| `git merge --squash {branch}` | ✅ 允许 | 测试通过后 squash merge（仅在分支策略开启时） |
| `git branch -d {branch}` | ✅ 允许 | 删除已合并的 feature 分支 |
| `git add` | ✅ 允许 | 暂存文件 |
| `git commit -m "..."` | ✅ 允许 | 必须写明 commit message，遵循 Conventional Commits |
| `git push` | ❌ **绝对禁止** | 推送操作必须由用户手动执行 |
| `git push --force` | ❌ **绝对禁止** | 任何变体都不允许 |
| `git push -u origin ...` | ❌ **绝对禁止** | 包含 -u 参数的也不允许 |
| `git reset --hard` | ❌ 禁止 | 破坏性操作 |
| `git rebase` | ❌ 禁止 | 可能导致冲突 |

**commit 规范：**
- message 必须遵循 Conventional Commits 格式：`type(scope): description`
- type 包含：feat / fix / docs / style / refactor / test / chore
- body 说明变更原因，footer 引用 PRD 功能点 ID
- ❌ **禁止**在 commit message 末尾追加 `Co-Authored-By: Claude ...` / `Co-Authored-By: <AI 助手>` 等行
  - Why: 用户不希望 GitHub Contributors 列表出现 AI 助手记录
  - 即使 AI 助手的系统默认 commit 模板包含此行，执行 codeman 流程时也必须移除
  - 适用于所有 codeman 触发的 commit（development / fix / testing / 任何场景）

```bash
# Cursor 环境

# Claude Code 环境
```

> 如不确定当前环境，可用 `{CODEMAN_HOME}/init.sh`（CODEMAN_HOME 见本文件顶部说明）。

**2. 读取 init.sh 输出的项目摘要**

其中 `HOST_SKILLS_ROOT` 按宿主环境取值：

| 宿主 | HOST_SKILLS_ROOT |

## 场景一初始化步骤（S0）

当判定为全新项目时，执行初始化：

1. 询问项目信息：

   ```
   .codeman/
   ├── config.yaml              ← 根据用户选择填充
   ├── docs/
   │   ├── STATUS.md            ← 初始化状态
   │   ├── DIRECTIVES.md        ← 空白，待填充
   │   ├── PROJECT-OVERVIEW.md  ← 项目概览（面向新成员，各阶段自动更新）
   │   ├── prd/INDEX.md
   │   ├── design/INDEX.md
   │   ├── api/INDEX.md
   │   ├── tests/INDEX.md
   │   ├── reviews/INDEX.md
   │   └── deploy/INDEX.md
   ├── skills/INDEX.md
   └── rules/INDEX.md
   ```

3. 根据技术栈复制适用的规范模板到 `.codeman/rules/`

4. **S0.x: 处理待生成规范（PENDING-RULES.md）**

   检查 `.codeman/rules/PENDING-RULES.md` 是否存在：

   - 如果**不存在**：跳过此步骤
   - 如果**存在**：读取文件，找出所有 `- [ ]` 开头的未完成条目，逐一生成规范文件

   **对每个待生成条目，执行以下操作：**

   a. 读取 `.codeman/config.yaml` 中的技术栈信息（`tech_stack` 字段）

   b. 根据技术栈和规范名称，生成 `.mdc` 规范文件内容，格式要求：
      ```
      ---
      description: "{规范描述}。当{触发条件}时自动加载。"
      globs: "{适用的文件 glob 模式}"
      alwaysApply: false
      ---

      # {规范名称}（L2 项目层）

      ## {规范类别1}
      - {具体可操作的规范条目}

      ## 禁止
      - 禁止{禁止行为}
      ```
      质量约束：< 50 行，聚焦单一关注点，条目必须具体可操作

   c. 写入 `.codeman/rules/{filename}`（filename 来自条目，如 `proj-svelte-component.mdc`）

   d. 同步到 `.cursor/rules/codeman-{filename}`

   e. 将 PENDING-RULES.md 中对应条目从 `- [ ]` 改为 `- [x]`，并在行尾追加 `（已生成）`

   f. 更新 `.codeman/rules/INDEX.md`，追加新规范条目

   **完成后输出摘要：**
   ```
   已动态生成 X 条规范：
     ✓ proj-svelte-component.mdc — Svelte 组件开发规范
     ✓ proj-rust-backend.mdc — Rust 后端开发规范
   已同步到 .cursor/rules/
   ```

5. 同步 Rules 到 `.cursor/rules/`（加 `codeman-` 前缀）

6. 更新 STATUS.md：`phase: requirements`

7. 调度需求分析 Skill：Read `{CODEMAN_HOME}/skills/requirements/SKILL.md` 并按其指令执行

```
  - .codeman/docs/reviews/INDEX.md
  - .codeman/docs/deploy/INDEX.md
```

**2. 汇总生成 PROJECT-OVERVIEW.md**

基于 `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md` 模板，用自然语言填充每个章节。**必须内联核心摘要内容，不能只放链接**：

- **项目简介**：从 config.yaml + PRD 碎片中提炼 2-3 段：做什么、给谁用、核心价值
- **技术架构**：从 config.yaml + arch-overview.md 汇总技术栈表（含选型理由）+ 2-3 段架构描述
- **功能地图**：按功能分类组织，每个功能附**一句话核心需求摘要**和**关键验收标准**，不只是表格行
- **模块结构**：每个模块附**一句话关键设计说明**（如"采用 JWT 无状态鉴权"），不是空表格
- **API 概览**：每个接口附**核心请求/响应字段摘要**（如"返回 user_id, name, email"），不是空表格
- **关键决策记录**：每个决策用 1-2 句话描述**决策内容和原因**，不是空列表

- **新成员上手指南**：根据技术栈填充实际的安装和启动命令

**生成原则：**
- **内联内容优先**：每个章节必须包含可读的核心摘要，不能只放 `→ 详见 xxx.md` 链接
- **链接作为补充**：每个章节末尾附 `→ 详见 xxx.md` 链接，供需要深入阅读的人使用
- **可独立阅读**：新成员只读本文档就能理解项目做了什么、用了什么技术、进度如何
- 用自然语言描述，不保留 `{{}}` 模板占位符
- 总量控制在 250 行以内（比原来增加 50 行，用于内联内容）
- 尚未生成的章节标注「待后续阶段自动补全」

**3. 写入文件**

写入 `.codeman/docs/PROJECT-OVERVIEW.md`（覆盖已有内容）。

**4. 输出确认**

### 注册（用户说"注册"、"挂载"、"添加"、"把 XX 挂到 XX"）

1. 从用户描述中提取信息，缺失的**逐个询问**（不要一次问完）：
   - **name**：Skill 名称（如"brainstorming"）
   - **path**：SKILL.md 文件路径（可自动搜索常见安装路径：`~/.claude/skills/`、`~/.cursor/skills/`）

## CodeMan 升级流程

用户说「CodeMan 升级」时执行，在当前项目中一键完成框架升级 + 项目配置迁移。

### Step 1: 拉取最新源码 + 同步框架文件

```bash
# 读取源码路径
SOURCE_PATH=$(cat {CODEMAN_HOME}/.source-path 2>/dev/null)

# 如果 .source-path 不存在，提示用户
if [ -z "$SOURCE_PATH" ] || [ ! -d "$SOURCE_PATH" ]; then
    echo "⚠️ 未找到 CodeMan 源码路径。请手动执行："
    echo "   bash /path/to/codeman/update.sh"
    return
fi

# 拉取最新源码
cd "$SOURCE_PATH" && git pull

# 执行 update.sh 同步到安装目录
bash "$SOURCE_PATH/update.sh"
```

展示 update.sh 的输出结果。

**重要：重新加载升级流程。** update.sh 已将最新版本的 orchestrator SKILL.md 同步到安装目录。为确保后续步骤使用最新的迁移表和逻辑，必须重新读取：

```
Read {CODEMAN_HOME}/skills/orchestrator/SKILL.md
```

然后跳转到该文件中「CodeMan 升级流程」的 **Step 2** 继续执行（不要重复 Step 1）。

### Step 2: 当前项目配置迁移

**先读取框架最新版本号：** 读取 `{CODEMAN_HOME}/templates/config.yaml` 的 `codeman_config_version` 字段，作为**目标版本**（不要使用本文件中硬编码的版本号，因为本文件可能是旧版本）。

然后读取当前项目 `.codeman/config.yaml` 的 `codeman_config_version` 字段：

- 等于目标版本 → 输出 "✅ 项目配置已是最新版本，无需迁移"
- 不存在或 < 目标版本 → 执行迁移：

**当前目标版本：3**（此为参考值，实际以 `{CODEMAN_HOME}/templates/config.yaml` 中读取的为准）

**迁移表：**

| 从版本 | 到版本 | 追加内容 | 默认值 |
|-------|-------|---------|-------|
| 无/1 | 2 | `codeman_config_version` | `2` |
| 无/1 | 2 | `external_skills` | `[]`（附注释示例） |
| 2 | 3 | （无新配置项，仅第三方 Skills 配置同步，见 Step 3.5） | — |

**迁移流程：**
1. 展示将追加的配置段，询问用户确认
2. 确认后在 config.yaml 末尾追加缺失的配置段（只追加不修改已有配置）
3. 更新 `codeman_config_version` 为目标版本（从 `{CODEMAN_HOME}/templates/config.yaml` 读取的值）

### Step 3: 全局规范文件更新

用 `{CODEMAN_HOME}/rules/global-*.mdc` 覆盖项目 `.codeman/rules/global-*.mdc`。

静默覆盖，不需要用户确认（纯框架内容，用户自定义规范在 `proj-*.mdc` 中不受影响）。

### Step 3.5: 第三方 Skills 配置同步

读取 `{CODEMAN_HOME}/external-skills-registry.json`（全局注册表，记录本机已安装的第三方 Skills）。

对比注册表与当前项目 `.codeman/config.yaml` 的 `external_skills` 列表：

- 以注册表中每个 entry 的 `skill_path` 为匹配键（因为同一个仓库可能有多个 Skills，如 superpowers 包含 brainstorming 和 subagent-driven-development）
- 找出注册表中有但项目 config 中没有的 entry → 这些是**新增的推荐 Skills**

**对于每个新增 entry：**

检查对应的 Skill 文件是否存在（根据当前宿主环境拼接路径，如 `~/.claude/skills/{name}/{skill_path}`）：

1. **已安装（文件存在）** → 展示 entry 信息，询问用户是否添加到项目配置
2. **未安装（文件不存在）** → 展示 entry 信息，询问用户是否安装（git clone）并添加到项目配置

**展示格式：**

```
检测到新增的推荐第三方 Skills：

  1. [已安装] subagent-driven-development
     说明：Superpowers 逐任务子代理派遣 + 两阶段 Review
     挂载点：development.execution（开发阶段执行托管）

是否添加到当前项目配置？[Y/n]
```

用户确认后，将对应 entry 转换为 `external_skills` 格式追加到 config.yaml：

```yaml
  - name: "subagent-driven-development"
    path: "{HOST_SKILLS_ROOT}/{name}/{skill_path}"
    hook: "{suggested_hook}"
    order: 1
    output: "{suggested_output}"
    description: "{description}"
    confirm: true
```

**`path:` 字段构造规则**（必须严格遵守）：

`HOST_SKILLS_ROOT` 按宿主环境取值：

| 宿主 | HOST_SKILLS_ROOT |
|------|------------------|
| Claude Code / OpenCode | `~/.claude/skills` |
| Cursor | `~/.cursor/skills` |
| Trae | `~/.trae/skills` |

⚠️ **严禁**在 `{name}` 前拼接 `.codeman/`、`external-skills/` 等任何子目录 —— 第三方 Skill 安装在 `HOST_SKILLS_ROOT` 下，与 `.codeman/` **同级**，不在它内部。注意区分：
- `external-skills-registry.json` 是一个 **JSON 文件**（在 `{CODEMAN_HOME}/` 下），不是目录
- 第三方 Skill 的 SKILL.md 文件在 `{HOST_SKILLS_ROOT}/{name}/` 下，与 `{CODEMAN_HOME}` **平级**

**正确示例**（Claude Code）：

| name | skill_path | ✅ 正确 path |
|------|------------|--------------|
| wireframe | `SKILL.md` | `~/.claude/skills/wireframe/SKILL.md` |
| ui-design-brain | `SKILL.md` | `~/.claude/skills/ui-design-brain/SKILL.md` |
| superpowers | `skills/brainstorming/SKILL.md` | `~/.claude/skills/superpowers/skills/brainstorming/SKILL.md` |

**错误示例（勿学）**：
- ❌ `~/.claude/skills/.codeman/external-skills/wireframe/SKILL.md`
- ❌ `{CODEMAN_HOME}/external-skills/wireframe/SKILL.md`

写入 config.yaml 之前必须验证 `path` 指向的文件真实存在（`Read` 或 `ls` 检查），不存在时提示用户并中止写入该条目。

- 无新增 entry → 输出 "✅ 第三方 Skills 配置已是最新"
- 注册表不存在 → 跳过此步骤
