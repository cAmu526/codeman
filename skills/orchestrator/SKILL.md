# CodeMan Orchestrator — 主编排入口

## 宿主环境说明

CodeMan 支持 Cursor、Claude Code、OpenCode 和 Trae 四种宿主环境。**CODEMAN_HOME** 的路径规则：
- Cursor：`~/.cursor/skills/.codeman`
- Claude Code：`~/.claude/skills/.codeman`
- OpenCode：`~/.claude/skills/.codeman`（通过 Claude Code 兼容层读取）
- Trae：`~/.trae/skills/.codeman`

在执行任何需要读取模板文件的步骤时，请根据当前运行环境自动选择正确的 CODEMAN_HOME 路径。如无法判断，优先尝试 `~/.claude/skills/.codeman`，其次 `~/.cursor/skills/.codeman`，再次 `~/.trae/skills/.codeman`。

---

## 概述

你是 CodeMan 工作流框架的主编排器。用户唤起 CodeMan 后，你负责：
1. 读取项目状态
2. 识别任务类型和规模
3. 选择合适的工作流通道
4. 向用户确认后调度对应 Skill 链

## 执行步骤

### Step 1: 读取项目状态

```
读取 .codeman/docs/STATUS.md
  → 如果文件不存在：判定为"需要初始化"场景
  → 如果文件存在：解析当前 phase、current_module、待处理事项
```

```
读取 .codeman/docs/DIRECTIVES.md
  → 加载关键指令和约束
  → 如果文件不存在：跳过，后续初始化时创建
```

**外部代码变更检测（自动，仅对工作类命令触发）：**

当用户执行工作类命令（继续、开始开发、新需求）时，在进入正式流程前自动检测是否有外部代码变更需要同步文档：

```bash
# 从 STATUS.md 的 last_updated 时间获取上次更新时间
# 检查此时间之后是否有他人的提交
git log --since="{last_updated}" --not --author="{当前 git 用户}" --oneline
```

- 无新提交 → 跳过，正常继续
- 有新提交 → 自动触发「增量文档补全」（见下方），完成后继续原本的工作流
- 非 git 仓库或无 git 记录 → 跳过（用户可手动 `CodeMan 同步`）

> 查看状态（`CodeMan 状态`/`CodeMan 概览`）不触发补全，只提示："检测到 {N} 个外部提交尚未同步文档，建议执行 `CodeMan 同步`"。

### Step 2: 分析用户输入，判断场景

根据 STATUS.md 内容和用户输入，判断以下场景之一：

**场景零：项目初始化（最高优先级）**
- 触发条件：用户说"CodeMan 初始化"，或 .codeman/ 目录不存在
- 用户说：CodeMan 初始化、初始化项目、init
- 执行：见下方「场景零：项目初始化流程」

**场景一：全新项目开发**
- 触发条件：.codeman/ 已初始化，STATUS.md 的 phase = "init"，且项目目录中无业务代码
- 用户说：CodeMan 开始开发、新项目、从零开始

**场景二：版本迭代/需求变更**
- 触发条件：STATUS.md 存在且 phase = "deploy-ready" 或 "completed"，或用户明确提到新需求/迭代
- 用户说：新需求、版本迭代、增加功能、修改需求

**场景三：旧项目接入**
- 触发条件：.codeman/ 已初始化，但项目中有已有业务代码，且 STATUS.md 的 phase = "init"
- 用户说：接入现有项目、已有代码、老项目
- 也可由场景零自动触发（init.sh 检测到有代码后自动进入）

**场景四：轻量快速修改（快速通道）**
- 触发条件：满足下方"轻量任务判定规则"的全部条件
- 用户说：修复、小改动、样式调整、Bug 修复

**场景五：批量迭代（混合新功能 + Bug 修复 + 优化）**
- 触发条件：用户说"迭代"、"本周迭代"、"这周要做"，且输入包含多条工作项
- 用户说：CodeMan 迭代：[混合列表]
- 执行：见下方「场景五：批量迭代」

**特殊：断点续做**
- 触发条件：STATUS.md 存在且 phase 不是 "completed"，且用户说"继续"
- 优先级高于其他场景判断

**特殊：文档同步**
- 触发条件：用户说"同步"、"同步文档"、"CodeMan 同步"
- 执行：见下方「增量文档补全」流程
- 适用场景：同事直接给了代码文件、非 git 管理的代码变更、手动触发同步

**特殊：查看状态**
- 触发条件：用户说"状态"、"进度"、"查看"
- 直接输出 STATUS.md 摘要，不启动工作流

**特殊：项目概览**
- 触发条件：用户说"概览"、"项目概览"、"CodeMan 概览"、"项目总览"
- 执行：见下方「特殊：生成/更新项目概览」

### Step 3: 轻量任务判定规则

**必须全部满足才判定为轻量任务：**

| # | 条件 | 说明 |
|---|------|------|
| 1 | 预估影响文件数 ≤ 3 | 修改范围小 |
| 2 | 不涉及架构变更 | 不新增模块或重构模块边界 |
| 3 | 不新增 API 端点 | 不涉及接口契约变更 |
| 4 | 不涉及数据模型变更 | 不修改表结构或核心数据结构 |
| 5 | 不涉及多模块联动 | 局限在单一模块内 |
| 6 | 用户未明确标注为复杂任务 | 尊重用户判断 |

**误判防护：** 快速通道执行过程中，如发现影响范围超出预估，立即暂停并提示用户切换到完整流程。

### Step 4: 向用户确认

输出以下格式的确认信息：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CodeMan 场景识别
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
识别结果：[场景名称]
当前状态：[STATUS.md 摘要，如无则显示"新项目"]
即将执行：[工作流步骤概述]
预估轮次：[X-Y 轮]
预估 Token：[~XK]
是否需要生成本次迭代的版本文档？（回复版本名如"v1.2.0"，不需要则留空）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认继续？（直接回复即可，或说"切换到[其他场景]"）
```

**版本文档处理：**

如果用户回复了版本名：
1. 创建目录：`.codeman/docs/iterations/{版本名}/`
2. 创建 `SUMMARY.md` 骨架（版本名、日期、场景类型）
3. 创建空的 `PRD.md` 和 `DESIGN.md`（后续 Skill 写碎片文件时同步写入）
4. 更新 `.codeman/docs/iterations/INDEX.md`（追加行）
5. 在 STATUS.md 中写入 `iteration_version: {版本名}`，后续 Skill 据此判断是否同步写入版本文档

如果用户未提供版本名，不创建版本文档目录，STATUS.md 中不写入 `iteration_version`。

### Step 5: 调度对应 Skill 链

用户确认后，按以下顺序调度：

**Skill 调度机制（跨 IDE 通用，必须遵守）：**

调度任何 Skill 时，必须通过 **Read 文件** 的方式加载并执行，不依赖斜杠命令或其他 IDE 特有机制。

具体操作：读取 `{CODEMAN_HOME}/skills/{skill-name}/SKILL.md`，然后按其指令执行。

Skill 名称与路径映射表：

| Skill 名称 | 目录名 | 路径 |
|------------|--------|------|
| 需求分析 Skill | requirements | `{CODEMAN_HOME}/skills/requirements/SKILL.md` |
| 技术方案 Skill | design | `{CODEMAN_HOME}/skills/design/SKILL.md` |
| 开发实现 Skill | development | `{CODEMAN_HOME}/skills/development/SKILL.md` |
| 测试验证 Skill | testing | `{CODEMAN_HOME}/skills/testing/SKILL.md` |
| Review Skill | review | `{CODEMAN_HOME}/skills/review/SKILL.md` |
| 修复闭环 Skill | fix | `{CODEMAN_HOME}/skills/fix/SKILL.md` |
| 部署清单 Skill | deploy | `{CODEMAN_HOME}/skills/deploy/SKILL.md` |
| 自进化引擎 Skill | evolve | `{CODEMAN_HOME}/skills/evolve/SKILL.md` |

> 示例：调度"需求分析 Skill"时，实际执行 `Read {CODEMAN_HOME}/skills/requirements/SKILL.md` 并按其内容执行。
> CODEMAN_HOME 的值见本文件顶部「宿主环境说明」。

**所有场景通用规则：**
每次阶段转换（S{N} → S{N+1}）时，重新读取 DIRECTIVES.md，
执行"周期性上下文回顾协议"，输出回顾摘要后再调度下一个 Skill。

**场景零（项目初始化）：**
```
执行 init.sh → 读取输出摘要 → 判断项目类型
  → 新项目（无业务代码）：直接进入场景一
  → 旧项目（有业务代码）：直接进入场景三
```

**场景一（全新项目）：**
```
S0: 初始化 → S1: 需求分析 Skill → S2: 技术方案 Skill
→ S3: 创建 feat/ 分支 → 开发实现 Skill（checkpoint commit）
→ S4: 测试验证 Skill → 通过后 squash merge 回主分支
→ S5: 修复闭环 Skill（如有失败）→ S6: 部署清单 Skill
```

**场景二（版本迭代 / 新需求）：**
```
S1: 变更范围识别 → S2: 增量需求分析 Skill
→ S3: 增量技术方案 Skill
→ S4: 创建 feat/ 分支 → 增量开发实现 Skill（checkpoint commit）
→ S5: 增量测试验证 Skill → 通过后 squash merge 回主分支
→ S6: 部署清单 Skill
```

**场景三（旧项目接入）：**
```
S0: 项目扫描与理解 → S1: 生成架构文档
→ S2: 反向提炼业务 PRD（从代码反推现有功能点）
→ S3: 融入文档体系（写入 STATUS.md / DIRECTIVES.md）
→ 后续按场景二（版本迭代）执行新需求
```

**场景四（轻量快速通道）：**
```
创建 fix/ 分支 → 修复 → 验证 → squash merge 回主分支
```

**场景五（批量迭代）：**
```
S0: 解析 + 分类 + 排序 → 用户确认迭代计划
→ S1: 批量 Bug 修复（快速通道，逐个修复）
→ S2: 合并需求分析（新功能 + 优化统一走需求 Skill）
→ S3: 合并技术方案 Skill
→ S4: 逐模块开发实现 Skill
→ S5: 统一测试验证 Skill（含 Bug 回归 + 新功能测试）
→ S6: 部署清单 Skill
```

## 场景四：快速通道执行

```
0. 创建 fix/ 分支（如 git.branch_strategy 开启）  (~0 tokens)
1. 读取 STATUS.md + DIRECTIVES.md + DOMAIN-RULES.md  (~700 tokens)
2. 定位问题代码（精确引用相关文件）        (~2K tokens)
3. 制定修复方案（简要说明）               (~1K tokens)
4. 【人工确认】方案确认后执行
5. 执行修复                              (~3K tokens)
6. 单元测试 + 快速回归                   (~2K tokens)
7. 轻量 Review（自检）                   (~1K tokens)
8. Git checkpoint（在 fix 分支上）
9. 验证通过 → squash merge 回主分支 → 删除 fix 分支
10.【后置异步】增量同步文档              (~2K tokens)

总 Token: ~12K（vs 完整流程 ~150K+）
```

## 场景五：批量迭代

用户说 `CodeMan 迭代：[内容]` 时执行。适用于周迭代、版本迭代等混合多种工作项的场景。

### S0: 解析、分类、排序

**解析用户输入**：从自由格式的文本中提取每条工作项，用户不需要标注类型。

**AI 自动分类**：根据描述内容判断每条工作项的类型：

| 类型 | 识别线索 | 执行通道 |
|------|---------|---------|
| 🔴 Bug 修复 | "报错"、"错误"、"丢失"、"异常"、"不显示"、"失败"等 | 快速通道（场景四） |
| 🟢 新功能 | "新增"、"支持"、"增加"、"实现"等，或描述中无现有功能引用 | 完整流程（合并走需求→设计→开发→测试） |
| 🟡 优化 | "优化"、"改进"、"调整"、"提升"、"重构"等 | 按复杂度判定走快速通道或完整流程 |

**排序原则**：Bug 修复优先（稳定现有功能）→ 优化 → 新功能

**输出迭代计划供用户确认**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
迭代计划 — 请确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔴 Bug 修复（{N} 项，优先执行）：
  B1. {Bug 描述} → 快速通道
  B2. {Bug 描述} → 快速通道

🟡 优化（{N} 项）：
  O1. {优化描述} → {快速通道 / 完整流程}

🟢 新功能（{N} 项，合并走完整流程）：
  F1. {功能描述} → 完整流程
  F2. {功能描述} → 完整流程

执行顺序：B1 → B2 → O1 → F1+F2（合并需求分析和设计）
测试策略：Bug 各自验证 → 新功能测试 → 最后统一回归
预估 Token：~{X}K
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认？（或调整分类/优先级/排序）
```

**用户可以在确认时调整**：
- "B2 不是 Bug 是新功能" → 重新分类
- "F1 优先级最高先做" → 调整顺序
- "O1 不做了" → 移除

### S1: 批量 Bug 修复

创建 Bug 修复分支（如 `git.branch_strategy` 开启）：
```bash
git checkout -b fix/iteration-{迭代标识}
```

逐个执行快速通道（场景四），但**共享一轮回归测试**：

```
对于每个 Bug：
  1. 定位根因
  2. 制定修复方案 → 用户确认
  3. 修复 + 单元测试
  4. 验证修复结果
  5. Git checkpoint（在 fix 分支上）

所有 Bug 修完后：
  → 统一回归测试（Smoke 级别，覆盖修复涉及的功能）
  → 缺陷回溯学习（批量执行，涉及隐性知识的写入领域规则）
  → 测试通过后 squash merge 回主分支
```

### S2-S4: 合并执行新功能 + 优化

创建功能分支（如 `git.branch_strategy` 开启）：
```bash
git checkout -b feat/iteration-{迭代标识}
```

将所有新功能和需要走完整流程的优化**合并为一批**，统一执行：

```
S2: 需求分析 Skill — 所有新功能一起分析，生成各自的 feat-*.md
S3: 技术方案 Skill — 统一做模块拆分和设计，共享架构决策
S4: 开发实现 Skill — 按模块依赖顺序逐模块编码（每个模块 checkpoint）
```

> 合并执行的好处：共享一轮需求确认和设计确认，避免每个功能单独走一遍。

### S5: 统一测试

所有工作项（Bug 修复 + 新功能 + 优化）完成后，执行一轮统一测试：

```
1. Bug 修复回归（确认之前修的 Bug 没被新功能引入回退）
2. 新功能测试（完整 L1-L5）
3. 全量回归（Smoke 级别，覆盖核心路径）
4. 测试通过后 squash merge feat 分支回主分支
```

在 STATUS.md 中记录迭代级别的进度：

## 场景三：旧项目接入流程

### S0: 项目规模判断

首先通过顶层扫描（只读目录结构，不读业务代码）估算项目规模：

```bash
# 统计源码文件数（排除 node_modules / vendor / dist 等）
find . -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.go" -o -name "*.py" \) \
  | grep -v node_modules | grep -v vendor | grep -v dist | wc -l
```

根据结果向用户确认接入模式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
检测到旧项目，请选择接入模式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
源码文件数：约 {N} 个
模块数估算：约 {M} 个

模式 A — 完整接入（适合 ≤ 20 个模块的项目）
  一次性扫描全部代码，生成完整文档体系
  预估耗时：1-3 小时 | Token：~50K-100K
  适用：中小型项目，希望立即拥有完整文档

模式 B — 骨架接入（适合任意规模，推荐大型项目）
  只读目录结构和配置文件，生成轻量架构骨架
  后续开发时按需自动补全涉及模块的详细文档
  预估耗时：5-10 分钟 | Token：~3K-5K
  适用：大型项目，希望快速启动后渐进式补全文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择：A 或 B？
```

### 模式 A：完整接入（原有逻辑）

#### S0-A: 项目扫描与理解

1. 顶层扫描（只读目录结构 + 配置文件）→ 模块依赖图
2. 按优先级排序（核心业务优先）
3. 分批扫描（每批 3-5 模块，避免上下文溢出）
4. 合并与交叉验证

输出：模块依赖图摘要，写入 `.codeman/docs/design/arch-overview.md`

#### S1-A: 生成架构文档

基于扫描结果，生成：
- **模块清单**：每个模块的职责、入口文件、对外接口
- **依赖关系**：模块间调用关系图（文字版）
- **技术债务清单**：明显的代码异味、过时依赖、缺失测试等

写入 `.codeman/docs/design/arch-overview.md`（追加）

#### S2-A: 反向提炼业务 PRD

从代码实现反推现有功能点，建立 PRD 基准。

**执行步骤：**

1. 遍历各业务模块，识别对外暴露的功能（页面、API、核心逻辑）
2. 为每个识别到的功能点，生成 PRD 碎片文件：

   **文件路径：** `.codeman/docs/prd/feat-{功能点英文名}.md`

   **文件内容：**
   ```markdown
   # F{ID}: {功能点名称}（现有功能）

   ## 功能描述
   {从代码行为推断的功能描述}

   ## 现有实现入口
   - 文件：{主要实现文件路径}
   - 入口：{函数/组件/路由}

   ## 已知行为（验收基准）
   - [ ] {可观测的行为1}
   - [ ] {可观测的行为2}

   ## 已知缺陷 / 技术债
   - {如有，列出；无则写"暂无"}

   ## 状态
   existing（现有功能，非本次新增）
   ```

3. 更新 `.codeman/docs/prd/INDEX.md`，列出所有现有功能点

4. **人工确认**：输出功能点清单，询问用户：
   - 是否有遗漏的核心功能？
   - 是否有描述不准确的地方？
   - 哪些功能点是后续迭代的重点？

   ```
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   反向提炼 PRD 完成
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   已识别 X 个现有功能点：
     F01: {功能名} — {一句话描述}
     F02: {功能名} — {一句话描述}
     ...
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   请确认：有遗漏或描述有误的功能点吗？
   ```

5. 用户确认或补充后，锁定 PRD 基准

#### S2.5-A: 反向提炼技术文档

在 PRD 确认后，继续从代码中提炼技术类文档，与 PRD 确认**合并为一次人工确认**。

**design/ — 模块技术方案（`mod-*.md`）**

对每个核心业务模块，基于 `{CODEMAN_HOME}/templates/design/mod-template.md` 生成：

- 模块职责（从代码注释/文件名推断）
- 技术选型（从 package.json / go.mod / requirements.txt 读取）
- 核心数据模型（从代码中读取现有结构定义）
- 模块对外接口（函数签名 / 组件 Props / 路由定义）
- 已知技术债务（代码异味、TODO 注释等）

写入：`.codeman/docs/design/mod-{模块名}.md`

**api/ — API 接口文档（`api-*.md`）**

如果项目有后端接口（检测到路由文件），基于 `{CODEMAN_HOME}/templates/api/api-template.md` 生成：

- 每个路由的请求/响应结构（从路由代码读取）
- 鉴权方式（从中间件代码推断）
- 错误码说明（从错误处理代码读取）
- 对应 PRD 功能点映射

写入：`.codeman/docs/api/api-{模块名}.md`

**生成策略：**
- 只生成能从代码中读出来的内容，不猜测
- 不确定的字段填 `TODO: 待补充`，不留空
- 纯前端项目跳过 api/ 文档生成

**合并确认输出：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文档反向提炼完成（请一并确认）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
业务 PRD（X 个功能点）：
  F01: {功能名} — {一句话描述}
  F02: {功能名} — {一句话描述}

模块技术方案（Y 个模块）：
  mod-{模块名}.md — {模块职责}

API 文档（Z 个接口）：
  api-{模块名}.md — {接口数量} 个接口
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
有遗漏或描述有误的地方吗？确认后锁定文档基准。
```

完成后跳转到 **S3: 融入文档体系**。

### 模式 B：骨架接入（大型项目推荐）

**核心原则：** 接入时只建立「架构骨架」，不读业务代码细节。后续每次开发新需求时，自动按需补全涉及模块的详细文档（「接触即文档化」）。

#### S0-B: 生成架构骨架

**只读以下内容（不读业务代码）：**
- 顶层目录结构（`ls -la` 或 `tree -L 3`）
- 依赖配置文件（`package.json` / `go.mod` / `requirements.txt` / `pom.xml` 等）
- 入口文件（`main.ts` / `app.py` / `main.go` / `index.ts` 等，只读文件头部 30 行）
- 路由注册文件（如有，只读路由列表，不读路由实现）

基于以上内容，生成架构骨架文件：

**文件路径：** `.codeman/docs/design/arch-overview.md`

**使用骨架模板：** `{CODEMAN_HOME}/templates/design/arch-overview-skeleton.md`

骨架内容包含：
- 技术栈（从依赖配置读取）
- 模块清单（名称 + 职责一句话 + 主要文件路径，**不含数据模型和接口细节**）
- 模块依赖关系图（ASCII，从目录结构和 import 推断）
- 文档覆盖状态表（所有模块初始标记为 `⬜ 待按需生成`）

**文档覆盖状态表格式：**
```markdown
## 模块文档覆盖状态

| 模块 | 职责 | 详细文档 | 最后更新 |
|------|------|---------|---------|
| {模块名} | {一句话职责} | ⬜ 待按需生成 | - |
| {模块名} | {一句话职责} | ⬜ 待按需生成 | - |
```

> 说明：⬜ 表示尚未生成详细文档（mod-*.md），首次开发该模块时自动触发生成。

**Token 预估：** ~2K-4K（仅读目录和配置，不读业务代码）

#### S1-B: 生成 DIRECTIVES.md 关键约束

从依赖配置和入口文件推断，写入：
- 技术栈版本（前端框架、后端语言、数据库）
- 命名规范（从已有文件名推断：camelCase / snake_case / PascalCase）
- 已知高风险模块（如有明显标注，否则留空）
- 骨架接入标记：`legacy_mode: skeleton`（供 development Skill 识别）

#### S2-B: 人工确认骨架

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
骨架接入完成 — 请确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
识别到 {N} 个模块：
  {模块名} — {一句话职责}
  {模块名} — {一句话职责}
  ...

技术栈：{技术栈}
命名规范：{推断的命名规范}

⚠️  骨架模式：详细文档（mod-*.md / feat-*.md）将在
    首次开发各模块时自动生成，无需现在全量扫描。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块列表是否有遗漏或错误？确认后完成接入。
```

完成后跳转到 **S3: 融入文档体系**（骨架版）。

## 增量文档补全（CodeMan 同步）

**触发方式：** ① 自动 — orchestrator 启动时检测到外部代码变更；② 手动 — 用户说 `CodeMan 同步`

**核心原则：** 只补缺失的文档，不覆盖已有的。比"骨架接入"更轻量——不重建整个文档体系，只填补空白。

### 执行步骤

**Step 1: 扫描代码变更范围**

两种模式根据触发方式选择：

- **有 git 记录（自动触发）：** 分析他人提交涉及的文件，聚合为模块粒度的变更列表
  ```bash
  git log --since="{last_updated}" --not --author="{当前用户}" --name-only --pretty=format:""
  ```
- **无 git 记录（手动触发）：** 全量扫描项目代码目录结构，与已有的 design/INDEX.md、prd/INDEX.md 对比，找出未覆盖的模块/功能

**Step 2: 对比已有文档，识别缺失**

```
扫描结果（涉及的模块/文件）
    ↓
对比 design/INDEX.md → 哪些模块缺少 mod-*.md？
对比 prd/INDEX.md → 哪些功能缺少 feat-*.md？
对比 api/INDEX.md → 哪些接口缺少 api-*.md？
    ↓
输出缺失清单
```

**Step 3: 向用户确认补全范围**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CodeMan 文档同步
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
检测到以下代码变更尚无对应文档：

新增/变更模块（缺少技术方案）：
  - {模块名}（{涉及文件数} 个文件）
  - {模块名}（{涉及文件数} 个文件）

新增 API（缺少接口文档）：
  - {方法} {路径}（{文件:行号}）

新增功能（缺少 PRD）：
  - {功能描述}（推断自 {文件}）

已有文档无需更新：{N} 个模块
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认补全以上文档？（或指定只补某些模块）
```

**Step 4: 增量生成缺失文档**

用户确认后，仅对缺失部分执行反向提炼（复用场景三「旧项目接入」的文档生成逻辑）：

- 缺少 `mod-*.md` → 读取模块代码，生成技术方案（复用 mod-template.md，含「领域规则」章节）
- 缺少 `api-*.md` → 读取路由代码，生成接口文档
- 缺少 `feat-*.md` → 从代码行为反推 PRD 骨架
- 更新所有 INDEX.md 索引文件

**Step 5: 更新状态和提示**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文档同步完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
已补全：
  ✅ mod-{模块名}.md（技术方案）
  ✅ api-{模块名}.md（接口文档）
  ✅ feat-{功能名}.md（PRD 骨架）

⚠️ 请检查：
  - DOMAIN-RULES.md 是否需要补充新的领域规则？
  - 新模块的 mod-*.md「领域规则」章节是否需要补充？
  - DIRECTIVES.md 的需求精粹卡是否需要更新？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

更新 STATUS.md 的 `last_updated` 时间戳。

> **Token 成本：** 检测阶段 ~500 tokens（git log + INDEX 对比）。补全阶段取决于缺失文档数量，每个模块约 ~3K-5K tokens。

## 不确定即停（全局硬规则，所有 Skill 必须遵守）

**核心原则：宁可暂停问用户，绝不自作主张。这是贯穿所有阶段的铁律。**

以下情况在**任何阶段**遇到时，必须立即暂停并向用户确认，**严禁自行推测、假设、补充或做出任何未经确认的决定**：

| # | 场景 | 说明 |
|---|------|------|
| 1 | 用户意图不明确 | 需求描述、指令、反馈存在多种理解方式 |
| 2 | 需求/方案/用例有歧义 | 同一句话可以有不同解释 |
| 3 | 业务规则不明确 | 如"适当的权限控制"、"合理的分页大小"等模糊表述 |
| 4 | 技术选型有多个可行方案 | 不确定选哪个时必须列出方案让用户决策 |
| 5 | 发现 PRD/方案/代码有矛盾 | 内容之间或内容与用户描述存在冲突 |
| 6 | 需要引入未明确约定的内容 | PRD 未提及的逻辑、未约定的第三方库、未定义的数据结构 |
| 7 | 边界场景处理方式不确定 | 不确定该怎么处理异常、边界、特殊情况 |
| 8 | 测试用例的预期结果不确定 | 不确定某个输入应该产生什么输出 |
| 9 | 修复方案有风险或不确定 | 修复可能引入新问题或不确定是否正确 |
| 10 | 项目状态异常 | 文件缺失、格式错误、进度不一致 |

**确认格式（统一使用）：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
关于"{不确定点}"，我理解到以下几种可能：

1. {理解A}：[对应的实现方式/影响]
2. {理解B}：[对应的实现方式/影响]

我的建议：{推荐方案}，原因：{简要理由}

请问您的意思是哪种？或者请补充说明。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**禁止行为：**
- ❌ 不确定时自行选择"最合理"的方案继续
- ❌ 用 `[UNCERTAIN]` 标记后不暂停直接继续
- ❌ 猜测用户意图后执行
- ❌ 在确认前生成文件或修改代码

## Git 操作限制（全局硬规则，所有 Skill 必须遵守）

**核心原则：AI 可以 commit，绝对不能 push。推送代码到远程仓库是用户的专属权利。**

| 操作 | 权限 | 说明 |
|------|------|------|
| `git status` | ✅ 允许 | 只读操作 |
| `git diff` | ✅ 允许 | 只读操作 |
| `git log` | ✅ 允许 | 只读操作 |
| `git checkout -b {branch}` | ✅ 允许 | 创建 feature 分支（仅在分支策略开启时） |
| `git checkout {branch}` | ✅ 允许 | 切换分支（仅在分支策略开启时） |
| `git merge --squash {branch}` | ✅ 允许 | 测试通过后 squash merge（仅在分支策略开启时） |
| `git branch -d {branch}` | ✅ 允许 | 删除已合并的 feature 分支 |
| `git add` | ✅ 允许 | 暂存文件 |
| `git commit -m "..."` | ✅ 允许 | 必须写明 commit message，遵循 Conventional Commits |
| `git push` | ❌ **绝对禁止** | 推送操作必须由用户手动执行 |
| `git push --force` | ❌ **绝对禁止** | 任何变体都不允许 |
| `git push -u origin ...` | ❌ **绝对禁止** | 包含 -u 参数的也不允许 |
| `git reset --hard` | ❌ 禁止 | 破坏性操作 |
| `git rebase` | ❌ 禁止 | 可能导致冲突 |

**commit 规范：**
- message 必须遵循 Conventional Commits 格式：`type(scope): description`
- type 包含：feat / fix / docs / style / refactor / test / chore
- body 说明变更原因，footer 引用 PRD 功能点 ID

## 场景零：项目初始化流程

用户说「CodeMan 初始化」时执行，无论新项目还是旧项目都从这里开始。

### 执行步骤

**1. 运行 init.sh**

使用 Shell 工具执行初始化脚本（根据当前宿主环境选择路径）：

```bash
# Cursor 环境
bash ~/.cursor/skills/.codeman/init.sh

# Claude Code 环境
bash ~/.claude/skills/.codeman/init.sh
```

> 脚本在当前项目目录（工作区根目录）执行，会自动检测技术栈、生成 .codeman/ 目录结构和规范文件。
> 如不确定当前环境，可用 `{CODEMAN_HOME}/init.sh`（CODEMAN_HOME 见本文件顶部说明）。

**2. 读取 init.sh 输出的项目摘要**

init.sh 完成后会输出一段结构化摘要，格式如下：

```
CODEMAN_INIT_RESULT
project_type: new|legacy
has_source_code: true|false
tech_stack: {技术栈}
project_name: {项目名}
```

**3. 根据项目类型自动衔接**

- `project_type: new`（无业务代码）→ 直接进入**场景一**（全新项目开发流程）
- `project_type: legacy`（有业务代码）→ 直接进入**场景三**（旧项目接入流程）

**4. 向用户确认**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CodeMan 初始化完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
项目类型：[新项目 / 旧项目]
技术栈：[检测到的技术栈]
下一步：[场景一：开始需求分析 / 场景三：扫描现有代码库]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认继续？
```

## 场景一初始化步骤（S0）

当判定为全新项目时，执行初始化：

1. 询问项目信息：
   - 项目名称
   - 技术栈（前端框架、后端语言、数据库等）
   - 测试框架偏好
   - 是否启用 UI 视觉测试（Midscene.js）

2. 生成 `.codeman/` 目录结构：
   ```
   .codeman/
   ├── config.yaml              ← 根据用户选择填充
   ├── docs/
   │   ├── STATUS.md            ← 初始化状态
   │   ├── DIRECTIVES.md        ← 空白，待填充
   │   ├── PROJECT-OVERVIEW.md  ← 项目概览（面向新成员，各阶段自动更新）
   │   ├── prd/INDEX.md
   │   ├── design/INDEX.md
   │   ├── api/INDEX.md
   │   ├── tests/INDEX.md
   │   ├── reviews/INDEX.md
   │   └── deploy/INDEX.md
   ├── skills/INDEX.md
   └── rules/INDEX.md
   ```

3. 根据技术栈复制适用的规范模板到 `.codeman/rules/`

4. **S0.x: 处理待生成规范（PENDING-RULES.md）**

   检查 `.codeman/rules/PENDING-RULES.md` 是否存在：

   - 如果**不存在**：跳过此步骤
   - 如果**存在**：读取文件，找出所有 `- [ ]` 开头的未完成条目，逐一生成规范文件

   **对每个待生成条目，执行以下操作：**

   a. 读取 `.codeman/config.yaml` 中的技术栈信息（`tech_stack` 字段）

   b. 根据技术栈和规范名称，生成 `.mdc` 规范文件内容，格式要求：
      ```
      ---
      description: "{规范描述}。当{触发条件}时自动加载。"
      globs: "{适用的文件 glob 模式}"
      alwaysApply: false
      ---

      # {规范名称}（L2 项目层）

      ## {规范类别1}
      - {具体可操作的规范条目}

      ## 禁止
      - 禁止{禁止行为}
      ```
      质量约束：< 50 行，聚焦单一关注点，条目必须具体可操作

   c. 写入 `.codeman/rules/{filename}`（filename 来自条目，如 `proj-svelte-component.mdc`）

   d. 同步到 `.cursor/rules/codeman-{filename}`

   e. 将 PENDING-RULES.md 中对应条目从 `- [ ]` 改为 `- [x]`，并在行尾追加 `（已生成）`

   f. 更新 `.codeman/rules/INDEX.md`，追加新规范条目

   **完成后输出摘要：**
   ```
   已动态生成 X 条规范：
     ✓ proj-svelte-component.mdc — Svelte 组件开发规范
     ✓ proj-rust-backend.mdc — Rust 后端开发规范
   已同步到 .cursor/rules/
   ```

5. 同步 Rules 到 `.cursor/rules/`（加 `codeman-` 前缀）

6. 更新 STATUS.md：`phase: requirements`

7. 调度需求分析 Skill：Read `{CODEMAN_HOME}/skills/requirements/SKILL.md` 并按其指令执行

## 特殊：生成/更新项目概览

用户说「CodeMan 概览」「项目概览」「项目总览」时执行。

### 执行步骤

**1. 读取所有碎片文档**

```
读取以下文件（存在则读取，不存在则跳过）：
  - .codeman/config.yaml
  - .codeman/docs/STATUS.md
  - .codeman/docs/DIRECTIVES.md
  - .codeman/docs/design/arch-overview.md
  - .codeman/docs/prd/INDEX.md
  - .codeman/docs/prd/feat-*.md（所有 PRD 碎片）
  - .codeman/docs/design/INDEX.md
  - .codeman/docs/design/mod-*.md（所有模块方案）
  - .codeman/docs/api/INDEX.md
  - .codeman/docs/api/api-*.md（所有 API 文档）
  - .codeman/docs/reviews/INDEX.md
  - .codeman/docs/deploy/INDEX.md
  - .codeman/rules/ 目录下的 .mdc 文件列表
```

**2. 汇总生成 PROJECT-OVERVIEW.md**

基于 `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md` 模板，用自然语言填充每个章节。**必须内联核心摘要内容，不能只放链接**：

- **项目简介**：从 config.yaml + PRD 碎片中提炼 2-3 段：做什么、给谁用、核心价值
- **技术架构**：从 config.yaml + arch-overview.md 汇总技术栈表（含选型理由）+ 2-3 段架构描述
- **功能地图**：按功能分类组织，每个功能附**一句话核心需求摘要**和**关键验收标准**，不只是表格行
- **模块结构**：每个模块附**一句话关键设计说明**（如"采用 JWT 无状态鉴权"），不是空表格
- **API 概览**：每个接口附**核心请求/响应字段摘要**（如"返回 user_id, name, email"），不是空表格
- **关键决策记录**：每个决策用 1-2 句话描述**决策内容和原因**，不是空列表
- **开发规范速查**：每条规范写**具体要点**（如"函数 ≤ 50 行，复杂度 ≤ 10"），不是空占位符
- **当前进度**：从 STATUS.md 提取，附待处理事项
- **新成员上手指南**：根据技术栈填充实际的安装和启动命令

**生成原则：**
- **内联内容优先**：每个章节必须包含可读的核心摘要，不能只放 `→ 详见 xxx.md` 链接
- **链接作为补充**：每个章节末尾附 `→ 详见 xxx.md` 链接，供需要深入阅读的人使用
- **可独立阅读**：新成员只读本文档就能理解项目做了什么、用了什么技术、进度如何
- 用自然语言描述，不保留 `{{}}` 模板占位符
- 总量控制在 250 行以内（比原来增加 50 行，用于内联内容）
- 尚未生成的章节标注「待后续阶段自动补全」

**3. 写入文件**

写入 `.codeman/docs/PROJECT-OVERVIEW.md`（覆盖已有内容）。

**4. 输出确认**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
项目概览已更新
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 .codeman/docs/PROJECT-OVERVIEW.md
包含：项目简介 | 技术架构 | 功能地图（{N} 个功能点）
      模块结构（{M} 个模块）| API 概览 | 关键决策
      开发规范 | 当前进度 | 新成员上手指南
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
新成员阅读此文档即可在 5 分钟内了解项目全貌。
```
