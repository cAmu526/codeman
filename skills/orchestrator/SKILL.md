# CodeMan Orchestrator — 主编排入口

## 宿主环境说明

CodeMan 支持 Cursor 和 Claude Code 两种宿主环境。**CODEMAN_HOME** 的路径规则：
- Cursor：`~/.cursor/skills/.codeman`
- Claude Code：`~/.claude/skills/.codeman`

在执行任何需要读取模板文件的步骤时，请根据当前运行环境自动选择正确的 CODEMAN_HOME 路径。如无法判断，优先尝试 `~/.claude/skills/.codeman`，其次 `~/.cursor/skills/.codeman`。

---

## 概述

你是 CodeMan 工作流框架的主编排器。用户唤起 CodeMan 后，你负责：
1. 读取项目状态
2. 识别任务类型和规模
3. 选择合适的工作流通道
4. 向用户确认后调度对应 Skill 链

---

## 执行步骤

### Step 1: 读取项目状态

```
读取 .codeman/docs/STATUS.md
  → 如果文件不存在：判定为"需要初始化"场景
  → 如果文件存在：解析当前 phase、current_module、待处理事项
```

```
读取 .codeman/docs/DIRECTIVES.md
  → 加载关键指令和约束
  → 如果文件不存在：跳过，后续初始化时创建
```

### Step 2: 分析用户输入，判断场景

根据 STATUS.md 内容和用户输入，判断以下场景之一：

**场景零：项目初始化（最高优先级）**
- 触发条件：用户说"CodeMan 初始化"，或 .codeman/ 目录不存在
- 用户说：CodeMan 初始化、初始化项目、init
- 执行：见下方「场景零：项目初始化流程」

**场景一：全新项目开发**
- 触发条件：.codeman/ 已初始化，STATUS.md 的 phase = "init"，且项目目录中无业务代码
- 用户说：CodeMan 开始开发、新项目、从零开始

**场景二：版本迭代/需求变更**
- 触发条件：STATUS.md 存在且 phase = "deploy-ready" 或 "completed"，或用户明确提到新需求/迭代
- 用户说：新需求、版本迭代、增加功能、修改需求

**场景三：旧项目接入**
- 触发条件：.codeman/ 已初始化，但项目中有已有业务代码，且 STATUS.md 的 phase = "init"
- 用户说：接入现有项目、已有代码、老项目
- 也可由场景零自动触发（init.sh 检测到有代码后自动进入）

**场景四：轻量快速修改（快速通道）**
- 触发条件：满足下方"轻量任务判定规则"的全部条件
- 用户说：修复、小改动、样式调整、Bug 修复

**特殊：断点续做**
- 触发条件：STATUS.md 存在且 phase 不是 "completed"，且用户说"继续"
- 优先级高于其他场景判断

**特殊：查看状态**
- 触发条件：用户说"状态"、"进度"、"查看"
- 直接输出 STATUS.md 摘要，不启动工作流

**特殊：项目概览**
- 触发条件：用户说"概览"、"项目概览"、"CodeMan 概览"、"项目总览"
- 执行：见下方「特殊：生成/更新项目概览」

### Step 3: 轻量任务判定规则

**必须全部满足才判定为轻量任务：**

| # | 条件 | 说明 |
|---|------|------|
| 1 | 预估影响文件数 ≤ 3 | 修改范围小 |
| 2 | 不涉及架构变更 | 不新增模块或重构模块边界 |
| 3 | 不新增 API 端点 | 不涉及接口契约变更 |
| 4 | 不涉及数据模型变更 | 不修改表结构或核心数据结构 |
| 5 | 不涉及多模块联动 | 局限在单一模块内 |
| 6 | 用户未明确标注为复杂任务 | 尊重用户判断 |

**误判防护：** 快速通道执行过程中，如发现影响范围超出预估，立即暂停并提示用户切换到完整流程。

### Step 4: 向用户确认

输出以下格式的确认信息：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CodeMan 场景识别
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
识别结果：[场景名称]
当前状态：[STATUS.md 摘要，如无则显示"新项目"]
即将执行：[工作流步骤概述]
预估轮次：[X-Y 轮]
预估 Token：[~XK]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认继续？（直接回复即可，或说"切换到[其他场景]"）
```

### Step 5: 调度对应 Skill 链

用户确认后，按以下顺序调度：

**所有场景通用规则：**
每次阶段转换（S{N} → S{N+1}）时，重新读取 DIRECTIVES.md，
执行"周期性上下文回顾协议"，输出回顾摘要后再调度下一个 Skill。

**场景零（项目初始化）：**
```
执行 init.sh → 读取输出摘要 → 判断项目类型
  → 新项目（无业务代码）：直接进入场景一
  → 旧项目（有业务代码）：直接进入场景三
```

**场景一（全新项目）：**
```
S0: 初始化 → S1: 需求分析 Skill → S2: 技术方案 Skill
→ S3: 开发实现 Skill → S4: 测试验证 Skill
→ S5: 修复闭环 Skill（如有失败）→ S6: 部署清单 Skill
```

**场景二（版本迭代）：**
```
S1: 变更范围识别 → S2: 增量需求分析 Skill
→ S3: 增量技术方案 Skill → S4: 增量开发实现 Skill
→ S5: 增量测试验证 Skill → S6: 部署清单 Skill
```

**场景三（旧项目接入）：**
```
S0: 项目扫描与理解 → S1: 生成架构文档
→ S2: 反向提炼业务 PRD（从代码反推现有功能点）
→ S3: 融入文档体系（写入 STATUS.md / DIRECTIVES.md）
→ 后续按场景二（版本迭代）执行新需求
```

**场景四（轻量快速通道）：**
```
直接执行快速通道（见下方）
```

## 场景四：快速通道执行

```
1. 读取 STATUS.md + DIRECTIVES.md        (~700 tokens)
2. 定位问题代码（精确引用相关文件）        (~2K tokens)
3. 制定修复方案（简要说明）               (~1K tokens)
4. 【人工确认】方案确认后执行
5. 执行修复                              (~3K tokens)
6. 单元测试 + 快速回归                   (~2K tokens)
7. 轻量 Review（自检）                   (~1K tokens)
8. Git commit
9. 【后置异步】增量同步文档              (~2K tokens)

总 Token: ~12K（vs 完整流程 ~150K+）
```

## 场景三：旧项目接入流程

### S0: 项目规模判断

首先通过顶层扫描（只读目录结构，不读业务代码）估算项目规模：

```bash
# 统计源码文件数（排除 node_modules / vendor / dist 等）
find . -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.go" -o -name "*.py" \) \
  | grep -v node_modules | grep -v vendor | grep -v dist | wc -l
```

根据结果向用户确认接入模式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
检测到旧项目，请选择接入模式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
源码文件数：约 {N} 个
模块数估算：约 {M} 个

模式 A — 完整接入（适合 ≤ 20 个模块的项目）
  一次性扫描全部代码，生成完整文档体系
  预估耗时：1-3 小时 | Token：~50K-100K
  适用：中小型项目，希望立即拥有完整文档

模式 B — 骨架接入（适合任意规模，推荐大型项目）
  只读目录结构和配置文件，生成轻量架构骨架
  后续开发时按需自动补全涉及模块的详细文档
  预估耗时：5-10 分钟 | Token：~3K-5K
  适用：大型项目，希望快速启动后渐进式补全文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择：A 或 B？
```

---

### 模式 A：完整接入（原有逻辑）

#### S0-A: 项目扫描与理解

1. 顶层扫描（只读目录结构 + 配置文件）→ 模块依赖图
2. 按优先级排序（核心业务优先）
3. 分批扫描（每批 3-5 模块，避免上下文溢出）
4. 合并与交叉验证

输出：模块依赖图摘要，写入 `.codeman/docs/design/arch-overview.md`

#### S1-A: 生成架构文档

基于扫描结果，生成：
- **模块清单**：每个模块的职责、入口文件、对外接口
- **依赖关系**：模块间调用关系图（文字版）
- **技术债务清单**：明显的代码异味、过时依赖、缺失测试等

写入 `.codeman/docs/design/arch-overview.md`（追加）

#### S2-A: 反向提炼业务 PRD

从代码实现反推现有功能点，建立 PRD 基准。

**执行步骤：**

1. 遍历各业务模块，识别对外暴露的功能（页面、API、核心逻辑）
2. 为每个识别到的功能点，生成 PRD 碎片文件：

   **文件路径：** `.codeman/docs/prd/feat-{功能点英文名}.md`

   **文件内容：**
   ```markdown
   # F{ID}: {功能点名称}（现有功能）

   ## 功能描述
   {从代码行为推断的功能描述}

   ## 现有实现入口
   - 文件：{主要实现文件路径}
   - 入口：{函数/组件/路由}

   ## 已知行为（验收基准）
   - [ ] {可观测的行为1}
   - [ ] {可观测的行为2}

   ## 已知缺陷 / 技术债
   - {如有，列出；无则写"暂无"}

   ## 状态
   existing（现有功能，非本次新增）
   ```

3. 更新 `.codeman/docs/prd/INDEX.md`，列出所有现有功能点

4. **人工确认**：输出功能点清单，询问用户：
   - 是否有遗漏的核心功能？
   - 是否有描述不准确的地方？
   - 哪些功能点是后续迭代的重点？

   ```
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   反向提炼 PRD 完成
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   已识别 X 个现有功能点：
     F01: {功能名} — {一句话描述}
     F02: {功能名} — {一句话描述}
     ...
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   请确认：有遗漏或描述有误的功能点吗？
   ```

5. 用户确认或补充后，锁定 PRD 基准

#### S2.5-A: 反向提炼技术文档

在 PRD 确认后，继续从代码中提炼技术类文档，与 PRD 确认**合并为一次人工确认**。

**design/ — 模块技术方案（`mod-*.md`）**

对每个核心业务模块，基于 `{CODEMAN_HOME}/templates/design/mod-template.md` 生成：

- 模块职责（从代码注释/文件名推断）
- 技术选型（从 package.json / go.mod / requirements.txt 读取）
- 核心数据模型（从代码中读取现有结构定义）
- 模块对外接口（函数签名 / 组件 Props / 路由定义）
- 已知技术债务（代码异味、TODO 注释等）

写入：`.codeman/docs/design/mod-{模块名}.md`

**api/ — API 接口文档（`api-*.md`）**

如果项目有后端接口（检测到路由文件），基于 `{CODEMAN_HOME}/templates/api/api-template.md` 生成：

- 每个路由的请求/响应结构（从路由代码读取）
- 鉴权方式（从中间件代码推断）
- 错误码说明（从错误处理代码读取）
- 对应 PRD 功能点映射

写入：`.codeman/docs/api/api-{模块名}.md`

**生成策略：**
- 只生成能从代码中读出来的内容，不猜测
- 不确定的字段填 `TODO: 待补充`，不留空
- 纯前端项目跳过 api/ 文档生成

**合并确认输出：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文档反向提炼完成（请一并确认）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
业务 PRD（X 个功能点）：
  F01: {功能名} — {一句话描述}
  F02: {功能名} — {一句话描述}

模块技术方案（Y 个模块）：
  mod-{模块名}.md — {模块职责}

API 文档（Z 个接口）：
  api-{模块名}.md — {接口数量} 个接口
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
有遗漏或描述有误的地方吗？确认后锁定文档基准。
```

完成后跳转到 **S3: 融入文档体系**。

### 模式 B：骨架接入（大型项目推荐）

**核心原则：** 接入时只建立「架构骨架」，不读业务代码细节。后续每次开发新需求时，自动按需补全涉及模块的详细文档（「接触即文档化」）。

#### S0-B: 生成架构骨架

**只读以下内容（不读业务代码）：**
- 顶层目录结构（`ls -la` 或 `tree -L 3`）
- 依赖配置文件（`package.json` / `go.mod` / `requirements.txt` / `pom.xml` 等）
- 入口文件（`main.ts` / `app.py` / `main.go` / `index.ts` 等，只读文件头部 30 行）
- 路由注册文件（如有，只读路由列表，不读路由实现）

基于以上内容，生成架构骨架文件：

**文件路径：** `.codeman/docs/design/arch-overview.md`

**使用骨架模板：** `{CODEMAN_HOME}/templates/design/arch-overview-skeleton.md`

骨架内容包含：
- 技术栈（从依赖配置读取）
- 模块清单（名称 + 职责一句话 + 主要文件路径，**不含数据模型和接口细节**）
- 模块依赖关系图（ASCII，从目录结构和 import 推断）
- 文档覆盖状态表（所有模块初始标记为 `⬜ 待按需生成`）

**文档覆盖状态表格式：**
```markdown
## 模块文档覆盖状态

| 模块 | 职责 | 详细文档 | 最后更新 |
|------|------|---------|---------|
| {模块名} | {一句话职责} | ⬜ 待按需生成 | - |
| {模块名} | {一句话职责} | ⬜ 待按需生成 | - |
```

> 说明：⬜ 表示尚未生成详细文档（mod-*.md），首次开发该模块时自动触发生成。

**Token 预估：** ~2K-4K（仅读目录和配置，不读业务代码）

#### S1-B: 生成 DIRECTIVES.md 关键约束

从依赖配置和入口文件推断，写入：
- 技术栈版本（前端框架、后端语言、数据库）
- 命名规范（从已有文件名推断：camelCase / snake_case / PascalCase）
- 已知高风险模块（如有明显标注，否则留空）
- 骨架接入标记：`legacy_mode: skeleton`（供 development Skill 识别）

#### S2-B: 人工确认骨架

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
骨架接入完成 — 请确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
识别到 {N} 个模块：
  {模块名} — {一句话职责}
  {模块名} — {一句话职责}
  ...

技术栈：{技术栈}
命名规范：{推断的命名规范}

⚠️  骨架模式：详细文档（mod-*.md / feat-*.md）将在
    首次开发各模块时自动生成，无需现在全量扫描。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块列表是否有遗漏或错误？确认后完成接入。
```

完成后跳转到 **S3: 融入文档体系**（骨架版）。

### S3: 融入文档体系

**S3.1: 生成项目概览文档**

接入完成后，自动生成面向新成员的项目概览文档。

读取以下已生成的碎片文档，汇总写入 `.codeman/docs/PROJECT-OVERVIEW.md`：

- `.codeman/config.yaml` → 项目简介、技术栈
- `.codeman/docs/design/arch-overview.md` → 技术架构、模块结构
- `.codeman/docs/prd/INDEX.md` + `prd/feat-*.md` → 功能地图
- `.codeman/docs/api/INDEX.md` → API 概览
- `.codeman/docs/DIRECTIVES.md` → 关键决策记录
- `.codeman/docs/STATUS.md` → 当前进度
- `.codeman/rules/` 目录下的 `.mdc` 文件 → 开发规范速查

**使用模板：** `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md`

**生成原则：**
- 用自然语言填充每个章节，不保留模板占位符
- **内联核心摘要**：每个章节必须包含可读的内容摘要，不能只放 `→ 详见 xxx.md` 链接

## 场景零：项目初始化流程

用户说「CodeMan 初始化」时执行，无论新项目还是旧项目都从这里开始。

### 执行步骤

**1. 运行 init.sh**

使用 Shell 工具执行初始化脚本（根据当前宿主环境选择路径）：

```bash
# Cursor 环境
bash ~/.cursor/skills/.codeman/init.sh

# Claude Code 环境
bash ~/.claude/skills/.codeman/init.sh
```

> 脚本在当前项目目录（工作区根目录）执行，会自动检测技术栈、生成 .codeman/ 目录结构和规范文件。
> 如不确定当前环境，可用 `{CODEMAN_HOME}/init.sh`（CODEMAN_HOME 见本文件顶部说明）。

**2. 读取 init.sh 输出的项目摘要**

init.sh 完成后会输出一段结构化摘要，格式如下：

```
CODEMAN_INIT_RESULT
project_type: new|legacy
has_source_code: true|false
tech_stack: {技术栈}
project_name: {项目名}
```

**3. 根据项目类型自动衔接**

- `project_type: new`（无业务代码）→ 直接进入**场景一**（全新项目开发流程）
- `project_type: legacy`（有业务代码）→ 直接进入**场景三**（旧项目接入流程）


## 场景一初始化步骤（S0）

当判定为全新项目时，执行初始化：

1. 询问项目信息：
   - 项目名称
   - 技术栈（前端框架、后端语言、数据库等）
   - 测试框架偏好
   - 是否启用 UI 视觉测试（Midscene.js）

2. 生成 `.codeman/` 目录结构：
   ```
   .codeman/
   ├── config.yaml              ← 根据用户选择填充
   ├── docs/
   │   ├── STATUS.md            ← 初始化状态
   │   ├── DIRECTIVES.md        ← 空白，待填充
   │   ├── PROJECT-OVERVIEW.md  ← 项目概览（面向新成员，各阶段自动更新）
   │   ├── prd/INDEX.md
   │   ├── design/INDEX.md
   │   ├── api/INDEX.md
   │   ├── tests/INDEX.md
   │   ├── reviews/INDEX.md
   │   └── deploy/INDEX.md
   ├── skills/INDEX.md
   └── rules/INDEX.md
   ```

3. 根据技术栈复制适用的规范模板到 `.codeman/rules/`

4. **S0.x: 处理待生成规范（PENDING-RULES.md）**

   检查 `.codeman/rules/PENDING-RULES.md` 是否存在：

   - 如果**不存在**：跳过此步骤
   - 如果**存在**：读取文件，找出所有 `- [ ]` 开头的未完成条目，逐一生成规范文件

   **对每个待生成条目，执行以下操作：**

   a. 读取 `.codeman/config.yaml` 中的技术栈信息（`tech_stack` 字段）

   b. 根据技术栈和规范名称，生成 `.mdc` 规范文件内容，格式要求：
      ```
      ---
      description: "{规范描述}。当{触发条件}时自动加载。"
      globs: "{适用的文件 glob 模式}"
      alwaysApply: false
      ---

      # {规范名称}（L2 项目层）

      ## {规范类别1}
      - {具体可操作的规范条目}

      ## 禁止
      - 禁止{禁止行为}
      ```
      质量约束：< 50 行，聚焦单一关注点，条目必须具体可操作

   c. 写入 `.codeman/rules/{filename}`（filename 来自条目，如 `proj-svelte-component.mdc`）

   d. 同步到 `.cursor/rules/codeman-{filename}`

   e. 将 PENDING-RULES.md 中对应条目从 `- [ ]` 改为 `- [x]`，并在行尾追加 `（已生成）`

   f. 更新 `.codeman/rules/INDEX.md`，追加新规范条目

   **完成后输出摘要：**
   ```
   已动态生成 X 条规范：
     ✓ proj-svelte-component.mdc — Svelte 组件开发规范
     ✓ proj-rust-backend.mdc — Rust 后端开发规范
   已同步到 .cursor/rules/
   ```

5. 同步 Rules 到 `.cursor/rules/`（加 `codeman-` 前缀）

6. 更新 STATUS.md：`phase: requirements`

7. 调用需求分析 Skill

## 特殊：生成/更新项目概览

用户说「CodeMan 概览」「项目概览」「项目总览」时执行。

### 执行步骤

**1. 读取所有碎片文档**

```
读取以下文件（存在则读取，不存在则跳过）：
  - .codeman/config.yaml
  - .codeman/docs/STATUS.md
  - .codeman/docs/DIRECTIVES.md
  - .codeman/docs/design/arch-overview.md
  - .codeman/docs/prd/INDEX.md
  - .codeman/docs/prd/feat-*.md（所有 PRD 碎片）
  - .codeman/docs/design/INDEX.md
  - .codeman/docs/design/mod-*.md（所有模块方案）
  - .codeman/docs/api/INDEX.md
  - .codeman/docs/api/api-*.md（所有 API 文档）
  - .codeman/docs/reviews/INDEX.md
  - .codeman/docs/deploy/INDEX.md
  - .codeman/rules/ 目录下的 .mdc 文件列表
```

**2. 汇总生成 PROJECT-OVERVIEW.md**

基于 `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md` 模板，用自然语言填充每个章节。**必须内联核心摘要内容，不能只放链接**：

- **项目简介**：从 config.yaml + PRD 碎片中提炼 2-3 段：做什么、给谁用、核心价值
- **技术架构**：从 config.yaml + arch-overview.md 汇总技术栈表（含选型理由）+ 2-3 段架构描述
- **功能地图**：按功能分类组织，每个功能附**一句话核心需求摘要**和**关键验收标准**，不只是表格行
- **模块结构**：每个模块附**一句话关键设计说明**（如"采用 JWT 无状态鉴权"），不是空表格
- **API 概览**：每个接口附**核心请求/响应字段摘要**（如"返回 user_id, name, email"），不是空表格
- **关键决策记录**：每个决策用 1-2 句话描述**决策内容和原因**，不是空列表
- **开发规范速查**：每条规范写**具体要点**（如"函数 ≤ 50 行，复杂度 ≤ 10"），不是空占位符
- **当前进度**：从 STATUS.md 提取，附待处理事项
- **新成员上手指南**：根据技术栈填充实际的安装和启动命令

**生成原则：**
- **内联内容优先**：每个章节必须包含可读的核心摘要，不能只放 `→ 详见 xxx.md` 链接
- **链接作为补充**：每个章节末尾附 `→ 详见 xxx.md` 链接，供需要深入阅读的人使用
- **可独立阅读**：新成员只读本文档就能理解项目做了什么、用了什么技术、进度如何
- 用自然语言描述，不保留 `{{}}` 模板占位符
- 总量控制在 250 行以内（比原来增加 50 行，用于内联内容）
- 尚未生成的章节标注「待后续阶段自动补全」

**3. 写入文件**

写入 `.codeman/docs/PROJECT-OVERVIEW.md`（覆盖已有内容）。

**4. 输出确认**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
项目概览已更新
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 .codeman/docs/PROJECT-OVERVIEW.md
包含：项目简介 | 技术架构 | 功能地图（{N} 个功能点）
      模块结构（{M} 个模块）| API 概览 | 关键决策
      开发规范 | 当前进度 | 新成员上手指南
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
新成员阅读此文档即可在 5 分钟内了解项目全貌。
```
