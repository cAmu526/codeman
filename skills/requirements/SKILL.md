# CodeMan 需求分析 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code 为 `~/.claude/skills/.codeman`。读取模板文件时请根据当前环境自动选择。

---

## 概述

将用户的原始需求描述转化为结构化的 PRD 碎片文件体系。核心流程：**拓展 → 收敛 → 确认 → 文档化**。

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束和关键指令
   → 如文件为空，继续执行

2. 读取 .codeman/docs/prd/INDEX.md
   → 了解已有需求（版本迭代时避免重复）
   → 如文件不存在，说明是全新项目

3. 读取 .codeman/config.yaml
   → 了解技术栈约束
```

---

## 执行步骤

### Step 1: 分析用户需求描述

仔细阅读用户提供的需求描述，识别：
- 明确提到的功能点
- 隐含的业务逻辑
- 可能遗漏的边界场景
- 潜在的技术约束

### Step 2: 头脑风暴拓展

从以下 6 个维度对需求进行拓展分析：

**① 用户体验维度**
- 用户操作流程是否顺畅？
- 是否有加载状态、空状态、错误状态的处理？
- 移动端适配是否需要考虑？

**② 边界场景维度**
- 极端输入（超长字符串、特殊字符、空值）
- 并发场景（多用户同时操作）
- 网络异常（超时、断网、重试）
- 数据一致性（事务边界）

**③ 安全维度**
- 权限控制（谁能访问、谁能操作）
- 数据验证（输入校验、SQL 注入防护）
- 敏感数据处理（加密、脱敏）

**④ 性能维度**
- 数据量级估算（列表分页、大文件处理）
- 缓存策略需求
- 慢操作的异步处理

**⑤ 可维护性维度**
- 日志记录需求
- 监控告警需求
- 数据迁移/回滚方案

**⑥ 兼容性维度**
- 与已有功能的交互
- 向后兼容要求
- 第三方服务依赖

### Step 3: 分类标注

将所有功能点（原始 + 拓展）分类标注：

```
🔴 核心需求（必须实现）
   → 用户明确要求的功能
   → 缺少则产品无法使用

🟡 建议增强（推荐实现）
   → 显著提升用户体验
   → 防止常见问题

🟢 可选项（用户决定）
   → 锦上添花
   → 可延后到下个版本
```

### Step 4: 【人工确认点】展示分类结果

输出以下格式，等待用户逐条确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
需求分析结果 — 请逐条确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔴 核心需求（必须实现）
  F001: [功能点名称]
        描述：[简要描述]
        验收标准：[可量化的验收条件]

  F002: [功能点名称]
        ...

🟡 建议增强（推荐实现，可调整为核心或可选）
  F003: [功能点名称]
        描述：[简要描述]
        建议原因：[为什么推荐]

🟢 可选项（可延后实现）
  F004: [功能点名称]
        描述：[简要描述]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受全部
  - 或说"F003 移到核心"/"删除 F004"/"F001 修改为..."
  - 或补充遗漏的需求点
```

### Step 5: 处理用户反馈

根据用户反馈调整功能点列表：
- 接受调整并重新分类
- 如用户提出新需求，重新进行头脑风暴（仅针对新需求）
- 所有调整完成后，再次展示最终确认版本

### Step 5.5: 收集 UI 设计资产

用户确认功能点列表后，询问是否有设计资产：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
UI 设计资产收集
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
以下功能点涉及 UI：{列出涉及UI的功能点}

请提供设计截图（如有）：
  - 方式1：将截图文件放入 .codeman/assets/design/ 目录，告知文件名
  - 方式2：提供截图文件的本地路径
  - 方式3：回复"无设计稿"，后续基于文字描述开发

提供后我会在 PRD 中记录引用路径。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

将用户提供的设计资产路径写入对应 feat-*.md 的"设计资产引用"表格。
如用户回复"无设计稿"，在 feat-*.md 中标注"待设计稿确认"。

### Step 6: 生成 PRD 碎片文件

用户最终确认后，为每个功能点生成独立的 PRD 碎片文件：

**文件路径：** `.codeman/docs/prd/feat-{功能点英文名}.md`

**文件内容（使用 PRD 碎片模板）：**
```markdown
# F{ID}: {功能点名称}

> 状态：✅ 已确认 | 优先级：{P0/P1/P2} | 创建：{YYYY-MM-DD}

## 功能描述
{一段话描述功能目标}

## 功能点明细
| # | 子功能 | 验收标准 | 优先级 |
|---|--------|---------|--------|
| F{ID}.1 | {子功能} | {验收标准} | {P0/P1} |

## UI 设计要求
{从用户描述中提取的 UI 要求，如无则标注"待设计稿确认"}

## 业务规则
{关键业务约束和规则}

## 边界场景
{需要特殊处理的边界情况}

## 依赖关系
{依赖的其他功能点或外部服务}
```

### Step 7: 更新索引文件

更新 `.codeman/docs/prd/INDEX.md`：

```markdown
# PRD 功能点索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}
> 功能点总数：{N} | 已确认：{N} | 待确认：0

| ID | 功能点 | 文件 | 状态 | 优先级 | 最后更新 |
|----|--------|------|------|--------|---------|
| F001 | {功能点名称} | feat-{name}.md | ✅ 已确认 | P0 | {日期} |
```

### Step 8: 更新 DIRECTIVES.md

将 PRD 关键约束摘要写入 DIRECTIVES.md：

```markdown
## PRD 关键约束摘要
- [P{N}] {功能点名称}关键约束 → 详见 prd/feat-{name}.md
```

### Step 8.5: 生成需求精粹卡

在 DIRECTIVES.md 中填充"需求精粹卡"章节：

1. 从已确认的 PRD 碎片文件中提炼：
   - 产品目标（从所有功能点归纳一句话）
   - 核心功能清单（每个 P0/P1 功能点一行：ID + 名称 + 一句话验收标准）
   - 绝对约束（从"业务规则"和"边界场景"中提取不可违反的硬约束）
   - 关键验收标准（提取最关键的 3-5 条可量化验收条件）
   - 设计资产摘要（如有截图路径）

2. **严格 Token 限制：** 精粹卡总内容 ≤ 500 tokens
   - 如功能点过多，只保留 P0 和 P1
   - 每条用一句话概括，不展开描述

> 精粹卡是"需求的 TL;DR"，后续所有阶段中作为快速回忆锚点，详细信息以 PRD 碎片文件为准。

### Step 9: 更新 STATUS.md

```markdown
- phase: design
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（PRD 列标记 ✅）。

### Step 10: 更新项目概览文档

读取 `.codeman/docs/PROJECT-OVERVIEW.md`，更新以下章节：

**「项目简介」章节：**
- 从已确认的 PRD 碎片中提炼 2-3 句话的项目业务目标描述
- 包含核心价值主张和目标用户群体

**「功能地图」章节：**
- 从 `prd/INDEX.md` 汇总所有功能点到表格
- 每行包含：ID、功能名、优先级、状态、一句话描述、详情链接
- 状态根据 STATUS.md 中的功能点进度表同步

**「当前进度」章节：**
- 从 STATUS.md 提取最新进度快照

> 如果 PROJECT-OVERVIEW.md 不存在，基于 `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md` 模板创建，并填充当前已有信息，其余章节标注「待后续阶段自动补全」。

## 版本迭代时的增量处理

当 prd/INDEX.md 已有内容时：

1. 识别变更类型：
   - **新增功能点**：创建新的 feat-*.md
   - **修改已有功能点**：更新对应 feat-*.md，记录变更历史
   - **删除功能点**：标记为 `❌ 已废弃`，不删除文件（保留历史）

2. 分析变更影响范围：
   - 哪些已有功能点受到影响？
   - 是否需要修改已有技术方案？

3. 向用户展示影响范围分析，确认后继续

4. 更新"需求精粹卡"：
   - 新增功能点追加到核心功能清单
   - 修改功能点更新对应条目
   - 新增约束追加到绝对约束列表
   - 保持 ≤500 tokens，必要时精简旧条目

---

## 不确定即停

遇到以下情况必须暂停，严禁自行推测：

- 需求描述模糊，存在多种理解方式
- 业务规则不明确（如"适当的权限控制"）
- 与已有功能存在潜在冲突
- 技术可行性存疑

**暂停格式：**
```
[需人工确认] 关于"{模糊点}"，我理解到以下几种可能：
1. {理解A}：[对应的实现方式]
2. {理解B}：[对应的实现方式]

请问您的意思是哪种？或者请补充说明。
```

```
PRD 文档已生成到 .codeman/docs/prd/

下一步：技术方案设计
是否立即开始？
```
