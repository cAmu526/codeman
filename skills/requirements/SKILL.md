# CodeMan 需求分析 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。读取模板文件时请根据当前环境自动选择。

---

## 概述

将用户的原始需求描述转化为结构化的 PRD 碎片文件体系。核心流程含**三次交互确认**，确保需求完整性达到行业水平：

```
拓展 → 收敛 → 【确认①：功能点分类】 → 生成 PRD → 【确认②：PRD 内容审阅 + 完整性校验】
→ 生成精粹卡 → 【确认③：精粹卡】 → 文档化
```

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束和关键指令
   → 如文件为空，继续执行

2. 读取 .codeman/docs/prd/INDEX.md
   → 了解已有需求（版本迭代时避免重复）
   → 如文件不存在，说明是全新项目

3. 读取 .codeman/config.yaml
   → 了解技术栈约束
```

---

## 执行步骤

### Step 1: 分析用户需求描述

仔细阅读用户提供的需求描述，识别：
- 明确提到的功能点
- 隐含的业务逻辑
- 可能遗漏的边界场景
- 潜在的技术约束
- 目标用户角色（谁用这个功能？）
- 业务价值（为什么要做这个功能？）

### Step 2: 头脑风暴拓展

从以下 **8 个维度** 对需求进行拓展分析（比行业标准多 2 个维度，覆盖更全面）：

**① 用户故事维度**
- 作为 [角色]，我希望 [行为]，以便 [价值]
- 不同用户角色是否有不同的操作权限和视图？
- 用户操作的前置条件和后置条件是什么？

**② 用户体验维度**
- 用户操作流程是否顺畅？关键路径步数是否 ≤ 3 步？
- 是否有加载状态、空状态、错误状态的处理？
- 移动端适配是否需要考虑？
- 可用性：是否需要键盘快捷键、无障碍支持？

**③ 边界场景维度**
- 极端输入（超长字符串、特殊字符、空值、负数）
- 并发场景（多用户同时操作同一资源）
- 网络异常（超时、断网、重试、幂等性）
- 数据一致性（事务边界、部分失败回滚）
- 时序问题（过期数据、竞态条件）

**④ 安全维度**
- 权限控制（谁能访问、谁能操作、RBAC/ABAC）
- 数据验证（输入校验、XSS/SQL 注入防护）
- 敏感数据处理（加密存储、日志脱敏、传输安全）
- 审计追踪（关键操作是否需要记录日志）

**⑤ 性能维度**
- 数据量级估算（用户量、数据量、增长趋势）
- 列表分页策略（默认页大小、最大页大小）
- 缓存策略需求（什么数据需要缓存、缓存多久）
- 慢操作的异步处理（文件上传、报表生成、批量操作）

**⑥ 数据模型维度**
- 涉及哪些核心实体？实体间关系是什么？
- 数据的生命周期（创建 → 使用 → 归档/删除）
- 数据迁移需求（从旧系统迁移、版本升级）
- 数据备份和恢复策略

**⑦ 可维护性维度**
- 日志记录需求（哪些操作需要日志、日志级别）
- 监控告警需求（关键指标、告警阈值）
- 配置化需求（哪些参数需要可配置而非硬编码）

**⑧ 兼容性维度**
- 与已有功能的交互影响
- 向后兼容要求（API 版本管理）
- 第三方服务依赖（SLA、降级方案）
- 浏览器/设备兼容性要求

### Step 3: 分类标注

将所有功能点（原始 + 拓展）分类标注：

```
🔴 核心需求（必须实现）
   → 用户明确要求的功能
   → 缺少则产品无法使用
   → 对应用户故事中的核心价值

🟡 建议增强（推荐实现）
   → 显著提升用户体验
   → 防止常见问题
   → 行业通用最佳实践

🟢 可选项（用户决定）
   → 锦上添花
   → 可延后到下个版本
```

### Step 4: 【确认① — 功能点分类确认】

输出以下格式，等待用户逐条确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
需求分析结果 — 请逐条确认（第 1 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔴 核心需求（必须实现）
  F001: [功能点名称]
        用户故事：作为 [角色]，我希望 [行为]，以便 [价值]
        核心验收标准：[可量化的验收条件]
        边界场景：[关键边界]

  F002: [功能点名称]
        ...

🟡 建议增强（推荐实现，可调整为核心或可选）
  F003: [功能点名称]
        用户故事：作为 [角色]，我希望 [行为]，以便 [价值]
        建议原因：[为什么推荐]
        核心验收标准：[可量化的验收条件]

🟢 可选项（可延后实现）
  F004: [功能点名称]
        用户故事：作为 [角色]，我希望 [行为]，以便 [价值]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受全部
  - 或说"F003 移到核心"/"删除 F004"/"F001 修改为..."
  - 或补充遗漏的需求点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 5: 处理用户反馈

根据用户反馈调整功能点列表：
- 接受调整并重新分类
- 如用户提出新需求，重新进行头脑风暴（仅针对新需求）
- 所有调整完成后，再次展示最终确认版本
- 用户确认后进入下一步

### Step 5.5: 收集 UI 设计资产

用户确认功能点列表后，询问是否有设计资产：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
UI 设计资产收集
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
以下功能点涉及 UI：{列出涉及UI的功能点}

请提供设计截图（如有）：
  - 方式1：将截图文件放入 .codeman/assets/design/ 目录，告知文件名
  - 方式2：提供截图文件的本地路径
  - 方式3：回复"无设计稿"，后续基于文字描述开发

提供后我会在 PRD 中记录引用路径。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

将用户提供的设计资产路径写入对应 feat-*.md 的"设计资产引用"表格。
如用户回复"无设计稿"，在 feat-*.md 中标注"待设计稿确认"。

### Step 6: 生成 PRD 碎片文件

用户最终确认后，为每个功能点生成独立的 PRD 碎片文件：

**文件路径：** `.codeman/docs/prd/feat-{功能点英文名}.md`

**使用模板：** `{CODEMAN_HOME}/templates/prd/feat-template.md`

**PRD 质量要求（行业标准）：**

每个 PRD 碎片文件必须包含以下完整内容：

1. **用户故事**：明确 `作为 [角色]，我希望 [行为]，以便 [价值]`
2. **业务功能流程图**：主流程 + 关键分支（成功/失败/权限不足等）
   - 使用 ASCII（`→`、`{判断?}`、缩进分支）
   - 步骤须与「功能点明细」验收标准可对齐
3. **界面原型（ASCII 方框图）**：凡有页面/弹窗/表单，用 `┌─┐│└┘` 画出区域与控件
4. **功能点明细**：每个子功能有明确的验收标准，使用 Given-When-Then 格式
   ```
   Given [前置条件]
   When  [用户操作/系统事件]
   Then  [预期结果（可量化）]
   ```
5. **数据字典**：涉及的核心数据字段定义（字段名、类型、约束、说明）
6. **UI 设计要求**：布局、交互、设计资产引用
7. **业务规则**：关键约束和不可违反的规则
8. **边界场景**：需要特殊处理的边界情况（含预期系统行为）
9. **非功能需求**：性能指标、安全要求、可用性要求（如适用）
10. **依赖关系**：依赖的其他功能点或外部服务
11. **假设与约束**：需求分析时的假设条件和技术约束

### Step 6.5: 【确认② — PRD 内容审阅 + 完整性校验】

PRD 生成后，执行**两次确认**（合并为一个确认环节）：

#### 6.5a: 完整性自检报告

AI 自动检查每个 PRD 碎片文件的完整性，输出报告：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PRD 完整性自检报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

功能点：F001 {名称}

检查项                    状态    说明
────────────────────────────────────────
用户故事完整性            ✅     角色/行为/价值均明确
业务流程图覆盖度          ✅     主流程 + 3 个分支路径
验收标准可量化性          ✅     6 条标准均有具体数值
边界场景覆盖度            ⚠️     缺少"并发提交"场景
数据字典完整性            ✅     5 个核心字段已定义
非功能需求明确性          ⚠️     性能指标未量化（建议补充）
依赖关系清晰度            ✅     依赖 F002 已标注
假设与约束记录            ✅     2 条假设已记录
UI 设计要求              ✅     有 ASCII 原型 + 设计截图引用
业务规则完整性            ✅     3 条核心规则已定义

总体评分：8/10 ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**完整性评分标准：**
- 10/10：全部检查项通过，无任何遗漏
- 8-9/10：仅有 1-2 项"建议"级别缺失，不影响核心需求
- 6-7/10：有关键项缺失，需用户确认是否补充
- < 6/10：不达标，必须补充后才能进入下一步

**如评分 < 8，AI 必须输出具体的补充建议并询问用户是否补充。**

#### 6.5b: PRD 内容逐条审阅

输出每个 PRD 碎片文件的核心内容摘要，让用户审阅：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PRD 内容审阅 — 请逐条确认（第 2 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📄 F001: {功能名称}
  用户故事：{一句话}
  核心流程：{主流程关键步骤（3-5 步）}
  验收标准（{N} 条）：
    F001.1: {验收标准1}
    F001.2: {验收标准2}
    ...
  边界场景（{K} 个）：
    - {边界场景1} → 预期行为：{预期}
    - {边界场景2} → 预期行为：{预期}
  数据字段：{核心字段列表}
  非功能需求：{性能/安全指标，如有}

📄 F002: {功能名称}
  ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受全部
  - 或指出具体问题："F001 的验收标准第 2 条应该改为..."
  - 或补充遗漏的边界场景/业务规则
  - 或说"完整性检查 {功能ID}"重新检查某个功能点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户反馈处理：**
- 如用户指出问题，修改对应 PRD 碎片文件后重新展示
- 如用户补充场景，补充后重新执行完整性自检
- 如用户确认，进入下一步

### Step 7: 更新索引文件

更新 `.codeman/docs/prd/INDEX.md`：

```markdown
# PRD 功能点索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}
> 功能点总数：{N} | 已确认：{N} | 待确认：0

| ID | 功能点 | 文件 | 状态 | 优先级 | 最后更新 |
|----|--------|------|------|--------|---------|
| F001 | {功能点名称} | feat-{name}.md | ✅ 已确认 | P0 | {日期} |
```

### Step 8: 更新 DIRECTIVES.md

将 PRD 关键约束摘要写入 DIRECTIVES.md：

```markdown
## PRD 关键约束摘要
- [P{N}] {功能点名称}关键约束 → 详见 prd/feat-{name}.md
```

### Step 8.5: 生成需求精粹卡

在 DIRECTIVES.md 中填充"需求精粹卡"章节：

1. 从已确认的 PRD 碎片文件中提炼：
   - 产品目标（从所有功能点归纳一句话）
   - 核心功能清单（每个 P0/P1 功能点一行：ID + 名称 + 一句话验收标准）
   - 绝对约束（从"业务规则"和"边界场景"中提取不可违反的硬约束）
   - 关键验收标准（提取最关键的 3-5 条可量化验收条件）
   - 设计资产摘要（如有截图路径）

2. **严格 Token 限制：** 精粹卡总内容 ≤ 500 tokens
   - 如功能点过多，只保留 P0 和 P1
   - 每条用一句话概括，不展开描述

> 精粹卡是"需求的 TL;DR"，后续所有阶段中作为快速回忆锚点，详细信息以 PRD 碎片文件为准。

### Step 8.6: 【确认③ — 需求精粹卡确认】

精粹卡生成后，输出给用户确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
需求精粹卡 — 请确认（第 3 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

产品目标（一句话）：
{产品目标}

核心功能清单（{N} 个）：
  F001: {名称} — {一句话验收标准}
  F002: {名称} — {一句话验收标准}
  ...

绝对约束（不可违反）：
  - {约束1}
  - {约束2}

关键验收标准（{M} 条）：
  [A1] {验收标准1}
  [A2] {验收标准2}
  ...

Token 估算：~{X} tokens（上限 500）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受
  - 或指出："产品目标不够准确，应该改为..."
  - 或补充："绝对约束缺少 xxx"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户反馈处理：**
- 如用户修改，更新 DIRECTIVES.md 中的精粹卡后重新展示
- 如用户确认，精粹卡锁定，进入下一步

### Step 9: 更新 STATUS.md

```markdown
- phase: design
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（PRD 列标记 ✅）。

### Step 10: 更新项目概览文档

读取 `.codeman/docs/PROJECT-OVERVIEW.md`，更新以下章节（**必须内联核心摘要，不能只放链接**）：

**「项目简介」章节：**
- 从已确认的 PRD 碎片中提炼 2-3 段自然语言描述：做什么、给谁用、核心价值
- 包含核心价值主张和目标用户群体
- 如有设计截图引用，写入截图路径

**「功能地图」章节：**
- **按功能分类组织**（如"用户管理"、"订单系统"），每个分类附一句话功能目标
- 从 `prd/INDEX.md` 和各 `feat-*.md` 汇总：每个功能附**一句话核心需求摘要**（如"用户可通过邮箱+密码注册，需邮箱验证"）
- 关键验收标准（每个分类提取 1-2 条最关键的可量化条件）
- 每个分类末尾附 `→ 详见 prd/feat-{{name}}.md` 链接
- 状态根据 STATUS.md 中的功能点进度表同步

**「当前进度」章节：**
- 从 STATUS.md 提取最新进度快照，附待处理事项

> 如果 PROJECT-OVERVIEW.md 不存在，基于 `{CODEMAN_HOME}/templates/PROJECT-OVERVIEW.md` 模板创建，并填充当前已有信息，其余章节标注「待后续阶段自动补全」。

## 版本迭代时的增量处理

当 prd/INDEX.md 已有内容时：

1. 识别变更类型：
   - **新增功能点**：创建新的 feat-*.md
   - **修改已有功能点**：更新对应 feat-*.md，记录变更历史
   - **删除功能点**：标记为 `❌ 已废弃`，不删除文件（保留历史）

2. 分析变更影响范围：
   - 哪些已有功能点受到影响？
   - 是否需要修改已有技术方案？

3. 向用户展示影响范围分析，确认后继续

4. 更新"需求精粹卡"：
   - 新增功能点追加到核心功能清单
   - 修改功能点更新对应条目
   - 新增约束追加到绝对约束列表
   - 保持 ≤500 tokens，必要时精简旧条目

---

## 不确定即停

遇到以下情况必须暂停，严禁自行推测：

- 需求描述模糊，存在多种理解方式
- 业务规则不明确（如"适当的权限控制"）
- 与已有功能存在潜在冲突
- 技术可行性存疑

**暂停格式：**
```
[需人工确认] 关于"{模糊点}"，我理解到以下几种可能：
1. {理解A}：[对应的实现方式]
2. {理解B}：[对应的实现方式]

请问您的意思是哪种？或者请补充说明。
```
