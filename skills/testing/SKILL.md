# CodeMan 测试验证 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code 为 `~/.claude/skills/.codeman`。

---

## 概述

执行完整的测试金字塔，确保所有 PRD 功能点都有对应的测试覆盖。**测试基准来源于 PRD，而非代码。**

```
                    ▲
                   / \       L5 非功能测试（选配）
                  /   \      L4 UI 视觉测试（Midscene.js）
                 /     \     L3 E2E 端到端测试（Playwright MCP）
                /       \    L2 集成测试
               /         \   L1 单元测试（数量最多）
              ─────────────
```

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/prd/INDEX.md
   → 获取所有 PRD 功能点（测试基准来源）

2. 读取相关 PRD 碎片文件 feat-*.md
   → 获取验收标准和 UI 设计要求

3. 读取 .codeman/docs/tests/INDEX.md（如存在）
   → 了解已有测试覆盖，避免重复

4. 读取 .codeman/config.yaml
   → 了解测试框架配置、是否启用 UI 视觉测试

5. 读取 .codeman/docs/DIRECTIVES.md
   → 加载"需求精粹卡"中的关键验收标准
   → 确保每条关键验收标准都有对应测试用例
```

## L1 单元测试

### 目标

验证开发 Skill 已同步生成的单元测试，补充遗漏的测试用例。

### 执行步骤

1. 运行项目单元测试：
   ```bash
   {项目测试命令}
   ```

2. 检查覆盖率报告：
   ```bash
   {项目测试命令} --coverage
   ```

3. 覆盖率不达标时，补充测试用例：
   - 核心业务逻辑：≥ 80%
   - 工具函数/纯函数：≥ 90%

4. 记录测试结果

### 覆盖维度检查

对每个核心函数/方法，确认以下测试用例存在：
- [ ] 正常路径（标准输入 → 预期输出）
- [ ] 边界条件（空值、最大/最小值）
- [ ] 异常输入（非法格式、类型错误）
- [ ] 错误处理（异常情况的错误响应）

## L2 集成测试

### API 接口测试

基于 `.codeman/docs/api/INDEX.md` 中的所有端点，逐一测试：

| 测试维度 | 测试内容 |
|---------|---------|
| 正常请求 | 标准参数 → 预期响应体和状态码 |
| 鉴权验证 | 无 token → 401；无权限 → 403 |
| 参数校验 | 缺少必填参数 → 400 + 错误信息 |
| 边界数据 | 超长字符串、特殊字符、空数组 |
| 幂等性 | 重复请求的行为是否符合预期 |

**测试工具：** 根据 config.yaml 中的技术栈选择（Jest/Supertest、Go testing、pytest 等）

### 数据库交互测试

- CRUD 操作正确性
- 事务回滚验证
- 并发写入的数据一致性
- 边界数据的存储和读取

### 前后端联调测试（如适用）

- 接口契约一致性（前端调用参数与后端定义匹配）
- 数据格式转换正确性

## L3 E2E 端到端测试

### 测试用例生成规则

**核心原则：每条验收标准对应至少 1 个 E2E 测试用例，PRD 边界场景也必须有对应用例。**

生成前，必须逐条读取 PRD 碎片文件（`feat-*.md`）中的：
- `## 功能点明细` 表格中每一行的验收标准
- `## 边界场景` 中列出的每一个边界情况

按以下分类生成测试用例：

| 用例类型 | 来源 | 要求 |
|---------|------|------|
| 正常路径用例 | PRD 功能点明细 → 验收标准 | 每条验收标准至少 1 个 |
| 边界场景用例 | PRD 边界场景 | 每个边界场景至少 1 个 |
| 异常场景用例 | PRD 边界场景中的错误情况 | 每个错误情况至少 1 个 |

**生成前输出用例规划表，等待确认：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
E2E 测试用例规划 — F{ID} {功能点名称}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
正常路径用例（来自验收标准）：
  TC-{ID}-01: {验收标准描述} → 测试场景名称
  TC-{ID}-02: {验收标准描述} → 测试场景名称

边界场景用例（来自 PRD 边界场景）：
  TC-{ID}-03: {边界场景描述} → 测试场景名称
  TC-{ID}-04: {边界场景描述} → 测试场景名称

  TC-{ID}-05: {错误情况描述} → 测试场景名称

确认后生成测试用例文档。
```

**测试用例文档格式（`.codeman/docs/tests/test-{功能名}.md`）：**
```markdown
# 测试用例：{功能点名称}

> PRD 来源：F{ID} | 创建：{日期}

## TC-{ID}-01: {测试场景名称}（正常路径）

**PRD 验收标准对照：** F{ID}.{子ID} — {验收标准原文}

**前置条件：** {测试前需要满足的条件}

**测试步骤：**
1. {操作步骤1}
2. {操作步骤2}
3. {操作步骤3}

**预期结果：** {预期的系统响应}

**断言要点：**
- [ ] {具体可验证的断言1，直接对应验收标准}
- [ ] {具体可验证的断言2}

---

## TC-{ID}-02: {测试场景名称}（边界场景）

**PRD 边界场景对照：** {PRD 边界场景原文}

**前置条件：** {测试前需要满足的条件}

**测试步骤：**
1. {操作步骤}

**预期结果：** {预期的系统响应}

**断言要点：**
- [ ] {具体可验证的断言，直接对应边界场景预期行为}
```

### Playwright MCP 执行

使用 Playwright MCP 自动执行 E2E 测试：

```
通过 Playwright MCP 工具：
1. 导航到目标页面
2. 按测试步骤执行操作
3. 验证预期结果
4. 截图记录（通过/失败状态）
```

**降级策略：** 如果 Playwright MCP 不可用，使用 browser-use 执行，或标记为 `[待人工执行]`。

### 测试执行记录

每个 E2E 测试用例执行后记录：
- 执行状态（通过/失败/跳过）
- 失败截图路径
- 失败原因描述

## L4 UI 视觉测试（Midscene.js）

### 前置条件

确认 config.yaml 中 `ui_visual_testing: true`，且 Midscene.js 已安装：
```bash
npx midscene --version
```

如未安装：
```bash
npm install @midscene/web --save-dev
```

### 设计截图对照验证（如 PRD 中有截图引用）

如 PRD 碎片文件中存在"设计资产引用"表格且有截图路径：

1. 读取设计截图文件（使用 Read 工具的多模态能力）
2. 通过 Playwright 截取当前实现的页面截图
3. 逐项对比：布局结构、元素位置、颜色方案、字体大小、间距
4. 将对比结果纳入 UI 设计对照表

对照表新增"设计截图参考"列：

| 设计稿要求 | 设计截图 | 实际实现 | 是否一致 | 差异描述 |
|-----------|---------|---------|---------|---------|
| {要求} | {截图路径} | {实际值} | ✅/❌ | {差异} |

### 测试执行

基于 PRD 碎片文件中的 UI 设计要求，逐页面、逐组件进行视觉验证：

**Midscene.js 自然语言测试脚本示例：**
```javascript
// midscene-test.js
import { PuppeteerAgent } from '@midscene/web/puppeteer';

const agent = new PuppeteerAgent(page);

// 验证布局
await agent.aiAssert('注册表单居中显示，宽度约为 400px');
await agent.aiAssert('主按钮颜色为蓝色，占满表单宽度');
await agent.aiAssert('输入框有合理的间距，标签清晰可读');

// 验证交互
await agent.aiAction('点击提交按钮');
await agent.aiAssert('显示加载状态');
```

## L5 非功能测试（版本发布前选配）

根据 config.yaml 中的配置决定是否执行：

### 性能测试

```bash
# Lighthouse 性能评分
npx lighthouse {URL} --output=json

# API 性能（k6）
k6 run performance-test.js
```

**阈值：**
- API 响应时间 P95 < 200ms
- 页面 LCP < 2.5s
- Lighthouse 性能评分 ≥ 80

### 安全扫描

```bash
# 依赖漏洞扫描
npm audit

# 或 Snyk
npx snyk test
```

**阈值：** 0 个高危（High）及以上漏洞

### 可访问性检查

```bash
# axe-core
npx axe {URL}
```

**阈值：** 可访问性评分 ≥ 90

## 需求覆盖率追踪矩阵

测试完成后，更新 `.codeman/docs/tests/INDEX.md` 中的覆盖矩阵。

**矩阵粒度要求：追踪到验收标准级别（子功能点），不只是功能点级别。**

```markdown
## 需求覆盖率追踪矩阵

| PRD 来源 | 验收标准 / 边界场景 | 单元测试 | 集成测试 | E2E 用例 | UI 视觉 | 覆盖状态 |
|---------|-----------------|---------|---------|---------|---------|---------|
| F001.1 {子功能} | {验收标准原文} | ✅ {N}个 | ✅ {N}个 | ✅ TC-001-01 | ✅ | ✅ 完整 |
| F001.1 {子功能} | [边界] {边界场景原文} | ✅ {N}个 | ⬜ | ✅ TC-001-03 | ⬜ | ⚠️ 缺集成 |
| F001.2 {子功能} | {验收标准原文} | ✅ {N}个 | ✅ {N}个 | ⬜ | ⬜ | ❌ 缺 E2E |
```

**覆盖规则：**
- 每条验收标准：必须有对应 E2E 用例（TC 编号可追溯）
- 每个边界场景：必须有对应 E2E 或集成测试用例
- 含业务逻辑的功能点：至少单元测试 + E2E
- 纯 UI 展示的功能点：至少 UI 视觉测试
- 涉及 UI 交互的功能点：必须有 UI 视觉测试

**覆盖率计算：**
- PRD 验收标准覆盖率 = 有 E2E 用例的验收标准数 / 总验收标准数
- 边界场景覆盖率 = 有测试用例的边界场景数 / 总边界场景数
- 目标：两项均达到 **100%**

**精粹卡验收标准交叉校验：**
生成覆盖矩阵后，额外对照 DIRECTIVES.md 中"需求精粹卡 → 关键验收标准"列表，
逐条确认每条 [A{N}] 验收标准都有 E2E 测试覆盖。如有遗漏，标记为 Critical 级别缺失。
