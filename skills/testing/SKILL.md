# CodeMan 测试验证 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。

---

## 概述

本 Skill 分两大阶段：**先根据 PRD 生成测试用例，再逐层执行**。确保所有 PRD 功能点都有对应的测试覆盖。**测试基准来源于 PRD，而非代码。**

```
阶段一：测试用例生成            阶段二：逐层执行
┌──────────────────────┐    ┌──────────────────────┐
│ 读取 PRD 验收标准       │    │ L1 单元测试（验证补充） │
│ → 生成 L2 集成测试用例  │    │ L2 集成测试（按用例）   │
│ → 生成 L3 E2E 用例     │ →  │ L3 E2E 测试（按用例）   │
│ → 生成 L4 UI 视觉用例  │    │ L4 UI 视觉（按用例）    │
│ → 生成 L5 非功能用例   │    │ L5 非功能（按用例）     │
│ → 用户确认用例          │    │ → 生成测试报告          │
└──────────────────────┘    └──────────────────────┘
```

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/prd/INDEX.md
   → 获取所有 PRD 功能点（测试基准来源）

2. 读取相关 PRD 碎片文件 feat-*.md
   → 获取验收标准、边界场景、UI 设计要求

3. 读取 .codeman/docs/tests/INDEX.md（如存在）
   → 了解已有测试覆盖，避免重复

4. 读取 .codeman/config.yaml
   → 了解测试框架配置、是否启用 UI 视觉测试

5. 读取 .codeman/docs/DIRECTIVES.md
   → 加载"需求精粹卡"中的关键验收标准
   → 确保每条关键验收标准都有对应测试用例
```

---

## 阶段一：测试用例生成（L2-L5，基于 PRD）

### 核心原则

**每个功能点的测试用例必须从 PRD 验收标准和边界场景中推导，不是从代码反推。**

用例生成前，逐条读取 PRD 碎片文件（`feat-*.md`）中的：
- `## 功能点明细` 表格中每一行的验收标准
- `## 边界场景` 中列出的每一个边界情况
- `## UI 设计要求` 中的视觉规范（如有）
- 需求精粹卡中的关键验收标准（DIRECTIVES.md）

### Step 1: 生成 L2 集成测试用例

基于 API 接口文档（`.codeman/docs/api/INDEX.md`）和 PRD 验收标准，为每个接口/数据交互生成用例：

**用例来源：**

| 用例类型 | 来源 | 要求 |
|---------|------|------|
| 正常请求 | PRD 功能点明细 → 验收标准 | 每个接口至少 1 个正常路径用例 |
| 鉴权验证 | PRD 约束 + 通用规则 | 无 token → 401；无权限 → 403 |
| 参数校验 | PRD 输入约束 | 缺少必填参数 → 400 + 错误信息 |
| 边界数据 | PRD 边界场景 | 超长字符串、特殊字符、空数组 |
| 幂等性 | PRD 业务规则 | 重复请求的行为是否符合预期 |
| 数据库交互 | PRD 数据规则 | CRUD 正确性、事务回滚、并发一致性 |

**输出格式（写入 test-*.md）：**
```markdown
## TC-{ID}-L2-{NN}: {测试场景名称}

**测试层级：** L2 集成测试
**PRD 来源：** F{ID}.{子ID} — {验收标准原文}
**测试类型：** API 接口 / 数据库交互 / 前后端联调

**前置条件：**
- {前置条件}

**测试步骤：**
1. {操作步骤}
2. {操作步骤}

**预期结果：** {预期的系统响应}

**断言要点：**
- [ ] {具体可验证的断言，直接对应验收标准}

**执行状态：** ⬜ 未执行
```

### Step 2: 生成 L3 E2E 端到端测试用例

**核心原则：每条验收标准对应至少 1 个 E2E 测试用例，PRD 边界场景也必须有对应用例。**

按以下分类生成测试用例：

| 用例类型 | 来源 | 要求 |
|---------|------|------|
| 正常路径用例 | PRD 功能点明细 → 验收标准 | 每条验收标准至少 1 个 |
| 边界场景用例 | PRD 边界场景 | 每个边界场景至少 1 个 |
| 异常场景用例 | PRD 边界场景中的错误情况 | 每个错误情况至少 1 个 |

**输出格式（写入 test-*.md）：**
```markdown
## TC-{ID}-L3-{NN}: {测试场景名称}

**测试层级：** L3 E2E 端到端
**PRD 来源：** F{ID}.{子ID} — {验收标准原文}

**前置条件：**
- {前置条件}

**测试步骤：**
1. {操作步骤1}
2. {操作步骤2}
3. {操作步骤3}

**预期结果：** {预期的系统响应}

**断言要点：**
- [ ] {具体可验证的断言1，直接对应验收标准}
- [ ] {具体可验证的断言2}

**执行状态：** ⬜ 未执行
```

### Step 3: 生成 L4 UI 视觉测试用例

基于 PRD 碎片文件中的 UI 设计要求和设计资产引用，为每个页面/组件生成用例：

| 用例类型 | 来源 | 要求 |
|---------|------|------|
| 布局验证 | PRD UI 设计要求 | 每个页面至少 1 个布局用例 |
| 交互验证 | PRD 交互规范 | 每个交互流程至少 1 个用例 |
| 设计稿对照 | 设计截图引用 | 有截图的页面必须对照验证 |
| 响应式验证 | PRD 兼容性要求 | 移动端/平板/桌面至少各 1 个 |

**输出格式（写入 test-*.md）：**
```markdown
## TC-{ID}-L4-{NN}: {测试场景名称}

**测试层级：** L4 UI 视觉测试
**PRD 来源：** F{ID} — {UI 设计要求原文}

**PRD UI 要求：**
- {UI 要求1，如：注册表单居中显示，宽度 400px}
- {UI 要求2}

**设计截图：** {截图路径，如有}

**Midscene.js 断言：**
```javascript
await agent.aiAssert('{自然语言描述的 UI 断言}');
```

**断言要点：**
- [ ] {具体可验证的视觉断言1}
- [ ] {具体可验证的视觉断言2}

**执行状态：** ⬜ 未执行
```

### Step 4: 生成 L5 非功能测试用例

根据 config.yaml 中的 `l5_nonfunctional` 配置决定是否生成：

| 用例类型 | 来源 | 要求 |
|---------|------|------|
| 性能测试 | PRD 性能约束 + 通用阈值 | API P95 < 200ms，LCP < 2.5s |
| 安全扫描 | PRD 安全要求 + 通用基线 | 0 个高危漏洞 |
| 可访问性 | PRD 无障碍要求（如有） | 评分 ≥ 90 |

**输出格式（写入 test-*.md）：**
```markdown
## TC-{ID}-L5-{NN}: {测试场景名称}

**测试层级：** L5 非功能测试
**测试类型：** 性能 / 安全 / 可访问性
**PRD 来源：** {相关 PRD 约束}

**测试命令：**
```bash
{测试命令，如 npx lighthouse {URL} --output=json}
```

**阈值：**
- {具体阈值，如：API 响应时间 P95 < 200ms}

**执行状态：** ⬜ 未执行
```

### Step 5: 输出用例规划总表并确认

生成完所有用例后，输出总表等待用户确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
测试用例规划总表
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
功能点：{N} 个 | PRD 验收标准：{M} 条 | 边界场景：{K} 个

L2 集成测试用例：{X} 个
  TC-F01-L2-01: {接口名} — 正常请求验证
  TC-F01-L2-02: {接口名} — 参数校验
  ...

  TC-F01-L3-01: {功能名} — 正常路径（验收标准 F01.1）
  TC-F01-L3-02: {功能名} — 边界场景

  ...

总计：{N} 个用例，覆盖 {M} 条验收标准（{覆盖率}%）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Step 6: 写入测试用例文档

用户确认后，为每个功能点写入 `.codeman/docs/tests/test-{功能名}.md`，并更新 `.codeman/docs/tests/INDEX.md`。

## 阶段二：逐层执行测试

### L1 单元测试

**目标：** 验证开发 Skill 已同步生成的单元测试，补充遗漏的测试用例。

**执行步骤：**

1. 运行项目单元测试：
   ```bash
   {项目测试命令}
   ```

2. 检查覆盖率报告：
   ```bash
   {项目测试命令} --coverage
   ```

3. 覆盖率不达标时，补充测试用例：
   - 核心业务逻辑：≥ 80%
   - 工具函数/纯函数：≥ 90%

4. 覆盖维度检查（对每个核心函数/方法）：
   - [ ] 正常路径（标准输入 → 预期输出）
   - [ ] 边界条件（空值、最大/最小值）
   - [ ] 异常输入（非法格式、类型错误）
   - [ ] 错误处理（异常情况的错误响应）

5. 记录测试结果

---

### L2 集成测试

**按阶段一生成的用例执行：**

1. 读取 `.codeman/docs/tests/test-*.md` 中所有 `L2` 层级用例
2. 逐一执行，使用 config.yaml 中配置的测试工具（Jest/Supertest、Go testing、pytest 等）
3. 每个用例执行后更新执行状态（✅ 通过 / ❌ 失败）

**执行记录：**
- 执行状态（通过/失败/跳过）
- 失败原因描述

---

### L3 E2E 端到端测试

**按阶段一生成的用例执行：**

1. 读取 `.codeman/docs/tests/test-*.md` 中所有 `L3` 层级用例
2. 使用 Playwright MCP 自动执行：

```
通过 Playwright MCP 工具：
1. 导航到目标页面
2. 按测试步骤执行操作
3. 验证预期结果
4. 截图记录（通过/失败状态）
```

**降级策略：** 如果 Playwright MCP 不可用，使用 browser-use 执行，或标记为 `[待人工执行]`。

**执行记录：**
- 执行状态（通过/失败/跳过）
- 失败截图路径
- 失败原因描述

---

### L4 UI 视觉测试

**前置条件：** 确认 config.yaml 中 `ui_visual_testing: true`，且 Midscene.js 已安装：
```bash
npx midscene --version
```

如未安装：
```bash
npm install @midscene/web --save-dev
```

**按阶段一生成的用例执行：**

1. 读取 `.codeman/docs/tests/test-*.md` 中所有 `L4` 层级用例
2. **设计截图对照**（如有截图引用）：
   - 读取设计截图文件（使用 Read 工具的多模态能力）
   - 通过 Playwright 截取当前实现的页面截图
   - 逐项对比：布局结构、元素位置、颜色方案、字体大小、间距

3. **Midscene.js 自然语言断言：**
```javascript
import { PuppeteerAgent } from '@midscene/web/puppeteer';

const agent = new PuppeteerAgent(page);
await agent.aiAssert('注册表单居中显示，宽度约为 400px');
await agent.aiAssert('主按钮颜色为蓝色，占满表单宽度');
```

4. **响应式测试**（使用 Playwright 多视口）：
   - 移动端：375px（iPhone SE）
   - 平板：768px（iPad）
   - 桌面：1440px

5. 输出 UI 设计对照表

**降级策略：**
- 使用 browser-use 截图并人工对照
- 标记为 `[待人工验证]`，不阻塞主流程

---

### L5 非功能测试

**按阶段一生成的用例执行（仅在 config.yaml 中 `l5_nonfunctional: true` 时执行）：**

1. 读取 `.codeman/docs/tests/test-*.md` 中所有 `L5` 层级用例
2. 逐一执行：

**性能测试：**
```bash
# Lighthouse 性能评分
npx lighthouse {URL} --output=json

# API 性能（k6）
k6 run performance-test.js
```
阈值：API P95 < 200ms，LCP < 2.5s，Lighthouse ≥ 80

**安全扫描：**
```bash
npm audit
# 或 Snyk
npx snyk test
```
阈值：0 个高危（High）及以上漏洞

**可访问性检查：**
```bash
npx axe {URL}
```
阈值：评分 ≥ 90

## 测试执行策略

```
阶段一：用例生成
  读取 PRD → 生成 L2 用例 → 生成 L3 用例 → 生成 L4 用例 → 生成 L5 用例 → 用户确认

阶段二：逐层执行
  L1 单元测试（验证补充）
  → L2 集成测试（按用例执行）
  → L3 E2E 测试（按用例执行）
  → L4 UI 视觉（按用例执行）
  → L5 非功能（按用例执行，选配）
  → 生成测试报告
```

## 需求覆盖率追踪矩阵

测试完成后，更新 `.codeman/docs/tests/INDEX.md` 中的覆盖矩阵。

**矩阵粒度要求：追踪到验收标准级别（子功能点），不只是功能点级别。**

```markdown
## 需求覆盖率追踪矩阵

| PRD 来源 | 验收标准 / 边界场景 | L1 单元 | L2 集成 | L3 E2E | L4 UI | L5 非功能 | 覆盖状态 |
|---------|-----------------|---------|---------|--------|-------|---------|---------|
| F001.1 {子功能} | {验收标准原文} | ✅ {N}个 | ✅ TC-L2-01 | ✅ TC-L3-01 | ✅ TC-L4-01 | ⬜ | ✅ 完整 |
| F001.1 {子功能} | [边界] {边界场景原文} | ✅ {N}个 | ✅ TC-L2-03 | ✅ TC-L3-03 | ⬜ | ⬜ | ⚠️ 缺UI |
| F001.2 {子功能} | {验收标准原文} | ✅ {N}个 | ⬜ | ⬜ | ⬜ | ⬜ | ❌ 未执行 |
```

**覆盖规则：**
- 每条验收标准：必须有对应 L3 E2E 用例（TC 编号可追溯）
- 每个边界场景：必须有对应 L2 或 L3 用例
- 含业务逻辑的功能点：至少 L1 + L3
- 纯 UI 展示的功能点：至少 L4
- 涉及 UI 交互的功能点：必须有 L4

**覆盖率计算：**
- PRD 验收标准覆盖率 = 有 E2E 用例的验收标准数 / 总验收标准数
- 边界场景覆盖率 = 有测试用例的边界场景数 / 总边界场景数
- 目标：两项均达到 **100%**

**精粹卡验收标准交叉校验：**
生成覆盖矩阵后，额外对照 DIRECTIVES.md 中"需求精粹卡 → 关键验收标准"列表，
逐条确认每条 [A{N}] 验收标准都有 E2E 测试覆盖。如有遗漏，标记为 Critical 级别缺失。

## 不确定即停

遇到以下情况必须暂停，严禁自行推测：

- PRD 验收标准描述模糊，不确定测试用例的预期结果
- 测试用例的前置条件不明确（如不确定测试数据应该是什么状态）
- 不确定某个 API 的正确响应格式（文档与实际不一致）
- 测试环境中某个依赖不可用，不确定是否跳过还是等待
- 发现 PRD 与技术方案矛盾（如 PRD 说"A"，实现是"B"）
- 测试失败但不确定是 bug 还是测试用例写错了
- 不确定某个边界场景应该怎么验证

**暂停格式：**
```
[需人工确认] 关于测试"{不确定点}"，我理解到以下几种可能：
1. {理解A}：[对应的测试方式/预期结果]
2. {理解B}：[对应的测试方式/预期结果]

请问您的意思是哪种？
```
