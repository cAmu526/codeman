## 阶段二：逐层执行测试

## 执行日志（强制产出，先于测试报告）

**目的：** 将"实际做了什么"与"报告里说做了什么"机械绑定，防止事后生成报告时凭记忆/想象填空。

**规则：**

1. **测试开始时**创建 `.codeman/docs/tests/execution-log.md`（追加型文件，不覆盖）
2. **每次执行动作**（运行 pytest / curl API / 运行 Playwright / 打开页面 / 查看截图）都必须**立即**在日志末尾追加一行：
   ```
   [YYYY-MM-DDTHH:MM:SS] <层级> <动作> → <结果>
   ```
3. **用例状态更新也同步记录**：
   ```
   [YYYY-MM-DDTHH:MM:SS] TC-F001-L3-01 ✅（基于 l3-01-login.png 截图 + 脚本 exit 0）
   [YYYY-MM-DDTHH:MM:SS] TC-F001-L4-02 ⏳（原因：Midscene 未安装）
   ```
4. **禁止批量事后补写**：日志是执行中记录的，不是事后回忆的。如果一次性"补"了 20 条，等于作废。

**示例日志：**

```markdown
# 执行日志 — v3.2 F6

[2026-04-17T17:45:00] L1 pytest 命令：`pytest server/tests/test_v3_2_dashboard_config_driven.py -v`
[2026-04-17T17:45:12] L1 结果：16 passed in 0.8s
[2026-04-17T17:46:03] L2b 命令：`python tests/test_v3_2_api_integration.py`
[2026-04-17T17:46:40] L2b 结果：16/16 passed
[2026-04-17T18:02:15] L3 TC-F6-L3-01 Playwright 运行 → 通过，截图 l3-01-step5-filter-found.png
[2026-04-17T18:05:33] L3 TC-F6-L3-03 Playwright 失败 → 选择器匹配列表表头，脚本 bug
[2026-04-17T18:07:12] L3 TC-F6-L3-03 修复后重跑 → 通过，截图 l3-03-opportunity-list-no-filters.png
[2026-04-17T18:30:00] L4 TC-F6-L4-01 ⏳（原因：工具未配置 — Midscene.js 未安装）
[2026-04-17T18:30:05] L4 TC-F6-L4-02 ⏳（原因：工具未配置 — Midscene.js 未安装）
[2026-04-17T18:30:10] L4 TC-F6-L4-03 ⏳（原因：工具未配置 — Midscene.js 未安装）
```

**生成测试报告时的强制规则：**

- **报告中任何 ✅ 用例**，必须在 execution-log.md 中能找到对应的"执行 → 通过"记录；否则该用例强制降级为 ⏳
- **报告中"各层通过分布"引用的运行命令**，必须在日志中出现过
- **⏳ 用例的原因**必须与日志中记录的原因一致
- **生成报告前**，先贴一份日志摘要到上下文，再生成报告；严禁跳过日志直接凭印象写报告

**若用户询问为什么某用例是 ⏳：** 直接引用日志中的时间戳和原因，不要现场编造"等价覆盖"之类的解释。

### 预检步骤：代码审查

**目标：** 在自动化测试执行前，先通过代码审查发现逻辑错误、规范问题、边界遗漏。

**⚠️ 全栈强制覆盖：代码审查必须覆盖本次改动涉及的所有文件类型，缺一不可。**

#### 审查范围强制检查

代码审查开始前，先通过 `git diff` 或实际文件列表，识别本次改动涉及的所有文件类型，**每种类型都必须出现在审查结论中**：

| 文件类型 | 识别规则 | 审查重点 |
|---------|---------|---------|
| **前端代码** | `*.tsx` `*.ts` `*.vue` `*.jsx` `src/components/` `src/pages/` | 组件逻辑、状态管理、Props 传递、事件处理、渲染条件 |
| **后端代码** | `*.py` `*.go` `*.ts(server)` `src/api/` `server/` | 业务逻辑、数据校验、错误处理、权限控制 |
| **数据库/SQL** | `*.sql` `migrations/` `prisma/schema.prisma` `*.migration.*` | DDL 安全性（无 `DROP` 误操作）、索引合理性、数据兼容性、回滚方案 |
| **脚本文件** | `*.sh` `*.py(scripts/)` `Makefile` `Dockerfile` | 命令安全性、路径正确性、幂等性、环境变量依赖 |
| **配置文件** | `*.yaml` `*.json` `*.toml` `*.env.*` `.config.*` | 环境差异、敏感信息暴露、默认值合理性 |
| **测试文件** | `tests/` `*.test.*` `*.spec.*` | 测试逻辑正确性、断言充分性、测试数据合理性 |
| **依赖变更** | `package.json` `go.mod` `requirements.txt` `Cargo.toml` | 新增依赖的安全性和必要性、版本锁定 |

**审查执行流程：**

1. **列出改动文件清单**，按上表分类
2. **逐类审查**，每类输出独立结论
3. **输出审查覆盖确认表**（缺任何一类 → 补审后再继续）

**代码审查检查项（每类文件通用 + 专项）：**

通用检查：
- 实现是否符合技术方案
- 是否有逻辑错误、类型错误
- 是否遵循项目代码规范（读取 `.codeman/rules/` 中的规则）
- 是否有遗漏的边界情况

专项检查：
- **SQL/迁移**：不得出现无条件 `DELETE`/`DROP`、必须有回滚方案、线上兼容性（不用 `AFTER`/`FIRST` 调整列顺序）
- **脚本**：不得硬编码路径/密码、必须有错误处理（`set -e`）、幂等可重复执行
- **配置**：不得包含明文密码/密钥、环境变量引用正确、默认值不影响生产
- **依赖变更**：新增依赖是否有已知漏洞（`npm audit` / `pip audit`）、License 是否兼容

**⚠️ 跨功能一致性检查（变更涉及共享定义时必做）：**

如果本次改动涉及了以下任何一种共享定义的变更，必须执行全局一致性检查：

| 变更类型 | 检查方式 | 不一致判定 |
|---------|---------|-----------|
| 枚举/常量变更 | Grep 枚举名，检查所有 switch/case、映射表、下拉选项、筛选器 | 有引用点未包含新值 → ❌ Critical |
| API 契约变更 | 检查所有调用该接口的前端代码，对比参数名/类型 | 调用方参数与新契约不匹配 → ❌ Critical |
| 数据库 Schema 变更 | 检查 ORM Model → API 序列化层 → 前端类型定义的链路 | 任一层未同步 → ❌ Critical |
| 组件/函数签名变更 | Grep 组件/函数名，检查所有使用方的调用参数 | 使用方未传新增必传参数 → ❌ Critical |
| 权限/角色变更 | 检查后端中间件 → 前端条件渲染 → 路由守卫 → 菜单配置 | 任一处未覆盖新角色/权限 → ❌ Major |
| 路由变更 | Grep 旧路径，检查跳转链接、面包屑、导航、重定向 | 有残留旧路径 → ❌ Major |
| 状态机变更 | 检查 UI 状态标签/颜色映射 → 按钮可用性条件 → 通知规则 | 未覆盖新状态 → ❌ Major |
| 错误码变更 | 检查前端错误处理映射表 → 用户提示文案 | 前端映射表缺少新错误码 → ❌ Major |

**检查流程：**
1. 从 `git diff` 识别本次涉及的变更类型
2. 按上表对应的检查方式执行全局搜索
3. 逐个引用点确认是否已同步
4. 未同步的 → 标记为 Critical/Major 问题

> 如果开发阶段已执行过变更影响面扫描并输出了一致性验证报告，此处验证该报告的完整性即可，不需要重复全量扫描。

**⚠️ 硬编码检查（前后端都查）：**

检查代码中是否存在硬编码的业务参数，这些应该从配置、API 或上下文获取：

| 检查项 | 危险模式示例 | 正确做法 |
|--------|------------|---------|
| 业务线/租户 ID | `businessLine = 2`、`tenantId = 'xxx'` | 从用户 Session / Context 获取 |
| 角色/权限 ID | `roleId = 1`、`permission = 'admin'` | 从认证接口 / 配置获取 |
| API 路径 | `fetch('/api/v1/...')` 中的版本号硬编码 | 从环境变量 / 配置文件获取 |
| 状态枚举值 | `status = 3`（魔术数字） | 使用命名常量或枚举类型 |
| 分页默认值 | `pageSize = 10` 散落在多处 | 统一从常量/配置获取 |
| 环境地址 | `http://localhost:3000` 或 `http://prod.xxx.com` | 从环境变量获取 |

发现硬编码 → 记录为 Major 问题，建议重构为配置化。

**⚠️ 前后端参数一致性检查：**

对照技术方案/API 文档，检查前端传入 API 的参数是否与后端期望一致：

| 检查项 | 检查方式 | 不一致示例 |
|--------|---------|-----------|
| 参数名 | 前端 `fetch` / `axios` 调用的参数名 vs API 文档定义 | 前端传 `userName`，后端期望 `username` |
| 参数类型 | 前端传的值类型 vs 后端期望类型 | 前端传 `"123"`（字符串），后端期望 `123`（数字） |
| 必传/选传 | 前端是否遗漏了后端的必传参数 | 后端要求 `businessLineId`，前端没传 |
| 枚举值 | 前端传的枚举值 vs 后端定义的枚举范围 | 前端传 `status: "active"`，后端枚举只有 `"ACTIVE"` |
| 嵌套结构 | 前端传的 JSON 结构 vs 后端期望的层级 | 前端传 `{ contact: { name } }`，后端期望 `{ contactName }` |

发现不一致 → 记录为 Critical 问题（直接导致接口报错）。

**⚠️ 代码审查的严格限制：**

代码审查**只能**做以下事情：
1. 发现问题并记录到测试报告「发现的问题」表格
2. 确认代码逻辑层面的正确性（如：算法正确、SQL 合理）

代码审查**严禁**做以下事情：
1. **严禁将 L2/L3/L4/L5 任何用例标记为 ✅ 通过** — 无论代码看起来多正确
2. **严禁用"代码已实现 xxx"作为任何功能性用例的通过依据**
3. **严禁跳过某类文件的审查** — 改动涉及的所有文件类型都必须出现在审查结论中

**代码审查完成后输出：**
```
【代码审查预检完成】

审查覆盖确认：
  ✅ 前端代码：审查了 {N} 个文件（{文件列表}）
  ✅ 后端代码：审查了 {N} 个文件（{文件列表}）
  ✅ SQL/迁移：审查了 {N} 个文件（{文件列表}）
  ✅ 脚本文件：审查了 {N} 个文件（{文件列表}）
  ⬜ 配置文件：本次无改动
  ⬜ 依赖变更：本次无改动

硬编码检查：
  ✅ 无硬编码业务参数 / ❌ 发现 {N} 处硬编码（{详情}）

前后端参数一致性检查：
  ✅ {N} 个接口调用参数与 API 文档一致 / ❌ 发现 {N} 处不一致（{详情}）

发现问题：{N} 个
  - [Critical] {问题描述}（{文件路径}:{行号}）
  - [Major] {问题描述}（{文件路径}:{行号}）

严重问题（阻塞测试执行）：{N} 个
L2-L5 用例状态：全部保持 ⏳（未执行任何自动化测试，不允许标记通过）
→ 进入 L1 单元测试
```

> 若有 Critical 问题（如 SQL 无条件删除数据、代码核心逻辑错误），应暂停执行，等修复后再继续。
> 若审查覆盖确认表中有应审查但未审查的文件类型 → 必须补审后再继续，不得跳过。

### L2a 数据库直查测试

**目标：** 绕过 API 层，直连数据库验证表结构、迁移、初始数据、约束等。无需 API 认证，可在服务未启动时运行。

**适用场景（有后端数据库时）：**
- 迁移脚本是否正确创建了表结构、字段、索引
- 初始化数据是否正确插入（枚举值、配置项等）
- 外键约束、唯一约束是否生效
- 字段类型、默认值是否与技术文档一致

**执行步骤：**

1. 确认数据库连接可用
2. 运行 DB 直查脚本：
   ```bash
   {数据库测试命令，如 python tests/db/test_db_direct.py / node tests/db/db-check.js}
   ```
3. 每个用例直接查询数据库验证，无需 API 认证
4. 记录通过/失败状态

**DB 脚本设计原则：**
- **只读查询**：DB 直查脚本只做 SELECT，不修改数据
- **无需认证**：直连数据库，不依赖 API Token
- **独立运行**：服务未启动也能执行

> 若项目无后端数据库（纯前端项目），跳过此步骤。

   - 执行状态（通过/失败/跳过）

   - **验证证据**（API 响应体摘要 / 脚本输出行号）

### L3 E2E 端到端测试

**全部 Playwright 浏览器自动化执行。这是有 UI 功能的主要测试方式。**

1. 启动测试服务器（如适用）：
   ```bash
   npm run dev:test &
   ```

2. 运行基础数据初始化（如有需要）：
   ```bash
   bash tests/setup.sh
   ```

3. **脚本断言 lint 扫描（机械化门控，不依赖 AI 判断）：**

   **用 Grep 扫描所有 E2E 脚本，检测禁止断言模式。扫描不通过的脚本必须修改后才能执行。**

   ```bash
   # 扫描禁止模式（根据项目语言调整正则）

   # 1. HTML 全文关键词匹配（最常见的反模式）
   grep -rn "page\.content()" tests/e2e/ tests/integration/
   grep -rn "\.innerHTML" tests/e2e/ tests/integration/
   # 匹配到 → 检查是否用于断言。用 page.content() 做断言 = 禁止模式

   # 2. 恒为真断言
   grep -rn "\.count().*>= 0" tests/e2e/ tests/integration/
   grep -rn "expect(true)" tests/e2e/ tests/integration/
   grep -rn "toBeTruthy()" tests/e2e/ tests/integration/
   # 匹配到 → 禁止模式，必须修改

   # 3. 只截图不断言
   grep -rn "screenshot" tests/e2e/ | grep -v "expect"
   # 匹配到且同一个 test block 中没有 expect → 禁止模式

   # 4. 操作后无状态验证（click/fill 后没有 expect）
   # 检查每个 test block 中是否有 click/fill 但没有对应的 expect
   ```

   **扫描结果处理：**

   ```
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   脚本断言 lint 扫描
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   扫描文件：{N} 个
   禁止模式命中：{N} 处
     ❌ tests/e2e/register.spec.ts:42 — page.content().includes('注册')（HTML 全文匹配）
     ❌ tests/e2e/list.spec.ts:78 — .count() >= 0（恒为真）
     ✅ tests/e2e/login.spec.ts — 无禁止模式

   命中 > 0 → 必须修改后重新扫描，全部通过后才能继续
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   ```

   **此步骤是机械化门控 — 不依赖 AI 判断脚本质量好不好，只检查是否包含已知的错误模式。**

4. **脚本冒烟验证（正式执行前必做）：** 逐个验证关键选择器有效性、页面加载正常、前置操作可达（详见「脚本冒烟验证」章节）。冒烟不通过的脚本必须修复后才能进入正式执行。

5. 执行所有 E2E 测试脚本（浏览器自动打开、操作、验证）：
   ```bash
   npx playwright test tests/e2e/
   ```

6. E2E 脚本会自动完成：
   - 打开浏览器，导航到目标页面
   - **通过 UI 操作创建测试数据**（填写表单、点击按钮）
   - **通过 UI 操作完成用户流程**（注册→登录→操作→验证）
   - **在浏览器中验证页面展示**（文字内容、元素可见性、URL 跳转）
   - 自动截图记录通过/失败状态

7. 执行记录：
   - 执行状态（通过/失败/跳过）
   - 失败截图路径
   - 失败原因（自动从 Playwright 输出中提取）

### 层间门禁自检（强制执行）

**每层执行完毕后，必须输出以下格式的自检报告，确认当层状态正确后才能进入下一层。** 此机制防止跨层级标记通过。

**门禁自检输出格式：**

```
【{层级名称} 完成 — 门禁自检】
- 本层用例：共 {N} 个
  - ✅ 通过：{N} 个（均有实际执行证据）
  - ❌ 失败：{N} 个
  - ⏳ 待验证：{N} 个
- 验证方式：{实际运行的命令/工具}
- 后续层级用例状态确认：L{X}-L{Y} 共 {N} 个用例，全部保持 ⏳（未执行，不允许标记通过）
→ 进入 {下一层级名称}
```

**门禁检查规则：**
1. **本层所有用例必须有明确状态**（✅ / ❌ / ⏳），不得遗漏
2. **本层 ✅ 用例必须有实际执行证据**，若发现无证据的 ✅ → 强制降级为 ⏳
3. **后续层级用例必须全部为 ⏳**，若发现后续层级用例已被标记 ✅ → 强制降级并警告
4. **本层有 Critical 级别失败**时 → 暂停，询问用户是否继续或先修复

## 回归测试策略

### 三级回归

| 级别 | 名称 | 触发时机 | 测试范围 | 预期耗时 |
|------|------|---------|---------|---------|
| **Smoke** | 冒烟测试 | 每次代码修改后 | L1 全跑 + L2b 抽样 + L3 主流程 | 3-5 分钟 |
| **Regression** | 标准回归 | PR 合并 / 每日定时 / 手动触发 | L1 + L2a + L2b + L3 + L4 | 10-20 分钟 |
| **Full** | 全量测试 | 发版前 / 手动触发 | L1-L5 全部 + 网络异常 + 兼容性 | 30-60 分钟 |

### Smoke 冒烟测试：分层抽样策略

**目标：3-5 分钟内回答"核心功能是不是挂了"。**

冒烟测试不是只跑 P0 用例，而是每层抽样覆盖，确保"接口能通 + 数据格式对 + 主流程走得通"：

| 层级 | 冒烟范围 | 抽样规则 |
|------|---------|---------|
| L1 单元测试 | **全跑** | 单元测试本身很快（秒级），全跑没有额外成本 |
| L2b API 集成 | **每接口 2 个用例** | 1 个正向主流程 + 1 个响应格式校验（覆盖可用性 + 数据正确性） |
| L3 E2E | **每功能 1 个主流程** | P0 功能点的正向完整流程用例（用户最核心的操作路径） |
| L2a / L4 / L5 / 网络异常 / 兼容性 | **默认跳过** | 按影响分析按需追加（见下文） |

**冒烟用例标记：** 阶段一生成用例时，自动标记每个功能的冒烟核心用例。标记规则：
- P0 功能点的正向主流程 E2E 用例 → 标记 `[smoke]`
- 每个 API 接口的正向用例 + 格式校验用例 → 标记 `[smoke]`
- 每个功能最多标记 2-3 个 `[smoke]` 用例

**执行命令：**
```bash
# 冒烟测试：只跑带 [smoke] 标记的用例
{单元测试命令}                                    # L1 全跑
npx playwright test tests/integration/ --grep smoke  # L2b 抽样
npx playwright test tests/e2e/ --grep smoke          # L3 主流程
```

### Regression 标准回归

**目标：10-20 分钟内全面验证功能正确性。**

覆盖 L1 + L2a + L2b + L3 + L4，跳过 L5/网络异常/兼容性（这些耗时长且不是每次都需要）。

### Full 全量测试

**目标：发版前完整验证，不留盲区。**

覆盖所有层级 + 网络异常批次 + 兼容性测试（需确认浏览器范围）。

### 影响分析驱动的按需追加

**默认回归范围是兜底，影响分析可以智能追加或裁剪。** 两层结合：

**第一层：路径规则自动匹配（确定性高，作为兜底）**

| 改动文件路径 | 自动追加的测试 |
|-------------|--------------|
| `src/api/` 或 `server/` 或后端路由 | 对应的 L2b 用例 + 依赖该 API 的 L3 用例 |
| `src/components/` 或前端组件 | 涉及该组件的 L3 + L4 用例 |
| `migrations/` 或数据库 Schema | L2a 全跑 + 相关 L2b 用例 |
| `src/styles/` 或全局样式 | L4 全跑 |
| `package.json` 或依赖变更 | L5 安全扫描（npm audit） |
| `tests/` 或测试脚本自身 | 修改的测试重跑 |

**第二层：AI 补充判断（修复闭环中使用）**

AI 在修复 bug 后读取 `git diff`，分析：
- 改动是否影响了公共模块（工具函数、通用组件、中间件）→ 如果是，扩大回归范围到所有依赖模块
- 改动是否涉及权限/认证逻辑 → 如果是，追加权限相关 L3 用例
- 改动是否涉及数据模型 → 如果是，追加 L2a + 关联接口的 L2b
- 改动是否涉及网络请求处理/错误处理 → 如果是，追加网络异常批次

**第三层：跨功能依赖追踪（防止"改一处漏多处"）**

当改动涉及共享定义（枚举/API 契约/组件签名/数据模型/权限/路由/状态机/错误码）时，自动触发：

1. **Grep 变更的标识符**（枚举名、接口路径、组件名等），找出所有引用方
2. **对每个引用方对应的功能点**，追加其 L2b + L3 用例到回归范围
3. **额外生成跨功能一致性验证用例**：验证引用方使用的值是否与定义方一致

示例：
```
改动：customer_type 枚举新增 "VIP"
Grep 引用方：CustomerList.tsx / Reports.tsx / export.ts / customer.py
追加回归：
  + L2b: TC-F002-L2b-01（客户列表接口 — 筛选 VIP 类型）
  + L3: TC-F002-L3-01（客户列表页 — 筛选器包含 VIP）
  + L3: TC-F005-L3-01（报表页 — VIP 分组正确）
```

### CI/CD 自动化集成（可选）

**CI/CD 是可选增强，默认不开启。** 在项目初始化时询问用户是否启用。

启用后，CodeMan 在阶段一生成测试脚本时，同时生成 CI 配置文件：

**CI 平台选择（config.yaml 配置）：**
- `github_actions`：生成 `.github/workflows/test.yml`
- `gitlab_ci`：生成 `.gitlab-ci.yml`
- `none`（默认）：不生成 CI 配置

**CI Pipeline 与回归级别的对应关系：**

```
代码推送（push）     → 触发 Smoke 冒烟测试
PR 创建/更新         → 触发 Regression 标准回归
合并到主分支（merge） → 触发 Regression 标准回归
手动触发 / Tag 发版   → 触发 Full 全量测试
```

**测试结果反馈机制：**
- Smoke/Regression 失败 → PR 上显示 ❌，**阻断合并**
- 测试报告摘要自动回写到 PR Comment（通过/失败数、失败详情链接）
- Full 测试结果记录到 Release Notes

## 通过用例抽样交叉验证

**所有测试层级执行完毕后、生成最终报告前，必须对 ✅ 通过的 L3 E2E 用例进行抽样交叉验证。**

**为什么：** 脚本"通过"不等于功能正确。如果断言逻辑有误（如断言了恒为真的条件），脚本会通过但实际功能可能是坏的。lint 扫描能拦住已知的错误模式，但无法拦住所有情况。交叉验证用独立视角重新检查通过的用例。

### 执行步骤

```
1. 从所有 ✅ 通过的 L3 E2E 用例中随机抽取 3-5 个（优先抽取涉及以下场景的用例）：
   - 下拉框/选择器数据加载
   - 列表数据展示
   - 表单提交完整流程
   - 布局结构相关

2. 对每个抽取的用例，独立执行验证：
   a. 用 Playwright 打开对应页面，截图
   b. 手动（AI 读取截图）对照 PRD 验收标准逐项检查：
      - 页面是否真的显示了预期内容？（不是只看脚本 log）
      - 布局是否符合 PRD ASCII 原型？
      - 交互操作后的状态变化是否正确？
   c. 将独立验证结果与脚本断言结果对比

3. 输出交叉验证报告：
```

**输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通过用例交叉验证（抽样 {N} 个）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
| 用例 | 脚本结果 | 独立验证 | 一致性 |
|------|---------|---------|--------|
| TC-F001-L3-01 注册流程 | ✅ | ✅ 截图确认注册成功跳转 | ✅ 一致 |
| TC-F002-L3-03 列表筛选 | ✅ | ❌ 截图显示筛选后数据未变化 | ❌ 不一致 |
| TC-F003-L3-01 弹窗提交 | ✅ | ✅ 截图确认提交后列表更新 | ✅ 一致 |

不一致用例：{N} 个
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**处理规则：**
- **全部一致** → 交叉验证通过，继续生成报告
- **有不一致** → 该用例降级为 ⏳，同时：
  1. 检查脚本的断言逻辑是否有问题（可能是断言了恒为真的条件）
  2. 修复脚本断言后重跑
  3. **对同一脚本文件中的其他用例也做排查**（反模式可能在文件内传染）
  4. 排查完毕后重新抽样验证

## 项目级稳定性验证（三层验证）

**时机：** L1-L5 各层级测试全部执行完毕后、通过用例抽样交叉验证完毕后。这是 Git squash merge 前的最后一道门禁。

**为什么需要：** 单个功能点的测试通过只证明该功能正确。当新需求涉及架构变更或修改了共享模块时，可能引发跨模块的连锁问题，只有项目级验证才能发现。

### V1: 构建验证

**目标：** 确认项目能正常编译/构建，无类型错误、引用断裂或语法问题。

**执行：** 根据 `config.yaml` 中的技术栈自动选择构建命令：

```bash
# TypeScript 项目
tsc --noEmit

# Go 项目
go build ./...

# Python 项目
python -m py_compile {主入口文件}
# 或 mypy（如有配置）

# Node.js（有 build script）
npm run build

# Rust
cargo check

# Shell 脚本项目
bash -n {主脚本}  # 语法检查
shellcheck {主脚本}  # 如有 shellcheck
```

**通过标准：** 退出码 0，零错误零警告（或只有已知的非阻塞警告）。

**失败处理：** 输出错误信息 → 进入修复闭环 → 修复后重新从 V1 开始。

### V2: 已有测试套件全量执行

**目标：** 确认本次改动没有破坏已有功能。

**执行：** 运行项目中**所有**已有的测试，不只是本次新写的：

```bash
# 执行全量测试
{项目测试命令}  # npm test / go test ./... / pytest / cargo test
```

**包含的测试：**
- 之前迭代写的单元测试和集成测试
- 本次新写的测试
- 代码回归用例（`[regression-from-code]` 标记的用例）

**通过标准：** 全部测试通过。

**失败处理：** 
- 之前的测试失败 → 本次改动破坏了老功能，进入修复闭环
- 本次新测试失败 → 应该在 L1-L5 阶段已处理，如遗漏则修复
- 回归用例失败 → 老功能被破坏，定位变更引入点并修复

### V3: 项目级 Smoke 测试

**目标：** 确认项目能正常启动运行，核心路径可达。

**执行：** 根据项目类型自动选择验证方式：

**Web 后端项目：**
```bash
# 1. 启动服务器（后台）
{启动命令} &
SERVER_PID=$!
sleep 3  # 等待启动

# 2. 请求 3-5 个核心接口
curl -s -o /dev/null -w "%{http_code}" http://localhost:{端口}/health
curl -s -o /dev/null -w "%{http_code}" http://localhost:{端口}/api/{核心接口1}
curl -s -o /dev/null -w "%{http_code}" http://localhost:{端口}/api/{核心接口2}

# 3. 关闭服务器
kill $SERVER_PID
```

**Web 前端 / 全栈项目：**
```typescript
// 用 Playwright 验证
const browser = await chromium.launch({ headless: true });
const page = await browser.newPage();
await page.goto('http://localhost:{端口}');
// 确认：不是白屏、无控制台错误、核心元素可见
const errors = [];
page.on('pageerror', err => errors.push(err.message));
await page.waitForLoadState('networkidle');
await page.screenshot({ path: 'tests/screenshots/v3-smoke.png' });
expect(errors).toHaveLength(0);
await browser.close();
```

**CLI 工具项目：**
```bash
# 1. help 命令正常输出
{主命令} --help
# 退出码 0 + 输出包含 usage 信息

# 2. 典型参数组合
{主命令} {典型参数1}  # 不报错
{主命令} {典型参数2}  # 不报错
```

**库/SDK 项目：**
```bash
# 验证主入口可导入，核心 API 可调用
node -e "const lib = require('{主入口}'); console.log(Object.keys(lib));"
# 或 Python
python -c "import {包名}; print(dir({包名}))"
```

**Shell 脚本项目：**
```bash
# 主脚本 help 正常
bash {主脚本} --help
# 关键子脚本语法检查
bash -n {子脚本1}
bash -n {子脚本2}
```

**通过标准：** 项目启动无崩溃，核心路径返回预期响应，无运行时错误。

**失败处理：** 记录错误信息和截图 → 进入修复闭环。

## 需求覆盖率追踪矩阵

测试完成后，更新 `.codeman/docs/tests/INDEX.md` 中的覆盖矩阵。

**矩阵粒度要求：追踪到验收标准级别（子功能点），不只是功能点级别。**

```markdown
## 需求覆盖率追踪矩阵

| PRD 来源 | 验收标准 / 边界场景 | L1 单元 | L2a DB | L2b API | L3 E2E | L4 UI | L5 非功能 | 覆盖状态 |
|---------|-----------------|---------|--------|---------|--------|-------|---------|---------|
| F001.1 {子功能} | {验收标准原文} | ✅ {N}个 | ✅ TC-L2a-01 | ✅ TC-L2b-01 | ✅ TC-L3-01 | ✅ TC-L4-01 | ⬜ | ✅ 完整 |
| F001.1 {子功能} | [边界] {边界场景原文} | ✅ {N}个 | ⬜ | ✅ TC-L2b-03 | ✅ TC-L3-03 | ⬜ | ⬜ | ⚠️ 缺UI |
| F001.2 {子功能} | {验收标准原文} | ✅ {N}个 | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | ❌ 未执行 |
```

**覆盖规则：**
- 每条验收标准：必须有对应 L3 E2E 用例（TC 编号可追溯）
- 每个边界场景：必须有对应 L2 或 L3 用例
- 含业务逻辑的功能点：至少 L1 + L3
- 纯 UI 展示的功能点：至少 L4
- 涉及 UI 交互的功能点：必须有 L4

**覆盖率计算：**
- PRD 验收标准覆盖率 = 有 E2E 用例的验收标准数 / 总验收标准数
- 边界场景覆盖率 = 有测试用例的边界场景数 / 总边界场景数
- 目标：两项均达到 **100%**

**精粹卡验收标准交叉校验：**
生成覆盖矩阵后，额外对照 DIRECTIVES.md 中"需求精粹卡 → 关键验收标准"列表，
逐条确认每条 [A{N}] 验收标准都有 E2E 测试覆盖。如有遗漏，标记为 Critical 级别缺失。
