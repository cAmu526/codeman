# CodeMan 技术方案 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。

---

## 概述

基于已确认的 PRD 碎片文件，生成模块化的技术设计方案。核心流程含**三次交互确认**，确保方案合理、API 契约清晰、细节可执行：

```
分析 PRD → 模块拆分 → 【确认①：模块划分】
  → 方案设计 + API 设计 → 【确认②：API 契约审阅】
  → Review 门禁 → 【确认③：方案细节审阅】
  → 文档化
```

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束（技术栈、架构约束等）
   → 特别关注"需求精粹卡"章节，确保设计方案覆盖所有核心功能和绝对约束

2. 读取 .codeman/config.yaml
   → 了解技术栈配置

3. 读取 .codeman/docs/prd/INDEX.md
   → 获取所有已确认的功能点列表

4. 按需加载 PRD 碎片文件
   → 读取当前任务相关的 feat-*.md

5. 读取 .codeman/docs/design/INDEX.md（如存在）
   → 了解已有架构，避免重复设计

6. 读取 .codeman/rules/INDEX.md
   → 加载项目规范约束

7. 执行周期性上下文回顾
   → 重新读取 DIRECTIVES.md，输出回顾摘要
```

---

## 执行步骤

### Step 1: 分析 PRD，识别模块边界

遍历所有 PRD 功能点，按以下维度进行模块归类：

- **业务领域**（用户、订单、支付、通知等）
- **技术层次**（前端组件、后端服务、数据层、基础设施）
- **依赖关系**（哪些模块依赖哪些模块）
- **PRD 功能点映射**（每个模块覆盖哪些 PRD 功能点）

输出模块依赖图（ASCII 格式）：
```
[模块A] ──依赖──→ [模块B]
                    │
                    └──依赖──→ [模块C]
```

### Step 1.5: 【确认① — 模块划分确认】

模块拆分后，**立即**输出给用户确认，不进入详细设计：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
技术方案 — 模块划分确认（第 1 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

模块划分（共 {N} 个模块）：

  {模块A}：{一句话职责}
    覆盖 PRD：F001（用户注册）、F002（用户登录）
    关键设计方向：{一句话，如"JWT 无状态鉴权"}

  {模块B}：{一句话职责}
    覆盖 PRD：F003（商品列表）、F004（商品详情）
    关键设计方向：{一句话，如"Redis 缓存 + 分页查询"}

  {模块C}：{一句话职责}
    覆盖 PRD：F005（下单）、F006（支付）
    关键设计方向：{一句话，如"事务一致性 + 幂等性"}

模块依赖关系：
  {模块A} → {模块B} → {模块C}

架构风格：{如：前后端分离，RESTful API，PostgreSQL + Redis}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受划分
  - 或提出调整："模块 A 和 B 应该合并"、"缺少 xxx 模块"
  - 或说"模块 C 的职责应该改为..."
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户反馈处理：**
- 如用户调整模块划分，重新输出依赖图和确认表
- 如用户确认，进入详细设计

### Step 2: 为每个模块生成技术方案

用户确认模块划分后，**逐模块**生成技术方案。

**每个模块的技术方案使用模板：** `{CODEMAN_HOME}/templates/design/mod-template.md`

**方案内容：**

```markdown
## 模块：{模块名称}

### 职责
{一句话描述模块职责}

### 覆盖 PRD
{列出本模块覆盖的 PRD 功能点}

### 技术选型
| 技术点 | 选型 | 理由 |
|--------|------|------|
| {技术点} | {选型} | {简要理由} |

### 数据模型
{关键数据结构定义（TypeScript/Go/Python 类型），含字段注释}

### 核心逻辑
{关键算法或业务逻辑的伪代码描述，含错误处理}

### 模块接口（对外暴露）
{函数签名或 API 端点列表}

### 依赖
{依赖的其他模块或外部服务}

### 架构决策记录
#### DR-{ID}: {决策名称}
- 日期：{YYYY-MM-DD}
- 备选：A. {方案A} B. {方案B}
- 决策：{选择的方案}
- 理由：{选择原因}
```

### Step 3: 生成 API 设计

为每个涉及 API 的模块生成 API 碎片文件：

**文件路径：** `.codeman/docs/api/api-{模块名}.md`

**使用模板：** `{CODEMAN_HOME}/templates/api/api-template.md`

**API 设计质量要求：**
- 每个端点必须明确：请求方法、路径、参数、响应体、错误码
- 必须标注鉴权方式和对应 PRD 功能点
- 请求/响应使用 JSON 示例，字段名和类型必须精确
- 错误码覆盖常见的 400/401/403/404/500 + 业务错误码

### Step 3.5: 【确认② — API 契约审阅】

API 设计生成后，**独立输出**给用户逐接口确认：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
技术方案 — API 契约审阅（第 2 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

接口总数：{N} 个

📄 {模块A} API（{X} 个接口）

  POST /api/auth/register
    描述：用户注册
    鉴权：无需
    请求：{ name: string, email: string, password: string }
    响应：{ user_id: string, token: string }
    错误码：400(参数校验失败) / 409(邮箱已注册)
    对应 PRD：F001.1, F001.2

  POST /api/auth/login
    描述：用户登录
    鉴权：无需
    请求：{ email: string, password: string }
    响应：{ user_id: string, token: string, expires_at: string }
    错误码：400(参数校验失败) / 401(密码错误)
    对应 PRD：F002.1

📄 {模块B} API（{Y} 个接口）
  ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受全部
  - 或指出具体问题："/api/auth/register 的响应应该包含 refresh_token"
  - 或补充遗漏的接口
  - 或说"重新设计 {接口路径}"要求重新设计某个接口
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户反馈处理：**
- 如用户指出问题，修改对应 API 文件后重新展示
- 如用户补充接口，补充后重新展示
- 如用户确认，进入下一步

### Step 4: 【自动 Review 门禁】

技术方案和 API 设计确认后，**立即调用 Review Skill** 进行方案审查：

审查重点：
- **PRD 一致性**：每个 PRD 功能点是否都有对应的技术实现
- **技术可行性**：选型是否合理、是否有技术风险
- **架构合理性**：模块边界是否清晰、依赖关系是否合理
- **边界场景覆盖**：PRD 中的边界场景是否在技术方案中有处理
- **安全性**：是否有明显的安全漏洞
- **API 契约完整性**：请求/响应/错误码是否完整、前后端契约是否一致

Review 发现问题 → 自行修复 → 再次 Review → 通过后继续

### Step 4.5: 【确认③ — 方案细节审阅】

Review 通过后，输出**每个模块的核心设计细节**供用户审阅：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
技术方案 — 细节审阅（第 3 次确认，共 3 次）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📄 {模块A}：{模块职责}

  技术选型：
    {技术点}：{选型}（{理由}）

  数据模型（核心字段）：
    {ModelName}:
      {field1}: {type} — {说明}
      {field2}: {type} — {说明}
      ...

  核心逻辑摘要：
    {关键流程的伪代码或文字描述}

  关键决策：
    DR-001: {决策名称}
      方案：{选择的方案}
      理由：{选择原因}
      影响：{对其他模块的影响}

  ⚠️ 风险提示（如有）：
    - {风险描述}，缓解措施：{措施}

📄 {模块B}：{模块职责}
  ...

Review 结果：✅ 通过（{N} 个 Medium 问题已修复）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请确认：
  - 直接回复"确认"接受全部
  - 或指出具体问题："模块 A 的数据模型缺少 xxx 字段"
  - 或质疑决策："DR-001 应该选择方案 B，原因是..."
  - 或说"重新设计 {模块名}"要求重新设计某个模块
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户反馈处理：**
- 如用户修改，更新对应 mod-*.md 后重新展示
- 如用户质疑决策，列出更多备选方案供决策
- 如用户确认，方案锁定，进入文档化

### Step 5: 生成技术方案碎片文件

用户最终确认后，写入 `.codeman/docs/design/mod-{模块名}.md`（使用 Step 2 的内容）。

### Step 6: 更新索引文件

更新 `.codeman/docs/design/INDEX.md`：
```markdown
# 技术方案索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}

| 模块 | 文件 | 状态 | PRD 覆盖 | 最后更新 |
|------|------|------|---------|---------|
| {模块名} | mod-{name}.md | ✅ 已确认 | F001,F002 | {日期} |
```

更新 `.codeman/docs/api/INDEX.md`：
```markdown
# API 接口索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}
> 接口总数：{N}

| 接口 | 方法 | 路径 | 模块 | 状态 |
|------|------|------|------|------|
| {接口名} | GET | /api/... | {模块} | ✅ 已设计 |
```

### Step 7: 更新 STATUS.md

```markdown
- phase: development
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（设计列标记 ✅）。

### Step 8: 更新项目概览文档

读取 `.codeman/docs/PROJECT-OVERVIEW.md`，更新以下章节（**必须内联核心摘要，不能只放链接**）：

**「技术架构」章节：**
- 从 config.yaml 和 arch-overview.md 汇总技术栈全景（表格含选型理由）
- 用 2-3 段自然语言描述整体架构风格、数据流向、关键技术决策

**「模块结构」章节：**
- 每个模块附**一句话关键设计说明**（如"采用 JWT 无状态鉴权"、"使用事件驱动解耦"）
- 从 arch-overview.md 提取模块依赖关系图
- 末尾附 `→ 详见 design/mod-{{name}}.md` 链接

**「API 概览」章节：**
- 每个接口附**核心请求/响应字段摘要**（如"接收 name, email, password；返回 user_id, token"）
- 纯前端项目标注「不适用」
- 末尾附 `→ 详见 api/INDEX.md` 链接

**「关键决策记录」章节：**
- 每个决策用 1-2 句话描述**决策内容和原因**（如"选择 PostgreSQL 而非 MongoDB：需要强关联查询和事务支持"）
- 从各 mod-*.md 和 DIRECTIVES.md 提取

**「开发规范速查」章节：**
- 每条规范写**具体可操作的要点**（如"函数 ≤ 50 行，复杂度 ≤ 10"），不是空占位符
- 从 .codeman/rules/ 下的 .mdc 文件提取

**「当前进度」章节：**
- 从 STATUS.md 同步最新进度

> 如果 PROJECT-OVERVIEW.md 不存在，基于模板创建并填充当前已有信息。

## 版本迭代时的增量处理

当 design/INDEX.md 已有内容时：

1. 识别受影响的模块（基于 PRD 变更分析）
2. 只更新受影响的 mod-*.md 文件
3. 在文件中记录变更历史
4. 检查 API 变更的向后兼容性
5. 向用户展示变更影响范围

---

## 不确定即停（遵守 orchestrator 全局硬规则）

遇到以下情况必须暂停，**严禁自行假设后继续**：

- **技术选型存在多个可行方案**：列出各方案优劣，暂停由用户决策
- **架构方向不明确**：如单体 vs 微服务、REST vs GraphQL
- **PRD 中有技术约束未明确**：如性能要求、并发量
- **依赖的外部服务不确定**：如第三方 API、云服务选型
- **数据模型设计有多种方案**：不确定哪种更合理
- **API 设计有歧义**：不确定请求/响应格式
- **安全方案不确定**：不确定应该采用哪种鉴权/加密方式

**暂停格式：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
在"{技术点}"的选型上，有以下几个可行方案：

方案 A：{方案名}
  优点：{优点}
  缺点：{缺点}
  适用场景：{场景}

方案 B：{方案名}
  优点：{优点}
  缺点：{缺点}
  适用场景：{场景}

我的推荐：方案 {A/B}，原因：{理由}。

请确认选择哪个方案？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
共设计 {N} 个模块，{N} 个 API 端点。
Review 已通过，方案文档已生成到 .codeman/docs/design/ 和 .codeman/docs/api/

三次确认均已完成：
  ✅ 确认①：模块划分
  ✅ 确认②：API 契约审阅
  ✅ 确认③：方案细节审阅

下一步：开发实现
是否立即开始？（可指定从哪个模块开始）
```
