# CodeMan 技术方案 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code 为 `~/.claude/skills/.codeman`。

---

## 概述

基于已确认的 PRD 碎片文件，生成模块化的技术设计方案。核心流程：**分析 PRD → 模块拆分 → 方案设计 → Review → 人工确认**。

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束（技术栈、架构约束等）
   → 特别关注"需求精粹卡"章节，确保设计方案覆盖所有核心功能和绝对约束

2. 读取 .codeman/config.yaml
   → 了解技术栈配置

3. 读取 .codeman/docs/prd/INDEX.md
   → 获取所有已确认的功能点列表

4. 按需加载 PRD 碎片文件
   → 读取当前任务相关的 feat-*.md

5. 读取 .codeman/docs/design/INDEX.md（如存在）
   → 了解已有架构，避免重复设计

6. 读取 .codeman/rules/INDEX.md
   → 加载项目规范约束
```

---

## 执行步骤

### Step 1: 分析 PRD，识别模块边界

遍历所有 PRD 功能点，按以下维度进行模块归类：

- **业务领域**（用户、订单、支付、通知等）
- **技术层次**（前端组件、后端服务、数据层、基础设施）
- **依赖关系**（哪些模块依赖哪些模块）

输出模块依赖图（ASCII 格式）：
```
[模块A] ──依赖──→ [模块B]
                    │
                    └──依赖──→ [模块C]
```

### Step 2: 为每个模块生成技术方案

**每个模块的技术方案包含：**

```markdown
## 模块：{模块名称}

### 职责
{一句话描述模块职责}

### 技术选型
| 技术点 | 选型 | 理由 |
|--------|------|------|
| {技术点} | {选型} | {简要理由} |

### 数据模型
{关键数据结构定义（伪代码或 TypeScript/Go 类型）}

### 核心逻辑
{关键算法或业务逻辑的伪代码描述}

### 模块接口（对外暴露）
{函数签名或 API 端点列表}

### 依赖
{依赖的其他模块或外部服务}

### 架构决策记录
#### DR-{ID}: {决策名称}
- 日期：{YYYY-MM-DD}
- 备选：A. {方案A} B. {方案B}
- 决策：{选择的方案}
- 理由：{选择原因}
```

### Step 3: 生成 API 设计

为每个涉及 API 的模块生成 API 碎片文件：

**文件路径：** `.codeman/docs/api/api-{模块名}.md`

**文件内容：**
```markdown
# API: {模块名称}

> 最后更新：{YYYY-MM-DD}

## 端点列表

### {HTTP方法} {路径}
- **描述：** {功能描述}
- **鉴权：** {是否需要 / 鉴权方式}
- **请求参数：**
  ```json
  {请求体示例}
  ```
- **响应：**
  ```json
  {响应体示例}
  ```
- **错误码：**
  | 状态码 | 含义 |
  |--------|------|
  | 400 | {含义} |
  | 401 | 未授权 |
  | 404 | 资源不存在 |
  | 500 | 服务器错误 |
- **对应 PRD：** F{ID}.{子ID}
```

### Step 4: 【自动 Review 门禁】

技术方案完成后，**立即调用 Review Skill** 进行方案审查：

审查重点：
- 与 PRD 的一致性（每个功能点是否都有对应的技术实现）
- 技术可行性（选型是否合理、是否有技术风险）
- 架构合理性（模块边界是否清晰、依赖关系是否合理）
- 边界场景覆盖（PRD 中的边界场景是否在技术方案中有处理）
- 安全性（是否有明显的安全漏洞）

Review 发现问题 → 自行修复 → 再次 Review → 通过后继续

### Step 5: 生成技术方案碎片文件

**文件路径：** `.codeman/docs/design/mod-{模块名}.md`

使用 Step 2 中的模块技术方案内容填充。

### Step 6: 更新索引文件

更新 `.codeman/docs/design/INDEX.md`：
```markdown
# 技术方案索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}

| 模块 | 文件 | 状态 | PRD 覆盖 | 最后更新 |
|------|------|------|---------|---------|
| {模块名} | mod-{name}.md | ✅ 已确认 | F001,F002 | {日期} |
```

更新 `.codeman/docs/api/INDEX.md`：
```markdown
# API 接口索引

> 最后更新：{YYYY-MM-DDTHH:MM:SS}
> 接口总数：{N}

| 接口 | 方法 | 路径 | 模块 | 状态 |
|------|------|------|------|------|
| {接口名} | GET | /api/... | {模块} | ✅ 已设计 |
```

### Step 7: 【人工确认点】展示技术方案摘要

向用户展示：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
技术方案摘要 — 请确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块划分：
  - {模块A}：{一句话描述}
  - {模块B}：{一句话描述}

关键技术选型：
  - {技术点}：{选型}（{理由}）

架构决策：
  - DR-001: {决策摘要}

API 端点：{N} 个
数据模型：{N} 个

Review 结果：✅ 通过（{N} 个 Medium 问题已修复）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
确认方案？（或提出修改意见）
```

### Step 8: 更新 STATUS.md

```markdown
- phase: development
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（设计列标记 ✅）。

## 不确定即停

遇到以下情况必须暂停，严禁自行假设：

- **技术选型存在多个可行方案**：列出各方案优劣，暂停由用户决策
- **架构方向不明确**：如单体 vs 微服务、REST vs GraphQL
- **PRD 中有技术约束未明确**：如性能要求、并发量
- **依赖的外部服务不确定**：如第三方 API、云服务选型

**暂停格式：**
```
[需人工确认] 在{技术点}的选型上，有以下几个可行方案：

方案 A：{方案名}
  优点：{优点}
  缺点：{缺点}
  适用场景：{场景}

方案 B：{方案名}
  优点：{优点}
  缺点：{缺点}
  适用场景：{场景}

我推荐方案 {A/B}，原因：{理由}。
请确认选择哪个方案？
```

## 版本迭代时的增量处理

当 design/INDEX.md 已有内容时：

1. 识别受影响的模块（基于 PRD 变更分析）
2. 只更新受影响的 mod-*.md 文件
3. 在文件中记录变更历史
4. 检查 API 变更的向后兼容性
5. 向用户展示变更影响范围

---

## 更新项目概览文档

技术方案 Review 通过后，读取 `.codeman/docs/PROJECT-OVERVIEW.md`，更新以下章节：

**「技术架构」章节：**
- 从 config.yaml 和 arch-overview.md 汇总技术栈全景
- 用 1-2 段自然语言描述整体架构风格和数据流向
- 附 `→ 详见 design/arch-overview.md` 链接

**「模块结构」章节：**
- 从 design/INDEX.md 汇总所有模块到表格（模块名、职责、核心文件、依赖模块、详情链接）
- 从 arch-overview.md 提取模块依赖关系图

**「API 概览」章节：**
- 从 api/INDEX.md 汇总所有接口到表格（接口名、方法、路径、所属模块、鉴权、详情链接）
- 纯前端项目标注「不适用」

**「关键决策记录」章节：**
- 从 DIRECTIVES.md 提取架构决策摘要
- 从各 mod-*.md 中提取重要的技术选型决策及其原因

**「开发规范速查」章节：**
- 从 .codeman/rules/ 下的 .mdc 文件提取核心规范要点（Git 规范、命名规范、代码质量、安全基线）

**「当前进度」章节：**
- 从 STATUS.md 同步最新进度

> 如果 PROJECT-OVERVIEW.md 不存在，基于模板创建并填充当前已有信息。
