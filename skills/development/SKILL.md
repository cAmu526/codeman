# CodeMan 开发实现 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。读取模板文件时请根据当前环境自动选择。

---

## 概述

按技术方案逐模块编码，每个模块遵循"前置对照 → 编码 → 单测 → 后置自检 → Review → Git commit"的完整循环。

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束
   → 检查是否包含 legacy_mode: skeleton（骨架接入标记）

2. 读取 .codeman/docs/design/INDEX.md（或 arch-overview.md）
   → 了解所有模块和当前任务的技术方案

3. 读取当前任务的技术方案碎片文件
   → .codeman/docs/design/mod-{当前模块}.md
   → 【关键】如文件不存在或标记为"待按需生成"，触发「按需文档化」流程（见下方）

4. 读取当前任务的 PRD 碎片文件
   → .codeman/docs/prd/feat-*.md（与当前模块相关的）
   → 【关键】如文件不存在，同样在「按需文档化」流程中一并生成

5. 读取 .codeman/rules/INDEX.md
   → 加载项目规范（编码规范、命名规范等）

6. 读取 .codeman/config.yaml
   → 了解技术栈、测试框架配置

7. 每个模块开始前重新读取 .codeman/docs/DIRECTIVES.md
   → 执行"周期性上下文回顾协议"
   → 对照"需求精粹卡"中的绝对约束和关键验收标准
```

## 按需文档化（骨架模式旧项目专用）

**触发条件：** 满足以下任一条件时触发：
- `DIRECTIVES.md` 中存在 `legacy_mode: skeleton`
- 当前模块的 `mod-{模块名}.md` 不存在
- `arch-overview.md` 中该模块状态为 `⬜ 待按需生成`

**执行前先向用户确认：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[按需文档化] 检测到 {模块名} 尚无详细文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
将在编码前扫描该模块代码，生成：
  - design/mod-{模块名}.md（技术方案）
  - prd/feat-{功能名}.md（功能点 PRD）
预估额外消耗：~3K-6K tokens，约 2-3 分钟

是否继续？（直接回复即可）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户确认后，执行以下步骤（精确范围，不扫描无关代码）：**

### 步骤 1：精确定位模块文件

从 `arch-overview.md` 的模块清单中找到当前模块的主要文件路径，**只读取该模块相关的文件**，不扫描整个项目。

读取范围：
- 模块主文件（Service / Controller / Component 等）
- 数据模型定义文件（types / models / schema 等）
- 路由文件（如有，仅该模块的路由）

### 步骤 2：生成模块技术方案

基于 `{CODEMAN_HOME}/templates/design/mod-template.md` 生成 `mod-{模块名}.md`：

- 模块职责（从代码推断）
- 技术选型（从已有实现推断）
- 核心数据模型（从类型定义/结构体读取）
- 模块对外接口（函数签名 / 组件 Props / 路由）
- 已知技术债务（TODO 注释 / 明显代码异味）

写入：`.codeman/docs/design/mod-{模块名}.md`

### 步骤 3：生成功能点 PRD

识别该模块对外暴露的功能，生成 `feat-{功能名}.md`：

```markdown
# F{ID}: {功能点名称}（现有功能）

## 功能描述
{从代码行为推断}

## 现有实现入口
- 文件：{主要实现文件路径}
- 入口：{函数/组件/路由}

## 已知行为（验收基准）
- [ ] {可观测的行为}

## 已知缺陷 / 技术债
- {如有}

## 状态
existing（现有功能，非本次新增）
```

写入：`.codeman/docs/prd/feat-{功能名}.md`

### 步骤 4：更新索引和覆盖状态

- 更新 `.codeman/docs/prd/INDEX.md`，追加新功能点
- 更新 `.codeman/docs/design/INDEX.md`，追加新模块
- 更新 `arch-overview.md` 中该模块状态：`⬜ 待按需生成` → `✅ {日期}`

### 步骤 5：人工确认文档准确性

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[按需文档化] {模块名} 文档已生成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块职责：{一句话}
识别到功能点：{N} 个
  F{ID}: {功能名} — {一句话描述}
数据模型：{N} 个
对外接口：{N} 个
技术债务：{N} 项（如有）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文档是否准确？确认后继续编码。
（如有偏差请指出，AI 立即修正）
```

**用户确认后，继续正常的编码流程（强制前置对照检查）。**

## 执行步骤

### Step 1: 复杂任务判定与拆分

**满足以下任一条件，必须强制拆分为子步骤：**

| 条件 | 说明 |
|------|------|
| 涉及 3+ 模块联动 | 跨模块影响 |
| 多层条件分支（嵌套 ≥ 3 层） | 逻辑复杂 |
| 新增 3+ API 端点 | 接口量大 |
| 涉及数据模型变更 | 影响面广 |
| 用户明确标注复杂 | 尊重判断 |

**拆分方案格式：**
```
本次任务较复杂，建议拆分为以下子步骤：

子步骤 1：{名称}
  输入：{依赖的前置条件}
  输出：{产出文件/功能}
  涉及文件：{文件列表}
  逻辑概要：{简要描述}

子步骤 2：{名称}
  ...

确认拆分方案后逐步执行，每步完成后展示结果。
```

### Step 2: 逐模块编码

按技术方案中的模块顺序（依赖关系决定顺序）逐一实现：

**编码原则（严格遵守）：**
- 严格按照技术方案中的数据模型、接口定义、核心逻辑实现
- 遵守 `.codeman/rules/` 中加载的所有规范
- 禁止凭推测补充 PRD 未提及的业务逻辑
- 禁止使用 `any` 类型（TypeScript 项目）
- 每个函数/方法必须有明确的输入输出类型
- 错误处理必须显式，不得静默吞掉错误
- 如 PRD 中有设计截图引用，编码 UI 组件前必须先用 Read 工具读取截图文件，理解视觉布局后再编码
- 如发现需要添加 PRD 未提及的功能或逻辑，必须标注 [UNCERTAIN: PRD 未提及] 并等待确认

**编码顺序（推荐）：**
1. 数据模型/类型定义
2. 数据访问层（Repository/DAO）
3. 业务逻辑层（Service）
4. 接口层（Controller/Handler/Route）
5. 前端组件（如适用）

### Step 3: 同步编写单元测试

**每个函数/组件完成后立即编写对应的单元测试，不得推迟。**

**测试覆盖维度：**
- 正常路径（Happy Path）：标准输入 → 预期输出
- 边界条件：空值、最大值、最小值、边界数据
- 异常输入：非法格式、超长字符串、类型错误
- 错误处理：网络超时、数据库错误、第三方服务失败

**覆盖率要求：**
- 核心业务逻辑：≥ 80%
- 工具函数/纯函数：≥ 90%

**测试文件命名：**
- 与被测文件同目录：`{filename}.test.{ext}` 或 `__tests__/{filename}.test.{ext}`

### Step 4: 运行测试验证

每个模块编码完成后立即运行测试：

```bash
# 运行当前模块的单元测试
{项目测试命令} --testPathPattern={模块路径}
```

- 测试全部通过 → 继续下一步
- 测试失败 → 修复代码 → 重新运行 → 通过后继续

### Step 5: 强制后置对照自检

编码完成后，逐条对照前置清单自检，输出对照表：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
需求实现对照表
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
| 功能点 | 实现状态 | 偏差说明 |
|--------|---------|---------|
| F001.1 {子功能} | ✅ 已完成 | - |
| F001.2 {子功能} | ✅ 已完成 | - |
| U1 {UI 要求} | ⚠️ 部分完成 | {偏差描述} [需人工确认] |
| T1 {技术约束} | ✅ 已完成 | - |
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**处理规则：**
- ✅ 已完成：继续
- ⚠️ 部分完成：说明偏差，标注 `[需人工确认]`，等待用户决策
- ❌ 未完成：必须修复，不得跳过

### Step 6: 【自动 Review 门禁】

后置自检通过后，**立即调用 Review Skill** 进行代码审查：

审查重点：
- 与技术方案的一致性
- 代码质量（可读性、可维护性）
- 规范符合度（命名、结构、注释）
- 潜在 Bug（边界条件、并发问题、内存泄漏）
- 安全漏洞（SQL 注入、XSS、权限绕过）

Review 通过 → 进入 Git commit
Review 发现 Critical/High 问题 → 自行修复 → 再次 Review → 通过

### Step 7: Git Checkpoint

每个模块编码完成并通过 Review 后，执行 Git commit：

```bash
git add {相关文件}
git commit -m "feat({模块名}): {功能描述}

- 实现 {功能点1}
- 实现 {功能点2}
- 添加单元测试（覆盖率 {N}%）

Refs: F{ID}"
```

### Step 8: 更新 STATUS.md

每个模块完成后更新进度：

```markdown
- current_module: {下一个模块}
- current_step: coding (module {X+1}/{N})
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（编码列、单测列标记 ✅）。

## 多模块迭代

所有模块按以下循环执行：

```
对于每个模块：
  0. **上下文回顾** → 重新读取 DIRECTIVES.md，输出回顾摘要（≤3行）
  1. 前置对照检查 → 人工确认
  2. 编码实现
  3. 单元测试
  4. 运行测试
  5. 后置对照自检
  6. Review 门禁
  7. Git commit
  8. 更新 STATUS.md
  → 继续下一个模块
```

---

## 不确定即停

遇到以下情况必须暂停，严禁自行推测：

- PRD 或技术方案中有歧义
- 实现过程中发现技术方案有遗漏或错误
- 需要引入 PRD 未提及的第三方库
- 发现潜在的安全风险，修复方案不确定
- 实现方式与技术方案有重大偏差

**标记格式：** `[UNCERTAIN: {描述}]` 或 `[需人工确认: {描述}]`

## 更新项目概览文档

所有模块编码完成后（进入测试前），读取 `.codeman/docs/PROJECT-OVERVIEW.md`，更新以下章节：

**「功能地图」章节：**
- 将已完成编码的功能点状态从 `📋 已设计` 更新为 `🔄 开发中` 或 `✅ 已上线`（视测试情况）

**「当前进度」章节：**
- 从 STATUS.md 同步最新的功能点进度表
- 更新当前阶段为 testing

**「新成员上手指南」章节：**
- 如果编码过程中产生了新的环境搭建步骤（如新增依赖、新增环境变量），同步更新

> 如果 PROJECT-OVERVIEW.md 不存在，跳过此步骤。

## 产出清单

| 产出 | 路径 | 说明 |
|------|------|------|
| 业务代码 | 项目源码目录 | 按技术方案实现 |

| 项目概览更新 | `.codeman/docs/PROJECT-OVERVIEW.md` | 更新功能地图状态和当前进度 |
