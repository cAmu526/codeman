# CodeMan 开发实现 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`，Trae 为 `~/.trae/skills/.codeman`。读取模板文件时请根据当前环境自动选择。

---

## 概述

按技术方案逐模块编码，每个模块遵循"前置对照 → 编码 → 单测 → 后置自检 → Review → Git checkpoint"的完整循环。开发在 feature 分支上进行，测试通过后 squash merge 回主分支。

---

## 前置条件

执行前必须完成：

```
0. 创建 feature 分支（如 config.yaml 的 git.branch_strategy 为 true）
   → 读取 config.yaml 中的 git.branch_prefix（默认 "feat/"）
   → 分支名格式：{prefix}{功能描述简写}，如 feat/batch-opportunity-creation
   → 如果当前已在 feature 分支上（断点续做场景），跳过此步
   → 执行：git checkout -b {分支名}

1. 读取 .codeman/docs/DIRECTIVES.md
   → 加载核心约束
   → 检查是否包含 legacy_mode: skeleton（骨架接入标记）

2. 读取 .codeman/docs/design/INDEX.md（或 arch-overview.md）
   → 了解所有模块和当前任务的技术方案

3. 读取当前任务的技术方案碎片文件
   → .codeman/docs/design/mod-{当前模块}.md
   → 【关键】如文件不存在或标记为"待按需生成"，触发「按需文档化」流程（见下方）

4. 读取当前任务的 PRD 碎片文件
   → .codeman/docs/prd/feat-*.md（与当前模块相关的）
   → 【关键】如文件不存在，同样在「按需文档化」流程中一并生成

5. 读取 .codeman/rules/INDEX.md
   → 加载项目规范（编码规范、命名规范等）

6. 读取 .codeman/config.yaml
   → 了解技术栈、测试框架配置

7. 读取 .codeman/docs/DOMAIN-RULES.md（如存在）
   → 加载全局领域规则（术语定义、跨模块不变量、跨功能影响、实体约束）
   → 如文件不存在，跳过（不阻塞流程）

8. 读取已锁定的测试用例（.codeman/docs/tests/test-*.md）
   → 这些用例在编码前已由 testing Skill 阶段一生成并经用户确认
   → 编码的目标是让代码通过这些用例
   → 编码时对照用例中的预期结果，确保实现覆盖所有验收标准

9. 每个模块开始前重新读取 .codeman/docs/DIRECTIVES.md + .codeman/docs/DOMAIN-RULES.md
   → 执行"周期性上下文回顾协议"，**必须输出以下格式的回顾摘要**（不得省略）：

   ```
   ━━━ 上下文回顾（模块 {模块名} 开始前）━━━
   精粹卡核心约束：
     1. {约束1}
     2. {约束2}
     3. ...（列出所有绝对约束）
   本模块相关的领域规则：
     - {规则1}（来自 DOMAIN-RULES.md / mod-*.md）
     - {规则2}
     - 无相关规则（如确实没有）
   本模块对应的测试用例（编码前已锁定）：
     - {用例1} — 预期：{预期结果}
     - {用例2} — 预期：{预期结果}
   上个模块遗留问题：{有：xxx / 无}
   偏离风险检查：{无偏离 / ⚠️ 发现偏离：xxx}
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   ```

   → 如果发现偏离风险，立即暂停报告用户
```

## 按需文档化（骨架模式旧项目专用）

**触发条件：** 满足以下任一条件时触发：
- `DIRECTIVES.md` 中存在 `legacy_mode: skeleton`
- 当前模块的 `mod-{模块名}.md` 不存在
- `arch-overview.md` 中该模块状态为 `⬜ 待按需生成`

**执行前先向用户确认：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[按需文档化] 检测到 {模块名} 尚无详细文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
将在编码前扫描该模块代码，生成：
  - design/mod-{模块名}.md（技术方案）
  - prd/feat-{功能名}.md（功能点 PRD）
预估额外消耗：~3K-6K tokens，约 2-3 分钟

是否继续？（直接回复即可）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户确认后，执行以下步骤（精确范围，不扫描无关代码）：**

### 步骤 1：精确定位模块文件

从 `arch-overview.md` 的模块清单中找到当前模块的主要文件路径，**只读取该模块相关的文件**，不扫描整个项目。

读取范围：
- 模块主文件（Service / Controller / Component 等）
- 数据模型定义文件（types / models / schema 等）
- 路由文件（如有，仅该模块的路由）

### 步骤 2：生成模块技术方案

基于 `{CODEMAN_HOME}/templates/design/mod-template.md` 生成 `mod-{模块名}.md`：

- 模块职责（从代码推断）
- 技术选型（从已有实现推断）
- 核心数据模型（从类型定义/结构体读取）
- 模块对外接口（函数签名 / 组件 Props / 路由）
- 已知技术债务（TODO 注释 / 明显代码异味）

写入：`.codeman/docs/design/mod-{模块名}.md`

### 步骤 3：生成功能点 PRD

识别该模块对外暴露的功能，生成 `feat-{功能名}.md`：

```markdown
# F{ID}: {功能点名称}（现有功能）

## 功能描述
{从代码行为推断}

## 现有实现入口
- 文件：{主要实现文件路径}
- 入口：{函数/组件/路由}

## 已知行为（验收基准）
- [ ] {可观测的行为}

## 已知缺陷 / 技术债
- {如有}

## 状态
existing（现有功能，非本次新增）
```

写入：`.codeman/docs/prd/feat-{功能名}.md`

### 步骤 4：更新索引和覆盖状态

- 更新 `.codeman/docs/prd/INDEX.md`，追加新功能点
- 更新 `.codeman/docs/design/INDEX.md`，追加新模块
- 更新 `arch-overview.md` 中该模块状态：`⬜ 待按需生成` → `✅ {日期}`

### 步骤 5：人工确认文档准确性

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[按需文档化] {模块名} 文档已生成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块职责：{一句话}
识别到功能点：{N} 个
  F{ID}: {功能名} — {一句话描述}
数据模型：{N} 个
对外接口：{N} 个
技术债务：{N} 项（如有）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文档是否准确？确认后继续编码。
（如有偏差请指出，AI 立即修正）
```

**用户确认后，继续正常的编码流程（强制前置对照检查）。**

## 第三方 API 文档查阅（编码前自动触发）

**目的：** 防止使用过时的第三方 API/SDK 用法，确保代码兼容当前版本。

**触发条件（满足任一即触发）：**
- 当前模块的技术方案（`mod-*.md`）中提到了第三方 API/SDK（如 Stripe、OpenAI、AWS SDK 等）
- 当前模块需要 `import` 或调用第三方库的接口
- 用户主动要求查阅第三方文档

### 流程

#### 1. 检测第三方 API 依赖

从技术方案和待编码的模块中识别涉及的第三方 API/SDK：
- 读取 `mod-{模块名}.md` 中的技术选型和依赖库
- 检查项目 `package.json` / `requirements.txt` / `go.mod` 等依赖声明文件

#### 2. 查找已配置的文档 URL

读取 `.codeman/config.yaml` 中的 `api_references` 字段，检查是否已声明对应 API 的文档 URL。

**情况 A：已配置文档 URL**
→ 使用 WebFetch 抓取官方文档中与当前模块相关的部分（如认证方式、具体端点、SDK 方法签名等），提取关键用法信息。

**情况 B：未配置文档 URL**
→ 提醒用户补充：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[第三方 API] 检测到本模块使用了 {API/SDK 名称}，但 config.yaml 中未配置文档链接
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
建议添加官方文档 URL，确保编码时参考最新 API 用法，避免使用过时接口。

请提供文档链接，我将添加到 config.yaml 的 api_references 中：
  name: "{API/SDK 名称}"
  doc_url: "{官方文档 URL}"
  version: "{API 版本，可选}"
  notes: "{备注，可选}"

或回复"跳过"继续编码（将基于已有知识，可能存在版本不兼容风险）。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

- 用户提供 URL → 写入 `config.yaml` 的 `api_references` → 继续抓取文档
- 用户回复"跳过" → 标注 `[⚠️ 未参考最新文档]`，继续编码

#### 3. 缓存文档摘要

将抓取到的关键 API 信息缓存到 `.codeman/docs/api-refs/{api名称}.md`，内容包括：

```markdown
# {API/SDK 名称} 参考摘要

> 抓取时间：{YYYY-MM-DD}
> 文档来源：{URL}
> API 版本：{版本号}

## 本模块用到的关键接口

### {接口/方法名}
- 签名：{函数签名或请求格式}
- 参数：{关键参数及类型}
- 返回：{返回值格式}
- 注意事项：{废弃警告、版本变更等}

### {接口/方法名}
...
```

**缓存规则：**
- 同一会话内，同一 API 不重复抓取（直接读取缓存文件）
- 缓存文件超过 7 天 → 重新抓取更新
- 缓存文件不提交到 Git（在 `.codeman/.gitignore` 中排除 `docs/api-refs/`），每个开发者本地维护

#### 4. 编码时参照

在 Step 2 逐模块编码时，涉及第三方 API 调用的代码必须参照缓存的文档摘要：
- 方法签名、参数类型、返回格式以文档为准
- 如发现文档中标注了废弃（deprecated）接口，使用推荐的替代方案
- 如文档版本与项目依赖版本不一致，暂停提醒用户确认

## 执行步骤

### Step 1: 复杂任务判定与拆分

**满足以下任一条件，必须强制拆分为子步骤：**

| 条件 | 说明 |
|------|------|
| 涉及 3+ 模块联动 | 跨模块影响 |
| 多层条件分支（嵌套 ≥ 3 层） | 逻辑复杂 |
| 新增 3+ API 端点 | 接口量大 |
| 涉及数据模型变更 | 影响面广 |
| 用户明确标注复杂 | 尊重判断 |

**拆分方案格式：**
```
本次任务较复杂，建议拆分为以下子步骤：

子步骤 1：{名称}
  输入：{依赖的前置条件}
  输出：{产出文件/功能}
  涉及文件：{文件列表}
  逻辑概要：{简要描述}

子步骤 2：{名称}
  ...

确认拆分方案后逐步执行，每步完成后展示结果。
```

### Step 2: 逐模块编码

按技术方案中的模块顺序（依赖关系决定顺序）逐一实现：

**编码原则（严格遵守）：**
- 严格按照技术方案中的数据模型、接口定义、核心逻辑实现
- 遵守 `.codeman/rules/` 中加载的所有规范
- 禁止凭推测补充 PRD 未提及的业务逻辑
- 禁止使用 `any` 类型（TypeScript 项目）
- 每个函数/方法必须有明确的输入输出类型
- 错误处理必须显式，不得静默吞掉错误
- 如 PRD 中有设计截图引用，编码 UI 组件前必须先用 Read 工具读取截图文件，理解视觉布局后再编码
- 如发现需要添加 PRD 未提及的功能或逻辑，必须标注 [UNCERTAIN: PRD 未提及] 并等待确认
- **前端交互元素必须添加 `data-testid` 属性**，为 E2E 测试提供稳定的定位锚点（不依赖文案/样式类名，改文案不会导致测试全挂）

**`data-testid` 命名规范：**

| 元素类型 | 命名格式 | 示例 |
|---------|---------|------|
| 按钮 | `{模块}-{动作}-btn` | `data-testid="user-register-btn"` |
| 表单 | `{模块}-form` | `data-testid="login-form"` |
| 输入框 | `{模块}-{字段}-input` | `data-testid="register-email-input"` |
| 列表容器 | `{模块}-list` | `data-testid="user-list"` |
| 列表行 | `{模块}-row` | `data-testid="user-row"` |
| 弹窗/抽屉 | `{模块}-{类型}` | `data-testid="visit-modal"` |
| 下拉触发器 | `{模块}-{字段}-select` | `data-testid="visit-opportunity-select"` |
| 提示/反馈 | `{类型}-msg` | `data-testid="success-msg"` |

**必须加 `data-testid` 的元素（最小集合）：**
- 所有用户可点击的按钮（提交/取消/新增/删除/编辑）
- 所有表单输入框
- 所有下拉选择器的触发器
- 列表/表格的容器和行
- 弹窗/抽屉的容器
- 成功/错误/loading 反馈元素
- 导航菜单项

**编码顺序（推荐）：**
1. 数据模型/类型定义
2. 数据访问层（Repository/DAO）
3. 业务逻辑层（Service）
4. 接口层（Controller/Handler/Route）
5. 前端组件（如适用）

#### 变更影响面扫描（每次修改现有代码时必须执行）

```
⚠️ 需人工确认
Review 发现 {N} 个 Critical/High 问题：

1. [{严重度}] {问题描述}
   文件：{文件路径}:{行号}
   修复方案：{具体修复方式}

确认修复？（直接回复即可，或说"跳过第 N 个"/"只修 Critical"）
```

### Step 8: Git Checkpoint

每个模块编码完成并通过 Review 后，在**当前 feature 分支**上执行 Git commit（安全存档，非最终提交）：

```bash
git add {相关文件}
git commit -m "wip({模块名}): {功能描述}

- 实现 {功能点1}
- 实现 {功能点2}
- 添加单元测试（覆盖率 {N}%）

Refs: F{ID}"
```

> 这些 commit 是开发过程中的 checkpoint，最终会在测试通过后 squash merge 为一个干净的 commit。如果 config.yaml 的 `git.branch_strategy` 为 false，则直接 commit 到当前分支（兼容旧行为）。

**⚠️ 绝对禁止执行 `git push`。推送操作必须由用户手动完成。**

### Step 8.5: 领域规则审查（毕业 / 升级 / 新增）

模块编码完成并通过 Review 后，审查与本模块相关的领域规则：

**1. 毕业检查 — 已被代码/测试覆盖的规则可以移除**

检查 `DOMAIN-RULES.md` 和当前 `mod-*.md` 的「领域规则」章节中与本模块相关的规则：
- 该规则是否已有对应的代码校验逻辑（validator/middleware/constraint）？
- 该规则是否已有对应的测试用例？

如有可毕业的规则，提示用户确认：
```
以下领域规则已被代码/测试覆盖，建议移除：
  - INV-001: 余额校验 → 已有 validateBalance() + test_balance_invariant
  - R2: 状态单向流转 → 已有 OrderStateMachine.transition() 校验

确认移除？[Y/n]（输入 n 或指定保留的规则编号）
```

**2. 升级检查 — 反复出现的同类问题应升级为编码规范**

回顾本模块开发过程中踩过的坑，检查是否与已有领域规则中的踩坑记录属于同类：
- 如果同类问题已出现 2 次以上 → 提示用户升级为 `.codeman/rules/` 编码规范

```
以下类型的问题重复出现，建议升级为编码规范（从源头预防）：
  - "并发写入未加锁" — 已在 mod-order 和 mod-inventory 中各出现一次
  建议生成: rules/proj-concurrency-guard.mdc

确认升级？[Y/n]
```

**3. 新增检查 — 开发中发现的新隐性知识**

如果本模块编码过程中发现了新的隐性知识（业务约束、术语歧义、跨模块影响等），提示写入：
- 仅涉及本模块 → 写入 `mod-*.md` 的「领域规则」章节
- 跨模块 → 写入 `DOMAIN-RULES.md`（写入前检查剩余容量，接近 1500 字时先提示清理）

> 如果本模块无需审查（无相关领域规则、无踩坑、无新发现），跳过此步。

```markdown
- current_module: {下一个模块}
- current_step: coding (module {X+1}/{N})
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```

更新功能点进度表（编码列、单测列标记 ✅）。

## 不确定即停（遵守 orchestrator 全局硬规则）

遇到以下情况必须暂停，**严禁自行推测、补充或执行未经确认的逻辑**：

- PRD 或技术方案中有歧义
- 实现过程中发现技术方案有遗漏或错误
- 需要引入 PRD 未提及的第三方库
- 发现潜在的安全风险，修复方案不确定
- 实现方式与技术方案有重大偏差
- 不确定某个字段的类型、约束或默认值
- 不确定某个边界场景该怎么处理

**暂停格式：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
关于"{不确定点}"，我理解到以下几种可能：

1. {理解A}：[对应的实现方式]
2. {理解B}：[对应的实现方式]

请问您的意思是哪种？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**绝对禁止：**
- ❌ 用 `[UNCERTAIN]` 标记后不暂停直接继续编码
- ❌ 选择"最合理"的方案自行实现
- ❌ 在确认前修改文件
