# CodeMan 修复闭环 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。

---

## 概述

对测试失败或用户报告的 Bug 进行完整的修复闭环：**定位根因 → 评估影响 → 修复 → 回归验证 → 文档同步**。最多自动修复 3 轮，超过则暂停人工介入。

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/tests/test-report-latest.md
   → 获取失败用例列表和失败原因

2. 读取相关 PRD 碎片文件
   → 了解预期行为（修复基准来源于 PRD）

3. 读取相关技术方案碎片文件
   → 了解设计意图，判断是实现问题还是设计问题

4. 读取 .codeman/docs/STATUS.md
   → 了解当前项目状态

5. 读取 .codeman/docs/DIRECTIVES.md
   → 执行"周期性上下文回顾协议"
   → 检查"需求精粹卡"中的绝对约束，确保修复方案不违反核心约束

6. 检查骨架模式
   → 如 DIRECTIVES.md 中存在 legacy_mode: skeleton
   → 检查涉及模块的 mod-{模块名}.md 是否存在
   → 不存在则触发「按需文档化」（见下方）
```

### 骨架模式：修复前按需文档化

**触发条件：** DIRECTIVES.md 中存在 `legacy_mode: skeleton`，且修复涉及的模块没有详细文档。

**执行：** 参照 development Skill 的按需文档化流程（见 development/SKILL.md「按需文档化」章节），在修复前扫描涉及模块的代码，生成对应的 `mod-{模块名}.md` 和 `prd/feat-{功能名}.md`。

**为什么修复前也要文档化：** 没有详细文档就无法准确判断根因是"实现 Bug"还是"设计缺陷"，可能导致修复方向错误。

---

## 执行步骤

### Step 1: 分析失败用例，定位根因

对每个失败用例进行根因分析：

**根因分类：**
| 类型 | 描述 | 处理方式 |
|------|------|---------|
| 实现 Bug | 代码逻辑错误 | 修复代码 |
| 测试用例错误 | 测试断言有误 | 修复测试用例 |
| 环境问题 | 测试环境配置问题 | 修复环境配置 |
| 设计缺陷 | 技术方案有问题 | 更新技术方案后修复 |
| PRD 歧义 | 需求不明确导致实现偏差 | 暂停，人工确认 |

**根因分析输出：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Bug 分析报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
失败用例：{用例名称}
失败原因：{错误信息}
根因类型：{类型}
根因描述：{详细分析}
影响范围：{受影响的功能点/模块}
修复方案：{具体修复步骤}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 2: 评估影响范围

修复前评估：
- 修复会影响哪些文件？
- 修复是否可能引入新的 Bug？
- 是否需要同步更新技术方案文档？
- 是否需要更新 API 文档？

**如果影响范围超出预期（影响 3+ 模块或涉及架构变更）：**
→ 暂停，向用户确认修复方案后再继续

### Step 3: 执行修复（最多 3 轮）

**每轮修复前必须向用户确认修复方案，确认后才能执行。**

**第 1 轮修复：**

向用户展示修复方案并确认：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 修复方案确认（第 1 轮）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
失败用例：{用例名称}
根因：{根因描述}
修复方案：{具体修复步骤}
影响文件：{文件列表}
影响范围：{受影响的功能点}

确认修复？（直接回复即可）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

用户确认后：
1. 实施修复方案
2. 运行受影响的测试用例（回归验证）
3. 记录结果

**如果第 1 轮回归通过 → 进入 Step 4**

**如果第 1 轮回归失败：**
```
第 1 轮修复后仍有 {N} 个测试失败，重新分析...
```
→ 重新分析根因 → 向用户展示新的修复方案 → 第 2 轮修复

**第 2 轮修复：**

向用户展示修复方案并确认：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 修复方案确认（第 2 轮）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
第 1 轮失败原因：{失败原因}
重新分析根因：{新根因}
新修复方案：{调整后的方案}
影响文件：{文件列表}

确认修复？（直接回复即可）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

用户确认后：
1. 实施修复
2. 运行回归测试

**如果第 2 轮回归通过 → 进入 Step 4**

**如果第 2 轮回归失败 → 第 3 轮修复**

**第 3 轮修复：**

向用户展示修复方案并确认：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 修复方案确认（第 3 轮，最后一轮）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
前 2 轮均失败，深入分析：
第 1 轮：{方案} → {失败原因}
第 2 轮：{方案} → {失败原因}

根因：{全面分析}
修复方案：{可能涉及重新设计}
影响范围：{较大范围的影响}

⚠️ 这是最后一轮自动修复，如仍失败将需要人工介入。

确认修复？（直接回复即可）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

用户确认后：
1. 实施修复
2. 运行回归测试

**如果第 3 轮回归仍失败：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[需人工介入] 已尝试 3 轮修复，问题仍未解决
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
问题描述：{详细描述}
已尝试的修复方案：
  1. {方案1} → 结果：{失败原因}
  2. {方案2} → 结果：{失败原因}
  3. {方案3} → 结果：{失败原因}

可能的原因：
  A. {可能原因A}
  B. {可能原因B}

建议：{建议的下一步行动}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请提供更多信息或指导修复方向。
```

### Step 4: 回归通过后的处理

**运行完整回归测试：**
```bash
# 运行所有受影响模块的测试
{项目测试命令} --testPathPattern={受影响模块路径}
```

确认没有引入新的失败用例。

**扩大回归范围：** 除运行受影响模块测试外，还应运行需求精粹卡中所有 P0 功能点
对应的测试用例，确保核心功能未被修复破坏。

### Step 5: 同步文档

根据修复内容，同步更新相关文档。**修改 PRD 或技术方案前必须向用户确认：**

**如果是实现 Bug（代码逻辑错误）：**
- 无需更新 PRD 和技术方案
- 更新测试报告

**如果是设计缺陷（技术方案有问题）：**
```
⚠️ 需人工确认
修复涉及技术方案变更，需要更新以下文档：
- {文件路径}：{变更内容}

确认更新？（直接回复即可）
```
→ 用户确认后更新 `.codeman/docs/design/mod-*.md`，记录变更原因

**如果是 PRD 歧义（需求不明确）：**
```
⚠️ 需人工确认
修复涉及需求变更，需要更新以下文档：
- {文件路径}：{变更内容}
- DIRECTIVES.md：{约束变更}

确认更新？（直接回复即可）
```
→ 用户确认后更新 `.codeman/docs/prd/feat-*.md` 和 DIRECTIVES.md

### Step 6: 更新测试报告

更新 `.codeman/docs/tests/test-report-latest.md`：
- 将修复的用例状态更新为 ✅ 通过
- 记录修复说明

更新 `.codeman/docs/STATUS.md`：
- 更新功能点进度表中的测试状态
- 如所有测试通过，更新 phase

### Step 7: Git Commit

```bash
git add {修复的文件}
git commit -m "fix({模块名}): {修复描述}

- 根因：{根因描述}
- 修复方案：{修复方案描述}
- 影响范围：{影响的功能点}

Fixes: {失败用例ID}"
```

**⚠️ 绝对禁止执行 `git push`。推送操作必须由用户手动完成。**

## 不确定即停（遵守 orchestrator 全局硬规则）

遇到以下情况必须暂停，**严禁自行推测或执行未经确认的修复**：

- 根因不明确，存在多种可能
- 修复方案涉及架构变更
- 修复可能影响其他已通过的功能
- 问题可能是 PRD 本身的歧义导致
- 3 轮修复后仍未解决
- 修复方案可能引入新的安全风险

**暂停格式：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
关于"{问题}"，我分析到以下根因可能：

可能 1：{原因} → 修复方案：{方案} → 影响范围：{范围}
可能 2：{原因} → 修复方案：{方案} → 影响范围：{范围}

我的判断：{最可能的原因}，原因：{简要理由}

请问您确认哪个根因？或者有补充信息？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 批量修复策略

当有多个失败用例时：

1. 按根因分类，相同根因的用例一起修复
2. 按严重程度排序（Critical → High → Medium → Low）
3. 每次修复后运行相关测试，确认修复有效
4. 所有修复完成后，运行完整测试套件

## 产出清单

| 产出 | 路径 | 说明 |
|------|------|------|
| 修复后的代码 | 项目源码 | Bug 修复 |
| 更新的测试报告 | `.codeman/docs/tests/test-report-latest.md` | 修复状态更新 |
| 文档更新（如需） | `.codeman/docs/design/` 或 `prd/` | 设计/需求文档同步 |
| Git commit | Git 历史 | 修复记录 |
| STATUS 更新 | `.codeman/docs/STATUS.md` | 进度更新 |

## 完成后

**所有 Bug 修复完成：**
```
修复闭环完成！
修复了 {N} 个 Bug，回归测试全部通过。

下一步：继续测试验证（L3 E2E / L4 UI 视觉）
或：进入部署清单生成（如所有测试已通过）
```
