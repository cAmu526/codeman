# CodeMan 修复闭环 Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`，Trae 为 `~/.trae/skills/.codeman`。

---

## 概述

对测试失败或用户报告的 Bug 进行完整的修复闭环：**定位根因 → 评估影响 → 修复 → 回归验证 → 文档同步**。最多自动修复 3 轮，超过则暂停人工介入。

---

## 前置条件

执行前必须完成：

```
0. 创建 fix 分支（如 config.yaml 的 git.branch_strategy 为 true）
   → 如果 config.yaml 中 git.branch_strategy 字段不存在，输出提示："⚠️ 分支策略未配置，将在当前分支直接修复"
   → 分支名格式：fix/{修复描述简写}，如 fix/opportunity-save-error
   → 如果当前已在 fix 分支上（批量修复或断点续做），跳过此步
   → 执行：git checkout -b {分支名}

1. 读取 .codeman/docs/tests/test-report-latest.md
   → 获取失败用例列表和失败原因

1.5. 【建立 Bug 回溯链】对每个失败用例，读取其测试用例文件 `.codeman/docs/tests/test-*.md`：
   → 提取该 TC 的 `**实现任务：**` 字段（如 `mod-auth:T2, mod-auth:T3`）
   → 对每个任务 ID，读取 `.codeman/docs/dev/tasks-{模块名}.md`，定位到对应任务条目，提取：
     - 涉及文件清单（修复时优先排查这些文件）
     - Commit hash（用 `git show {hash}` 查看变更内容，辅助根因定位）
     - 执行步骤（理解任务原始意图，避免修复偏离）
   → 如失败用例的 `impl_tasks` 为空或 `-`（旧测试用例未关联）：
     - 记录在"备注"中，跳过任务级回溯，按通用流程定位
     - 修复完成后，建议补全该用例的 `impl_tasks` 字段

2. 读取相关 PRD 碎片文件
   → 了解预期行为（修复基准来源于 PRD）

3. 读取相关技术方案碎片文件
   → 了解设计意图，判断是实现问题还是设计问题

4. 读取 .codeman/docs/STATUS.md
   → 了解当前项目状态

5. 读取 .codeman/docs/DIRECTIVES.md
   → 执行"周期性上下文回顾协议"
   → 检查"需求精粹卡"中的绝对约束，确保修复方案不违反核心约束

6. 检查骨架模式
   → 如 DIRECTIVES.md 中存在 legacy_mode: skeleton
   → 检查涉及模块的 mod-{模块名}.md 是否存在
   → 不存在则触发「按需文档化」（见下方）
```

### 骨架模式：修复前按需文档化

**触发条件：** DIRECTIVES.md 中存在 `legacy_mode: skeleton`，且修复涉及的模块没有详细文档。

**执行：** 参照 development Skill 的按需文档化流程（见 development/SKILL.md「按需文档化」章节），在修复前扫描涉及模块的代码，生成对应的 `mod-{模块名}.md` 和 `prd/feat-{功能名}.md`。

**为什么修复前也要文档化：** 没有详细文档就无法准确判断根因是"实现 Bug"还是"设计缺陷"，可能导致修复方向错误。

## 执行步骤

### Step 1: 分析失败用例，定位根因

对每个失败用例进行根因分析：

**根因分类：**
| 类型 | 描述 | 处理方式 |
|------|------|---------|
| 实现 Bug | 代码逻辑错误 | 修复代码 |
| 测试用例错误 | 测试断言有误 | 修复测试用例 |
| 环境问题 | 测试环境配置问题 | 修复环境配置 |
| 设计缺陷 | 技术方案有问题 | 更新技术方案后修复 |
| PRD 歧义 | 需求不明确导致实现偏差 | 暂停，人工确认 |

**根因分析输出：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Bug 分析报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
失败用例：{用例名称}
失败原因：{错误信息}
根因类型：{类型}
根因描述：{详细分析}
影响范围：{受影响的功能点/模块}

回溯链（来自 Step 1.5）：
  失败用例 → impl_tasks: {mod-auth:T2, mod-auth:T3}
    └─ T2: {任务名} | commit {hash} | 文件 {path1, path2}
    └─ T3: {任务名} | commit {hash} | 文件 {path3}
  （如 impl_tasks 为空，标注"无任务级关联，按通用流程定位"）

修复方案：{具体修复步骤}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 2: 评估影响范围

修复前评估：
- 修复会影响哪些文件？
- 修复是否可能引入新的 Bug？
- 是否需要同步更新技术方案文档？
- 是否需要更新 API 文档？

**如果影响范围超出预期（影响 3+ 模块或涉及架构变更）：**
→ 暂停，向用户确认修复方案后再继续

### Step 3: 执行修复（最多 3 轮）

**确认规则：影响小（≤ 1 文件）直接修，影响大或方案不确定才确认。**

**第 1 轮修复：**
1. 如修复方案明确且影响 ≤ 1 个文件 → 直接实施修复
2. 如修复方案不确定或影响 > 1 个文件 → 向用户展示修复方案，确认后修复
3. 运行受影响的测试用例（回归验证）
4. 记录结果

**如果第 1 轮回归通过 → 进入 Step 4**

**如果第 1 轮回归失败：**
```
第 1 轮修复后仍有 {N} 个测试失败，重新分析...
```
→ 重新分析根因

**第 2 轮修复：**
1. 如新修复方案明确 → 直接实施
2. 如新修复方案不确定 → 向用户展示，确认后修复
3. 运行回归测试

**如果第 2 轮回归通过 → 进入 Step 4**

**如果第 2 轮回归失败 → 第 3 轮修复**

**第 3 轮修复（必须确认，可能涉及重新设计）：**
```
⚠️ 需人工确认
前 2 轮均失败，深入分析：
第 1 轮：{方案} → {失败原因}
第 2 轮：{方案} → {失败原因}

根因：{全面分析}
修复方案：{可能涉及重新设计}
影响范围：{较大范围的影响}

⚠️ 这是最后一轮自动修复，如仍失败将需要人工介入。

确认修复？（直接回复即可）
```

用户确认后实施修复并运行回归测试

**如果第 3 轮回归仍失败：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[需人工介入] 已尝试 3 轮修复，问题仍未解决
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
问题描述：{详细描述}
已尝试的修复方案：
  1. {方案1} → 结果：{失败原因}
  2. {方案2} → 结果：{失败原因}
  3. {方案3} → 结果：{失败原因}

可能的原因：
  A. {可能原因A}
  B. {可能原因B}

建议：{建议的下一步行动}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请提供更多信息或指导修复方向。
```

**必须更新 STATUS.md：**
```markdown
- phase: blocked-needs-human
- blocked_reason: 修复 3 轮失败，需人工介入
- blocked_at: {YYYY-MM-DDTHH:MM:SS}
- last_updated: {YYYY-MM-DDTHH:MM:SS}
```
> 这样 orchestrator 在断点续做时能识别到阻塞状态，提示用户处理。

### Step 4: 回归通过后的处理

**运行完整回归测试：**
```bash
# 运行所有受影响模块的测试
{项目测试命令} --testPathPattern={受影响模块路径}
```

确认没有引入新的失败用例。

**扩大回归范围：** 除运行受影响模块测试外，还应运行需求精粹卡中所有 P0 功能点
对应的测试用例，确保核心功能未被修复破坏。

### Step 5: 同步文档

根据修复内容，同步更新相关文档。**修改 PRD 或技术方案前必须向用户确认：**

**如果是实现 Bug（代码逻辑错误）：**
- 无需更新 PRD 和技术方案
- 更新测试报告

**如果是设计缺陷（技术方案有问题）：**
```
⚠️ 需人工确认
修复涉及技术方案变更，需要更新以下文档：
- {文件路径}：{变更内容}

确认更新？（直接回复即可）
```
→ 用户确认后更新 `.codeman/docs/design/mod-*.md`，记录变更原因

**如果是 PRD 歧义（需求不明确）：**
```
⚠️ 需人工确认
修复涉及需求变更，需要更新以下文档：
- {文件路径}：{变更内容}
- DIRECTIVES.md：{约束变更}

确认更新？（直接回复即可）
```
→ 用户确认后更新 `.codeman/docs/prd/feat-*.md` 和 DIRECTIVES.md

### Step 6: 更新测试报告

更新 `.codeman/docs/tests/test-report-latest.md`：
- 将修复的用例状态更新为 ✅ 通过
- 记录修复说明

更新 `.codeman/docs/STATUS.md`：
- 更新功能点进度表中的测试状态
- 如所有测试通过，更新 phase

### Step 7: 缺陷回溯学习（每个产品 Bug 必须执行）

**修复完成后，对每个产品 Bug（不含测试用例错误和环境问题）强制执行回溯分析，找出流程中哪个环节应该拦住但没拦住，并将改进建议沉淀为规则。**

**为什么：** 同类 Bug 不应该反复出现。如果一个 Bug 能通过需求分析、技术方案、编码、Review 四道关卡到达测试阶段才被发现，说明流程有漏洞。回溯分析找到漏洞，规则沉淀堵住漏洞。

**回溯分析模板：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
缺陷回溯分析 — {Bug 编号}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Bug 描述：{简述}
根因：{详细根因}

回溯检查：
① 需求阶段：PRD 是否明确定义了该行为？
   → {是：PRD feat-xxx.md §X 有明确描述 / 否：PRD 未覆盖此场景}
   → 遗漏原因：{如"PRD 边界场景未列出该情况" / "验收标准描述模糊"}

② 设计阶段：技术方案是否有对应的设计？
   → {是：mod-xxx.md §X 有设计 / 否：技术方案未覆盖}
   → 遗漏原因：{如"方案只设计了正常路径，未考虑并发场景"}

③ 开发阶段：编码对照清单是否包含了这个点？
   → {是：前置清单有但未实现 / 否：清单未列出}
   → 遗漏原因：{如"前置清单粒度不够细，未列出错误码定义"}

④ Review 阶段：代码 Review 是否应该发现？
   → {是：Review 维度覆盖但漏掉了 / 否：Review 维度未覆盖}
   → 遗漏原因：{如"Review 未检查错误处理路径"}

⑤ 测试阶段：为什么用例没覆盖到？
   → {用例缺失：推导规则不包含该场景 / 用例存在但断言不够精确}
   → 遗漏原因：{如"隐性需求推导表缺少'并发操作'场景"}

应拦截阶段：{最早应该被发现的阶段}
改进建议：{具体的流程改进}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**规则沉淀（将回溯结果转化为持久改进）：**

根据回溯结果，将改进建议写入对应 Skill 的规则或项目的 `.codeman/rules/` 中：

| 应拦截阶段 | 沉淀位置 | 示例 |
|---------|---------|------|
| 需求阶段 | `feat-*.md` 模板补充边界场景 | 追加"并发操作"到边界场景模板 |
| 设计阶段 | `mod-*.md` 增加设计检查项 | "每个写操作必须设计幂等性方案" |
| 开发阶段 | `.codeman/rules/` 新增编码规则 | "所有数据库写操作必须有幂等性保护" |
| Review 阶段 | Review Skill 新增审查维度 | "检查所有写操作是否有并发保护" |
| 测试阶段 | 测试 Skill 隐性需求推导表补充 | 追加"并发操作"场景到推导表 |
| **隐性知识** | **领域规则文件** | **代码/测试/规范都无法表达的业务约束** |

**领域规则沉淀（额外检查）：**

如果 Bug 的根因涉及"代码无法表达的隐性业务知识"（如业务术语歧义、跨模块影响关系、实体间的隐性约束），除了上述规则沉淀外，还需写入领域规则：
- 仅涉及单模块 → 写入 `mod-*.md` 的「领域规则」章节
- 跨模块 → 写入 `.codeman/docs/DOMAIN-RULES.md`
- 写入前先检查文件是否存在：不存在则基于 `{CODEMAN_HOME}/templates/DOMAIN-RULES.md` 模板创建
- 文件存在时检查容量，接近 1500 字上限时先提示清理

**永久回归用例沉淀（每个产品 Bug 必做）：**

规则沉淀是改进**未来同类问题**的推导能力，但**这个具体 Bug 的场景**也必须变成永久测试用例，防止同一 Bug 复发：

- 把 Bug 的复现步骤和断言写成一条 ✅ 用例，追加到对应功能点的 `.codeman/docs/tests/test-*.md`
- 用例编号标记 `[regression-from-bug: B-xx]`（与 v1.1 的 `[regression-from-code]` 同体系）
- 该用例自动进入后续每次测试的 Smoke 层回归范围，无需额外配置

**输出格式（追加到测试报告末尾）：**
```
## 缺陷回溯学习
| Bug | 应拦截阶段 | 遗漏原因 | 改进措施 | 已沉淀到 |
|-----|---------|---------|---------|---------|
| B-01 | 设计 | 方案未考虑并发 | 写操作必须设计幂等性 | .codeman/rules/concurrency.md |
| B-02 | 测试 | 推导表缺少场景 | 隐性需求增加"多角色同时操作" | testing/SKILL.md 隐性推导表 |
```

### Step 8: Git Checkpoint + 合并

在**当前 fix 分支**上 commit（安全存档）：

```bash
git add {修复的文件}
git commit -m "wip-fix({模块名}): {修复描述}

- 根因：{根因描述}
- 修复方案：{修复方案描述}
- 影响范围：{影响的功能点}

Fixes: {失败用例ID}"
```

回归测试通过后，squash merge 回主分支（如 `git.branch_strategy` 开启）：

```bash
git checkout {主分支}
git merge --squash {fix 分支}
git commit -m "fix({模块名}): {修复描述}

- 根因：{根因描述}
- 修复方案：{修复方案描述}
- 回归测试：✅ 通过

Fixes: {失败用例ID}"
git branch -d {fix 分支}
```

> 如果 `git.branch_strategy` 为 false，直接在当前分支 commit（兼容旧行为）。

**⚠️ 绝对禁止执行 `git push`。推送操作必须由用户手动完成。**

**⚠️ 绝对禁止在 commit message 末尾追加 `Co-Authored-By: Claude ...` 行**（详见 orchestrator commit 规范）。即使 AI 助手默认模板包含此行，也必须移除。

## 不确定即停（遵守 orchestrator 全局硬规则）

遇到以下情况必须暂停，**严禁自行推测或执行未经确认的修复**：

- 根因不明确，存在多种可能
- 修复方案涉及架构变更
- 修复可能影响其他已通过的功能
- 问题可能是 PRD 本身的歧义导致
- 3 轮修复后仍未解决
- 修复方案可能引入新的安全风险

**暂停格式：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 需人工确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
关于"{问题}"，我分析到以下根因可能：

可能 1：{原因} → 修复方案：{方案} → 影响范围：{范围}
可能 2：{原因} → 修复方案：{方案} → 影响范围：{范围}

我的判断：{最可能的原因}，原因：{简要理由}

请问您确认哪个根因？或者有补充信息？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 批量修复策略

当有多个失败用例时：

1. 按根因分类，相同根因的用例一起修复
2. 按严重程度排序（Critical → High → Medium → Low）
3. 每次修复后运行相关测试，确认修复有效
4. 所有修复完成后，运行完整测试套件
