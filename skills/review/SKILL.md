# CodeMan Review Skill

## 宿主环境说明

**CODEMAN_HOME** 路径规则：Cursor 为 `~/.cursor/skills/.codeman`，Claude Code / OpenCode 为 `~/.claude/skills/.codeman`。

---

## 概述

以**独立审查者**视角审视产出物，基于 PRD 和技术文档进行审查，而非仅基于代码逻辑。发现问题后自行修复，修复方案不确定时暂停人工确认。

---

## 触发场景

| 触发时机 | 审查类型 | 调用方 |
|---------|---------|--------|
| 技术方案完成后 | 方案 Review | 技术方案 Skill |
| 编码完成后 | 代码 Review | 开发实现 Skill |
| 测试用例编写后 | 测试 Review | 测试验证 Skill |
| 用户手动触发 | 指定类型 | 用户 |

---

## 前置条件

执行前必须完成：

```
1. 读取 .codeman/docs/prd/INDEX.md + 相关 feat-*.md
   → 获取 PRD 功能点和验收标准（审查基准）

1.5 读取 .codeman/docs/DIRECTIVES.md
   → 对照"需求精粹卡"中的绝对约束，确保审查不遗漏核心要求

2. 根据审查类型，读取对应文档：
   - 方案 Review：.codeman/docs/design/mod-*.md + api/api-*.md
   - 代码 Review：相关源码文件 + design/mod-*.md
   - 测试 Review：.codeman/docs/tests/test-*.md + prd/feat-*.md
```

---

## 方案 Review（技术方案完成后触发）

### 审查维度

**① PRD 一致性（最高优先级）**
- 每个 PRD 功能点是否都有对应的技术实现？
- 验收标准是否在技术方案中有明确的实现路径？
- UI 设计要求是否在前端方案中有对应处理？
- 边界场景是否在技术方案中有处理逻辑？

**② 技术可行性**
- 选型是否成熟稳定？是否有已知的重大缺陷？
- 性能是否满足 PRD 中的隐含要求？
- 依赖的外部服务是否可靠？

**③ 架构合理性**
- 模块边界是否清晰？职责是否单一？
- 依赖关系是否合理？是否有循环依赖？
- 数据流是否清晰？是否有数据一致性风险？

**④ 安全性**
- 是否有明显的权限控制漏洞？
- 敏感数据是否有加密/脱敏处理？
- API 是否有输入校验？

**⑤ 可维护性**
- 方案是否过度复杂？是否有更简单的实现？
- 是否有明显的技术债务？

## 代码 Review（编码完成后触发）

### 审查维度

**① 与技术方案一致性（结构化逐项对照，不可笼统判断）**

**此维度必须通过 Grep 工具逐项比对产出结构化报告，不得只凭代码阅读做笼统判断（如"基本一致""实现正确"）。**

审查方法：
1. 读取 `mod-*.md` 和 `api-*.md`，提取所有可验证的设计条目（API 端点、数据模型字段、设计决策）
2. 对每个条目，用 Grep 在代码中搜索对应实现
3. 逐一比对，输出结构化实现率报告

**输出格式（必须包含在 Review 报告中）：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Review — 技术方案实现率
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
API 端点实现率：{N}/{M}
  ❌ 缺失：GET /api/users/:id（api-users.md §1）
  ⚠️ 偏差：POST /api/orders 响应缺少 estimated_delivery 字段（api-orders.md §3）

数据模型实现率：{N}/{M}
  ❌ 缺失：Order.discount_amount（mod-order.md §数据模型）

设计决策实现率：{N}/{M}
  ❌ 缺失：Redis 缓存未实现（mod-product.md §性能 要求 TTL 5 分钟）
  ✅ JWT 过期时间 24h — auth.ts:23 确认
  ✅ bcrypt rounds 10 — auth.ts:8 确认

总实现率：{N}/{M}（{%}）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**判定规则：**
- 总实现率 100% → ✅ 通过
- 有 ❌ 缺失 → 🔴 Critical，必须补充实现
- 有 ⚠️ 偏差 → 🟠 High，必须修正或向用户确认

> **注意：** 如果开发阶段已执行了 Step 5a 机械化比对且结果保存在对照表中，Review 可直接复用该结果，不必重复 Grep。但如果开发与 Review 之间有代码变更，必须重新执行比对。

**② 代码质量**
- 函数/方法是否职责单一？
- 是否有重复代码（DRY 原则）？
- 命名是否清晰、有意义？
- 复杂逻辑是否有必要的注释？
- 是否有魔法数字/魔法字符串？

**③ 规范符合度**
- 是否遵守 `.codeman/rules/` 中的规范？
- 文件结构是否符合项目约定？
- 命名规范是否一致？

**④ 潜在 Bug**
- 是否有未处理的 null/undefined？
- 是否有数组越界风险？
- 是否有并发竞争条件？
- 异步操作是否有正确的错误处理？
- 是否有内存泄漏风险（事件监听未清理等）？

**⑤ 安全漏洞**
- SQL 注入防护（是否使用参数化查询）？
- XSS 防护（输出是否转义）？
- CSRF 防护？
- 敏感信息是否硬编码在代码中？
- 权限检查是否在服务层而非仅在路由层？

**⑥ UI 设计一致性（有设计截图时）**
- 如 PRD 中有设计截图引用，读取截图并对比前端实现
- 布局结构是否与截图一致？
- 关键 UI 元素（按钮、表单、导航）是否与截图匹配？
- 不一致项标记为 Medium（视觉偏差）或 High（功能性 UI 缺失）

## 测试 Review（测试用例编写后触发）

### 前置准备

执行测试 Review 前，必须读取以下文件作为审查基准：
- 所有相关 `prd/feat-*.md` — 提取每条验收标准和每个边界场景的**原文**
- `tests/test-*.md` — 待审查的测试用例文档
- `tests/INDEX.md` — 需求覆盖率追踪矩阵

### 审查维度

**① 验收标准逐条对照（最高优先级）**

逐条读取 PRD 中 `## 功能点明细` 表格的每一行验收标准，逐一核查：

| 检查项 | 通过标准 | 不通过等级 |
|--------|---------|----------|
| 该验收标准是否有对应 E2E 测试用例（TC 编号可追溯）？ | 有 → ✅ | 无 → 🔴 Critical |
| 测试用例的断言是否**直接对应**该验收标准的可量化条件？ | 断言精确 → ✅ | 断言宽泛/缺失 → 🟠 High |
| 测试用例的预期结果是否与验收标准描述一致？ | 一致 → ✅ | 不一致 → 🔴 Critical |

输出逐条对照表：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
验收标准覆盖对照
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
F{ID}.1 验收标准：{原文}
  → 对应用例：TC-{ID}-01 ✅ / ⬜ 缺失
  → 断言精确度：✅ 精确 / ⚠️ 宽泛（{描述}）/ ❌ 缺失

F{ID}.2 验收标准：{原文}
  → 对应用例：...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
验收标准覆盖率：{N}/{M}（目标 100%）
```

**② 边界场景覆盖验证**

逐条读取 PRD 中 `## 边界场景` 的每一项，逐一核查：

| 检查项 | 通过标准 | 不通过等级 |
|--------|---------|----------|
| 该边界场景是否有对应测试用例（E2E 或集成测试）？ | 有 → ✅ | 无 → 🟠 High |
| 测试用例是否验证了边界场景的**预期系统行为**（而非只验证不崩溃）？ | 验证行为 → ✅ | 只验证不崩溃 → 🟡 Medium |

输出边界场景覆盖表：
```
边界场景覆盖：
  [边界场景原文] → TC-{ID}-{N} ✅ / ⬜ 缺失
  [边界场景原文] → ⬜ 缺失 🟠 High
边界场景覆盖率：{N}/{M}（目标 100%）
```

**③ UI 设计要求覆盖验证**
- PRD 中 `## UI 设计要求` 的每一条，是否都有 UI 视觉测试（Midscene.js 断言）覆盖？
- 响应式要求是否有多视口测试（375px / 768px / 1440px）？

**④ 测试质量**
- 测试用例是否独立（不依赖执行顺序）？
- 断言是否足够精确（避免 `expect(result).toBeTruthy()` 这类宽泛断言）？
- 是否有测试用例在测试实现细节而非用户可见行为？
- 测试数据是否覆盖了边界值（空值、最大值、特殊字符）？

## 问题严重等级

| 等级 | 定义 | 处理方式 |
|------|------|---------|
| 🔴 Critical | 功能无法正常工作 / 严重安全漏洞 / PRD 功能点未实现 / **验收标准无对应 E2E 用例** / **测试预期结果与验收标准不一致** | 必须修复，不得交付 |
| 🟠 High | 重要功能有缺陷 / 潜在安全风险 / 性能严重问题 / **边界场景无对应测试用例** / **E2E 断言宽泛未直接对应验收标准** | 必须修复，不得交付 |
| 🟡 Medium | 代码质量问题 / 规范不符合 / **边界场景测试只验证不崩溃未验证预期行为** / 测试覆盖不足 | 建议修复，可标注说明 |
| 🟢 Low | 代码风格 / 命名优化 / 注释完善 | 可选修复，记录即可 |

**Critical 和 High 问题必须自行修复后再交付。**

## 执行流程

### Step 1: 审查并记录问题

逐条审查，记录发现的所有问题：

```
[Review 进行中...]
审查维度：{当前维度}
发现问题：{问题描述}
等级：{Critical/High/Medium/Low}
位置：{文件路径:行号 或 文档位置}
修复建议：{具体修复方案}
```

### Step 2: 修复 Critical/High 问题

对于 Critical 和 High 等级的问题：

1. 分析修复方案
2. Critical 问题 → **必须向用户确认修复方案后才能修复**（涉及安全/架构/核心逻辑）
3. High 问题 → 如修复方案明确且影响 ≤ 1 个文件 → 直接修复
4. High 问题 → 如修复方案不确定或影响 > 1 个文件 → 向用户确认
5. 修复后再次 Review 验证

**Critical 问题修复前必须确认：**
```
⚠️ 需人工确认
Review 发现 Critical 问题：

[{严重度}] {问题描述}
位置：{文件路径}:{行号}
修复方案：{具体修复方式}
影响范围：{受影响的部分}

确认修复？（直接回复即可）
```

**修复方案不确定的判定：**
- 修复涉及架构变更
- 修复可能影响其他模块
- 存在多种修复方案，各有取舍
- 修复需要 PRD 中未明确的业务决策

### Step 3: 生成 Review 报告

**文件路径：** `.codeman/docs/reviews/review-{YYYYMMDD}-{类型}-{模块}.md`

```markdown
# Review 报告：{类型} - {模块/范围}

> 时间：{YYYY-MM-DDTHH:MM:SS}
> 触发方：{触发的 Skill}
> 审查范围：{文件/模块列表}

  → 用户确认后，调用自进化 Skill 创建 Rule
```
