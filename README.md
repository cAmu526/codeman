# CodeMan v1.2

> 全流程开发工作流框架，支持 **Cursor**、**Claude Code**、**OpenCode** 和 **Trae** 四种 AI IDE。用自然语言驱动 AI 完成从需求分析到部署的全链路开发。每个关键阶段含**多次交互确认**，确保需求完整、方案合理、测试充分。

---

## 30 秒理解 CodeMan

CodeMan 是一套 AI IDE 插件（支持 Cursor / Claude Code / OpenCode / Trae）。安装后，你在对话框里用自然语言说需求，AI 就会按 **需求分析 → 技术方案 → 测试用例 → 编码 → 测试执行 → 部署** 的流程帮你开发。

**你只需要做两件事：** ① 描述需求 ② 在关键节点确认 AI 的方案

下面是一个完整的使用示例：

```
你：CodeMan 初始化                          ← 第一次用，初始化项目
AI：检测到新项目，技术栈 React + Node.js...
    初始化完成，进入需求分析。

你：我要做一个待办事项应用，支持增删改查和分类   ← 描述你的需求
AI：【需求分析】八维拓展后识别到 6 个功能点...
    ━━━━━ 确认①：功能点分类 ━━━━━
    🔴 核心：增删改查、分类管理
    🟡 增强：搜索、排序
    🟢 可选：标签、导出
    确认？

你：核心和增强都要，可选的先不做              ← 你做决策
AI：好的，生成 PRD 中...
    ━━━━━ 确认②：PRD 内容审阅 ━━━━━
    （展示每个功能点的详细 PRD）
    确认？

你：确认                                     ← 确认 PRD
AI：生成需求精粹卡...
    ━━━━━ 确认③：精粹卡确认 ━━━━━
    确认？

你：确认                                     ← 需求分析完成
AI：需求分析完成！共 4 个功能点。
    下一步：技术方案设计
    是否立即开始？

你：开始                                     ← AI 自动进入下一阶段
AI：【技术方案】模块划分：3 个模块...
    ━━━━━ 确认①：模块划分 ━━━━━
    ...（后续类似，每个阶段都有确认点）

你：确认 / 调整 / 跳过                       ← 每个节点你都可以决策
```

**关键理解：**
- 你不需要记任何命令 —— 每个阶段完成后 AI 会告诉你下一步是什么
- 关掉 IDE 也没关系 —— 下次打开说 `CodeMan 继续` 就能接着干
- 不确定的地方 AI 会停下来问你，确定的地方 AI 直接做

## 快速开始

### 1. 安装（一次性，约 10 秒）

```bash
bash /path/to/codeman/install.sh
```

安装脚本自动检测已安装的 IDE 并配置。安装过程中会询问是否安装推荐的第三方 Skills（头脑风暴、线框图、UI 组件规则库），也支持输入自定义 GitHub URL 安装其他 Skills。**Cursor 用户安装后需重启。**

<details>
<summary>安装位置详情</summary>

| 环境 | 安装位置 | Bootstrap 方式 |
|------|---------|--------------|
| Cursor | `~/.cursor/skills/.codeman/` | `~/.cursor/rules/codeman-bootstrap.mdc`（alwaysApply） |
| Claude Code | `~/.claude/skills/.codeman/` | `~/.claude/CLAUDE.md`（自动注入片段） |
| OpenCode | `~/.claude/skills/.codeman/`（兼容层） | `~/.config/opencode/AGENTS.md`（自动注入片段） |
| Trae | `~/.trae/skills/.codeman/` | `~/.trae/rules/codeman-bootstrap.md`（alwaysApply） |

多个 IDE 都安装了会询问你安装到哪个（或全部）。

</details>

### 2. 初始化项目

用 AI IDE 打开你的项目，说：

```
CodeMan 初始化
```

AI 会交互式询问项目信息（名称、技术栈等），然后在项目根目录生成 `.codeman/` 文档体系。

| 项目情况 | AI 的处理方式 |
|---------|-------------|
| **全新项目**（无业务代码） | 直接进入需求分析 |
| **已有代码库 — 完整接入** | 扫描全部代码，反向提炼文档（适合 ≤20 模块，1-3 小时） |
| **已有代码库 — 骨架接入** | 只读目录结构，5-10 分钟出骨架（推荐大项目） |

### 3. 开始开发

```
CodeMan 开始开发
```

AI 会自动走完整流程。每个关键节点暂停等你确认，确认后自动进入下一阶段。

## 完整工作流详解

说完命令后，AI 自动识别场景并按以下流程执行。每个阶段完成后会**提示下一步并询问是否继续**，用户确认后自动加载下一阶段（所有 IDE 通用，无需记忆命令）：

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 需求分析 │ → │ 技术方案 │ → │ 测试用例 │ → │ 开发实现 │ → │ 测试执行 │ → │ 部署清单 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
                    ↓           编码前锁定          ↓              ↓
               Review 门禁     验收基准        Review 门禁    失败 → 修复闭环
                                                              (最多 3 轮自动修复)
```

> **ATDD 模式：** 测试用例在编码前生成并锁定，基于 PRD + 技术方案 + 领域规则三者推导，不受代码实现影响。编码的目标是通过这些用例。编码完成后再编写自动化脚本并执行。

### 1. 需求分析

AI 通过**八维**头脑风暴帮你拓展需求，然后收敛为结构化的 PRD 功能点碎片文件。含**三次交互确认**。支持在任意步骤前后挂载外部 Skill（如 Superpowers brainstorming），通过 `config.yaml` 的 `external_skills` 配置驱动，详见「外部 Skill 注册」章节。

- **输入**：你的需求描述（一句话或一段话都行）
- **代码现状扫描（版本迭代时）**：分析需求前先深入阅读涉及的现有代码（数据模型、API、业务逻辑、关联模块），输出代码现状摘要请你确认，确保 AI 理解了系统当前的真实行为再写需求
- **过程**：代码现状确认 → 八维拓展 → 红/黄/绿分类 → **【确认①：功能点分类】** → 收集 UI 设计截图 → 生成 PRD → **【确认②：PRD 内容审阅 + 完整性校验】** → 生成需求精粹卡 → **【确认③：精粹卡确认】**
- **PRD 质量（行业标准）**：每个功能点必含用户故事、业务功能流程图、ASCII 界面原型、Given-When-Then 验收标准、数据字典、非功能需求、假设与约束
- **完整性自检**：PRD 生成后自动输出 10 项检查评分（用户故事、流程图覆盖、验收标准可量化性、边界场景、数据字典等）
- **领域规则提取**：从 PRD 中自动识别业务术语、跨模块不变量、实体约束等隐性知识，写入领域规则文件；版本迭代时自动检查新需求与已有领域规则的冲突
- **产出**：`.codeman/docs/prd/feat-*.md`（每个功能点一个文件）+ 需求精粹卡（`DIRECTIVES.md`，≤500 tokens）+ 领域规则更新（`DOMAIN-RULES.md`，如有）

### 2. 技术方案

基于确认的 PRD，AI 做模块拆分和技术方案设计。含**三次交互确认**。

- **输入**：PRD 功能点 + 项目配置 + 需求精粹卡
- **代码深度阅读（版本迭代时）**：设计前先逐文件阅读涉及模块的实际代码实现（以代码为准，不以旧文档为准），理解真实的数据结构、调用链路和约束。**骨架模块自动补全**：骨架接入后首次涉及的模块（无 `mod-*.md`），设计阶段自动读取代码并生成技术文档，确保设计方案有充分的现状基线
- **过程**：代码阅读 → 模块边界划分 → **【确认①：模块划分】** → 每模块技术选型与数据模型 → API 接口设计 → **【确认②：API 契约审阅（逐接口）】** → 自动 Review 门禁 → **【确认③：方案细节审阅（数据模型/核心逻辑/决策）】**
- **精粹卡对照**：设计前加载需求精粹卡，确保方案覆盖所有核心功能和绝对约束
- **领域规则覆盖检查**：Review 时检查领域规则中的不变量和实体约束是否在技术方案中有对应保障措施，未覆盖的标记为风险项
- **版本迭代影响面扫描**：新需求/修改时，Grep 扫描代码中所有引用点（数据模型、API、枚举、组件等），展示直接引用 + 间接影响 + 前端页面影响，用户确认哪些纳入本次迭代后再继续设计
- **产出**：`.codeman/docs/design/mod-*.md` + `.codeman/docs/api/api-*.md` + 模块级领域规则（`mod-*.md`「领域规则」章节）

### 3. 开发实现

按技术方案编码，所有任务先拆分为 2-5 分钟的原子任务再执行。支持 subagent 并行（Claude Code）或串行执行（Cursor/Trae）。

- **外部代码变更检测**：启动时自动检测是否有同事的新提交尚未同步文档，有则自动触发增量补全后再继续；也可手动 `CodeMan 同步` 处理非 git 方式获得的代码
- **对照已锁定的测试用例编码**：编码前已有经确认的测试用例，编码目标是通过这些用例，不是自由发挥
- **上下文回顾**：每个模块开始前重读 DIRECTIVES.md + DOMAIN-RULES.md + 测试用例，**必须输出结构化回顾摘要**（精粹卡约束列表 / 本模块领域规则 / 对应测试用例及预期 / 遗留问题 / 偏离风险检查），不得敷衍"已回顾无偏离"
- **第三方 API 文档查阅**：检测到模块依赖第三方 API/SDK 时，自动从 `config.yaml` 中配置的文档 URL 抓取最新用法；未配置时提醒你补充链接，避免使用过时 API 导致兼容性问题
- **编码前**：从技术方案机械化提取细粒度 Checklist，然后拆分为 **2-5 分钟的原子任务**（每个任务含具体文件路径、代码内容、验证步骤），你确认任务计划后才开始
- **执行策略**：任务数 ≥ 3 且在 Claude Code 中 → **subagent 并行执行**（每个任务独立 subagent，互不干扰）；否则串行执行
- **两阶段 Review**：每个任务完成后先查**规格符合度**（有没有多做/少做），再查**代码质量**
- **编码中**：禁止凭推测补充 PRD 未提及的逻辑，遇到歧义暂停问你。如需添加 PRD 未提及的功能必须标注 `[UNCERTAIN: PRD 未提及]`
- **变更影响面扫描**：修改现有代码时，自动识别 10 类变更类型（枚举/API 契约/组件签名/Schema/权限/路由/配置/业务规则/状态机/错误码），全局搜索所有引用点，逐一同步修改，修改后执行一致性验证（类型检查 + exhaustive check + 单元测试）
- **分支策略**：开发自动在 feature 分支上进行，每个模块完成后 checkpoint commit（安全存档），测试全部通过后 squash merge 回主分支，保证主分支每个 commit 都是经过测试的可靠代码
- **编码后**：**机械化 Grep 比对**（技术方案 API 列表 vs 代码路由定义、数据模型 vs 类型定义、设计决策 vs 代码实现，产出结构化 diff 表）→ 需求实现对照表 + 领域规则合规检查 + 跨功能一致性自检 → **浏览器冒烟验证**（有 UI 的模块用 Playwright 走核心路径截图）→ **自动 Review 门禁** → Git commit → **领域规则审查**（已覆盖的规则提示毕业、反复踩坑的提示升级为编码规范、新发现的隐性知识提示写入）
- **复杂任务**：满足复杂条件（3+ 模块联动、嵌套 ≥ 3 层等）时强制拆分子步骤，逐步确认后执行
- **同步产出**：单元测试（覆盖率 ≥ 80%）

### 4. 测试验证

**遵循 ATDD（验收测试驱动开发）模式：测试用例在编码前生成，测试脚本在编码后编写。** 测试基准来源于 PRD（不是代码），能自动化的一律自动化，有 UI 的功能一律用 E2E 浏览器测试。

- **阶段一（编码前）：用例生成** — 在技术方案完成后、编码开始前执行。从**三类来源**推导测试用例 — ① PRD 验收标准和边界场景 ② 技术方案设计决策（缓存策略、错误码、中间件行为等）③ 领域规则（从不变量和实体约束自动派生验证测试）。**⑤ 代码回归基线**（无 PRD 的受影响老功能，从代码实际行为提取回归用例，标记 `[regression-from-code]`，锁定改之前的行为）。隐性需求自动推导（8 类场景）。用户确认前自动执行**五项质量校验**：完整性校验、浅测试检测、覆盖率前置校验、技术方案覆盖校验、代码分支覆盖分析。**验收标准理解确认**：逐条输出 AI 对每条验收标准的理解和验证方式，用户确认后锁定用例
- **用例锁定**：用户确认用例后锁定为验收基准，编码的目标就是通过这些用例
- **阶段二（编码后）：脚本编写 + 合规审计 + 逐层执行**：基于已锁定的用例编写自动化脚本（此时代码存在，可获取选择器/API 路径）。**代码合规审计**：Grep 扫描代码中所有路由/接口，与文档对比，发现文档未定义的代码 → 标记违规暂停确认（保留需先补文档，不需要则删除）。然后按测试分级（轻量/标准/完整）逐层执行：代码审查预检 → L1 → L2a → L2b → L3 → L3 网络异常 → L4 → 兼容性 → L5

| 层级 | 测试类型 | 自动化方式 |
|------|---------|-----------|
| 代码审查 | 预检 | **全栈强制覆盖**（7 类文件缺一不可）+ 硬编码检查 + 前后端参数一致性 + **跨功能一致性检查**（枚举/API/Schema 等共享定义变更时自动检测所有引用方是否同步） |
| L1 | 单元测试 | 编码时已同步生成，此阶段自动验证并补充 |
| L2a | DB 直查 | 直连数据库验证表结构、迁移、初始数据（无需 API 认证） |
| L2b | API 集成 | API 测试脚本 + **响应数据格式全面校验**（字段类型、枚举值、嵌套结构、数组格式，基于 API/技术文档而非代码） |
| L3 | E2E 端到端 | **强制 UI 探索 → 脚本冒烟 → 正式执行**三步流程 + **Playwright 浏览器自动化** + **对照 PRD 原型图校验布局结构和交互逻辑**（含 8 类前端场景 + 4 类网络异常场景 + 3 类组件标准化验证模板：下拉框/列表页/表单完整流程） |
| L4 | UI 视觉测试 | Midscene.js 自然语言断言 + Playwright 截图对比 + **逐项对照设计稿校验样式** |
| 兼容性 | 多浏览器 + 多设备 | L3/L4 在 Chromium/Firefox/WebKit + iPhone/Android/iPad 上复跑（默认 Chromium，执行前确认范围，支持 BrowserStack 真机增强） |
| L5 | 非功能测试 | **性能**（Lighthouse 逐页面 + k6 三场景压测 + 内存泄漏检测）+ **安全**（OWASP 自动化检测 + 手动建议清单）+ **可访问性**（axe-core WCAG AA 逐页面扫描），各子项独立开关，执行前逐项确认 |

**防虚假通过机制：** 每个用例必须有实际执行证据（脚本输出 / API 响应体 / 截图）才能标 ✅，严禁凭代码审查推断通过。标记前强制自检 7 项 checklist（含描述质量校验和 UI 实际页面验证），续测模式自动审计降级无证据的 ✅。**描述质量最低标准**：所有报告描述必须包含「定位 + 实际值 + 期望值 + 来源」四要素，附 10 类笼统 vs 具体反面示例对照表，禁止"样式不对""接口异常"等模糊描述。**API ≠ 页面规则**：涉及 UI 的功能即使 API 测试通过，仍必须通过 Playwright 实际打开页面验证渲染结果，附 6 类"API 正确但页面出错"的典型踩坑场景。

**脚本可信性保障：** E2E 脚本编写前强制执行 UI 探索（先用 Playwright 打开页面截图观察实际 UI，严禁直接基于 PRD 假设写脚本）；**断言模式黑名单**（禁止 HTML 全文匹配、恒为真断言、硬编码通过、只截图不断言等 7 类反模式）+ **白名单**（按 11 类场景规定必须使用的断言方式）；**脚本 lint 门控**（Grep 正则扫描禁止模式，不依赖 AI 判断，扫描不通过不允许执行）；冒烟验证选择器有效性；失败归因流程（区分脚本 bug vs 产品 bug）；**通过用例抽样交叉验证**（对 ✅ 用例随机抽查 3-5 个独立截图比对，不一致则降级）。

**测试校验来源原则：** API 数据格式的预期来自技术文档/API 文档（严禁从代码反推）；UI 布局和交互的预期来自 PRD ASCII 原型图、UI 设计要求、设计截图/Figma（多来源冲突时暂停确认）。

**回归测试策略：** 三级回归（Smoke 3-5 分钟 / Regression 10-20 分钟 / Full 30-60 分钟），每次修改后提醒触发。影响分析**三层机制**：路径规则匹配 + AI 补充判断 + **跨功能依赖追踪**（Grep 共享定义的引用方，自动追加引用方的测试用例）。可选 CI/CD 集成（GitHub Actions / GitLab CI）。

测试完成后额外执行精粹卡验收标准交叉校验，确保每条关键验收标准都有 E2E 覆盖。

**总文档自动同步：** 测试全部通过后，AI 自动将碎片文档同步到 `PRD-FULL.md`（产品需求全貌）和 `DESIGN-FULL.md`（技术设计全貌，含架构/模块/API/数据库表）。你看这两个文件就能随时了解项目完整状态，不用翻碎片文件。

**版本迭代文档（可选）：** 开始迭代时 AI 会询问是否需要版本文档，回复版本名（如"v1.2.0"）后，测试通过时统一生成到 `.codeman/docs/iterations/{版本名}/`（PRD + 技术方案 + 测试报告 + 迭代摘要）。

**碎片归档：** 部署完成后 AI 扫描过时的碎片文件并建议归档到 `archive/`，保持工作目录干净。

**项目级稳定性验证：** L1-L5 全部通过后、Git 合并前，额外执行三层项目级验证：V1 构建验证（编译/类型检查零错误）→ V2 已有测试套件全量执行（含之前迭代的测试 + 代码回归用例，确认未破坏老功能）→ V3 项目级 Smoke 测试（启动项目 → 核心路径验证 → 无崩溃）。三层全过才允许 squash merge。

测试全部通过 → 进入部署清单；有失败 → 自动进入修复闭环（最多 3 轮）。

### 5. 修复闭环

测试失败时自动触发，也可手动用 `CodeMan 修复：[描述]` 触发。

- 修复前自动执行上下文回顾，对照需求精粹卡确保修复方向不偏离核心约束
- 根因分析 → 影响评估 → 修复 → 回归验证（含 P0 功能点扩大回归）→ **文档同步**
- 最多自动修复 3 轮，修不好会停下来让你介入

**文档同步规则**（根据根因类型自动处理）：

| 根因类型 | 同步哪些文档 | 说明 |
|---------|------------|------|
| 实现 Bug（代码逻辑错误） | 测试报告 | PRD 和技术方案不用动 |
| 设计缺陷（技术方案有问题） | `.codeman/docs/design/mod-*.md` | 更新技术方案，记录变更原因 |
| PRD 歧义（需求不明确） | `.codeman/docs/prd/feat-*.md` + DIRECTIVES.md | 明确业务规则和约束 |

不会只改代码不更新文档。

**缺陷回溯学习**：每个产品 Bug 修复后强制回溯分析（需求→设计→开发→Review→测试，哪个环节应该拦住但没拦住），将改进建议沉淀为规则（如新增编码规则、Review 审查维度、测试推导场景）；涉及隐性业务知识的 Bug 额外写入领域规则文件，防止同类问题重复出现。

### 6. 部署清单

所有测试通过后自动生成，包含：环境变量、数据库迁移、依赖变更、部署步骤、API 变更说明、风险与回滚方案。

### 7. 自进化（自动触发）

Review 中发现同类问题出现 ≥ 2 次时，AI 会自动草拟新的编码规范（Rule）或工作流技能（Skill），经你确认后写入项目，后续开发自动遵守。

## 质量保障机制

以下机制在开发过程中逐步引入，解决 AI 辅助开发中的常见痛点：

### UI 设计资产引用系统

**解决问题：** 前端实现与 UI 原型图/设计稿不一致。

PRD 模板中新增"设计资产引用"表格，支持在每个功能点中引用设计截图文件路径。引用会贯穿整个开发链路：

| 阶段 | 如何使用截图 |
|------|------------|
| 需求分析 | 收集设计截图，写入 PRD 的设计资产引用表 |
| 开发实现 | 编码前强制读取截图文件（利用 AI 多模态能力），理解视觉要求后再编码 |
| 测试验证 | L4 UI 视觉测试时，读取设计截图与实际页面截图逐项对比 |
| Review | 代码审查新增第⑥维度"UI 设计一致性"，对比截图与前端实现 |

截图文件建议存放在 `.codeman/assets/design/` 目录。如无设计稿，标注"待设计稿确认"即可，不阻塞流程。

### 需求精粹卡

**解决问题：** 中间文档过多（PRD → 设计 → 代码 → 测试），跨阶段传递时需求细节丢失。

需求确认后自动生成一张 ≤500 tokens 的"需求精粹卡"，存入 `DIRECTIVES.md`，内容包含：

- **产品目标**（一句话）
- **核心功能清单**（每个 P0/P1 功能点一行）
- **绝对约束**（不可违反的硬约束）
- **关键验收标准**（最关键的 3-5 条可量化条件）
- **设计资产摘要**（截图路径，如有）

精粹卡随 `DIRECTIVES.md` 被每个 Skill 自动读取，无需额外文件管理。版本迭代时同步更新。

### 周期性上下文回顾协议

**解决问题：** 长对话中 AI 遗忘早期的关键约束和指令。

在两个时机强制执行上下文回顾：

1. **每个模块开发完成后**（多模块迭代循环的第 0 步）
2. **每次阶段转换时**（Orchestrator 调度下一个 Skill 之前）

回顾动作：重新读取 `DIRECTIVES.md`（含需求精粹卡）+ `DOMAIN-RULES.md`（领域规则），输出一行回顾摘要。如发现当前实现方向与核心约束或领域规则存在偏离，立即暂停并报告。

### 测试可信性保障

**解决问题：** 测试脚本本身有 bug 导致结果不可信；测试报告描述笼统无法定位问题；只跑 API 不验页面漏过前端缺陷。

通过四项机制形成闭环，从源头保障测试结果可信：

| 机制 | 解决什么问题 | 核心做法 |
|------|------------|---------|
| **UI 探索前置** | 脚本基于 PRD 假设编写，选择器/操作路径可能与实际 UI 不符 | 写脚本前必须先 Playwright 打开页面截图，观察实际 UI 结构后再编写 |
| **脚本冒烟验证** | 脚本写完直接批量跑，一旦有基础问题则大面积失败 | 正式执行前先验证选择器有效性、页面可加载、前置操作可达 |
| **失败归因流程** | 脚本 bug 被误报为产品 bug，浪费开发时间 | 失败时按决策树排查（选择器/断言/运行错误），区分脚本 bug 与产品 bug |
| **描述质量标准** | 报告写"样式不对""接口异常"无法定位问题 | 所有描述必须包含「定位 + 实际值 + 期望值 + 来源」四要素，附 10 类反面示例 |

### 机械化比对

**解决问题：** AI 自己写代码、自己检查，盲区一样 — 漏掉的编码时漏，自检时也漏。

用结构化 Grep diff 替代 AI 主观判断，在三个环节强制执行：

| 环节 | 比对内容 | 做法 |
|------|---------|------|
| **开发后置检查** | 技术方案 vs 实际代码 | Grep 代码路由定义、类型定义、关键实现，与技术方案逐项 diff |
| **测试覆盖校验** | 实际 API vs 测试覆盖 | Grep 代码中所有路由定义 → 与测试文件中的请求 URL 交叉比对 |
| **Review 审查** | 技术方案实现率 | 逐项 Grep 验证，输出结构化实现率报告（而非笼统"基本一致"） |

### 第三方 API 文档查阅

**解决问题：** AI 基于训练数据中的旧版 API 知识编码，导致使用了已废弃的接口、变更的参数签名或过时的认证方式，运行时报错。

通过"声明 + 抓取 + 缓存"三步机制，确保编码时参考最新的官方文档：

| 步骤 | 做什么 | 用户成本 |
|------|--------|---------|
| **声明** | 在 `config.yaml` 的 `api_references` 中填写第三方 API 名称和文档 URL | 每个 API 一次性填写 |
| **抓取** | 编码前自动 fetch 官方文档，提取当前模块用到的接口签名、参数、返回格式 | 零（自动） |
| **缓存** | 将摘要存入 `.codeman/docs/api-refs/`，同会话不重复抓取，超 7 天自动刷新 | 零（自动） |

**未配置时的提醒机制：** 如果检测到模块使用了第三方 API 但 `config.yaml` 中没有对应的文档 URL，会暂停提醒用户补充。用户可以提供链接（自动写入配置）或选择跳过（标注风险继续编码）。

### 增量文档补全

**解决问题：** 团队协作中同事不使用 CodeMan，推送的代码没有对应文档，导致文档与代码不同步。

通过"自动检测 + 手动同步"两个入口，确保文档始终跟上代码变化：

| 入口 | 触发方式 | 适用场景 |
|------|---------|---------|
| **自动检测** | 执行工作类命令（继续/开始开发/新需求）时，自动检查是否有他人的新提交 | 同事通过 git 推送的代码 |
| **手动同步** | `CodeMan 同步` | 同事直接给的代码文件、非 git 管理的变更 |

补全流程：扫描代码变更 → 对比已有文档找出缺失 → 用户确认范围 → 仅对缺失部分反向生成（技术方案/API 文档/PRD 骨架）→ 更新索引。只补缺失的，不动已有的。

### 领域规则积累

**解决问题：** 代码和测试无法表达的业务隐性知识（术语定义、实体约束、跨模块影响）没有记录，AI 不知道就会踩坑写出业务 Bug。

通过"两层存储 + 四入口写入 + 四 Skill 消费 + 毕业/升级闭环"机制，让项目越开发越好用：

| 机制 | 说明 |
|------|------|
| **两层存储** | 全局规则在 `DOMAIN-RULES.md`（≤1500 字，始终加载），模块级规则在 `mod-*.md`「领域规则」章节（按需加载） |
| **四入口写入** | 需求分析时从 PRD 提取、技术设计时识别、开发踩坑时补充、修复 Bug 时沉淀 |
| **四 Skill 消费** | 需求分析检查冲突、技术设计检查覆盖、开发编码前后对照、测试从不变量派生用例 |
| **毕业机制** | 模块完成时审查：已被代码/测试覆盖的规则提示移除，释放容量 |
| **升级机制** | 同类踩坑出现 2+ 次，提示升级为 `.codeman/rules/` 编码规范，从源头预防 |

### 缺陷回溯学习

**解决问题：** 同类 Bug 反复出现，流程没有学习能力。

每个产品 Bug 修复后强制回溯到源头阶段（需求→设计→开发→Review→测试），分析哪个环节应该拦住但没拦住，将改进建议自动沉淀为对应 Skill 的规则或项目编码规范，堵住同类漏洞。

## 项目中会生成什么

### 你需要看的文档（3 个）

| 文档 | 内容 | 何时更新 |
|------|------|---------|
| `PRD-FULL.md` | 产品需求全貌 — 所有功能点的累积完整版 | 每次测试通过后自动同步 |
| `DESIGN-FULL.md` | 技术设计全貌 — 架构 + 模块 + API + 数据库表 | 每次测试通过后自动同步 |
| `PROJECT-OVERVIEW.md` | 项目概览 — 简介、技术栈、进度、上手指南 | 每个阶段完成后自动更新 |

其余文件（`prd/feat-*.md`、`design/mod-*.md` 等）是 **AI 的工作文件**，AI 在开发过程中用来写需求、技术方案和测试用例，你不需要主动去看。

### 完整目录结构

初始化后，项目根目录会多出 `.codeman/` 目录（以及 IDE 适配目录）：

```
.codeman/
├── config.yaml                 ← 项目配置（技术栈、测试框架、Token 阈值、外部 Skills 挂载等）
├── assets/
│   └── design/                 ← UI 设计截图存放目录（如有）
├── docs/
│   ├── PROJECT-OVERVIEW.md     ← 项目概览（内容聚合文档，内联核心摘要，从头读到尾即可了解全貌）
│   ├── STATUS.md               ← 当前开发进度（AI 跨会话的记忆）
│   ├── DIRECTIVES.md           ← 关键决策、约束 + 需求精粹卡 + 上下文回顾协议
```

## 外部 Skill 注册

CodeMan 支持将第三方 Skill（如 [Superpowers](https://github.com/obra/superpowers)）作为子工具挂载到工作流中。外部 Skill 保持独立安装和更新，CodeMan 在执行到指定阶段时自动调用。

### 安装时自动推荐

运行 `install.sh` 时，脚本会推荐三个第三方 Skills 并询问是否安装：

| Skill | 用途 | 来源 |
|-------|------|------|
| superpowers | 协作式头脑风暴 + 浏览器可视化 mockup | [obra/superpowers](https://github.com/obra/superpowers) |
| wireframe | HTML 线框图生成（5 套 UX 方案） | [Magdoub/claude-wireframe-skill](https://github.com/Magdoub/claude-wireframe-skill) |
| ui-design-brain | 60+ UI 组件最佳实践规则库 | [carmahhawwari/ui-design-brain](https://github.com/carmahhawwari/ui-design-brain) |

你可以全部安装、选择安装、跳过，或输入自定义 GitHub URL 安装其他 Skills（脚本会引导你配置挂载点）。

安装后的 Skills 会写入注册表（`external-skills-registry.json`）。执行 `CodeMan 初始化` 时，orchestrator 会读取注册表并询问是否为当前项目启用。

升级时（`update.sh`），已安装的第三方 Skills 会自动 `git pull` 更新。

### 通过自然语言配置

直接用对话注册，不需要手动编辑配置文件：

```
你：把 Superpowers 的 brainstorming 挂到需求分析之前
AI：好的，我找到了文件路径 ~/.claude/skills/superpowers/skills/brainstorming/SKILL.md
    它的产出会写到哪个目录？
你：docs/superpowers/specs/
AI：━━━ 外部 Skill 注册 — 请确认 ━━━
    名称：brainstorming
    路径：~/.claude/skills/superpowers/skills/brainstorming/SKILL.md ✅
    挂载点：requirements.before_step1（需求分析 Step 1 之前）
    产出路径：docs/superpowers/specs/
    确认写入 config.yaml？
你：确认
AI：已注册。下次执行需求分析时会先调用 brainstorming。
```

其他管理命令：
- **查看**："查看已注册 Skill" — 列出当前项目所有已挂载的外部 Skill
- **修改**："把 brainstorming 改到技术方案之前" — 调整挂载点
- **移除**："移除 brainstorming" — 取消挂载

### 挂载点格式

`{阶段}.{before|after}_step{N}`，阶段名对应内部 Skill：

| 阶段名 | 对应 Skill |
|--------|-----------|
| requirements | 需求分析 |
| design | 技术方案 |
| development | 开发实现 |
| testing | 测试验证 |
| review | Review |
| fix | 修复闭环 |
| deploy | 部署清单 |

示例：`requirements.before_step1`（需求分析第 1 步之前）、`design.after_step2`（技术方案第 2 步之后）

### 手动配置

也可以直接编辑 `.codeman/config.yaml`：

```yaml
external_skills:
  - name: "brainstorming"
    path: "~/.claude/skills/superpowers/skills/brainstorming/SKILL.md"
    hook: "requirements.before_step1"
    order: 1
    output: "docs/superpowers/specs/"
    description: "Superpowers 协作式需求头脑风暴"
    confirm: true
```

| 字段 | 说明 |
|------|------|
| name | 显示名称 |
| path | SKILL.md 文件的绝对路径 |
| hook | 挂载点（格式见上） |
| order | 同一挂载点有多个 Skill 时的执行顺序（升序） |
| output | 产出目录（相对于项目根目录），后续阶段可读取 |
| description | 一句话说明 |
| confirm | `true`（默认）每次执行前询问是否调用；`false` 静默执行 |

## 升级与卸载

### 升级框架

在 CodeMan **源码目录**下执行（不是安装目录）：

```bash
bash /path/to/codeman/update.sh
```

升级只更新框架文件，不会覆盖各项目的 `.codeman/docs/` 文档和项目级规范。同时自动更新已安装的第三方 Skills（`git pull`）。脚本会自动检测已安装的环境并同时升级。Cursor 用户升级后需重启 Cursor。

### 全局卸载

彻底移除 CodeMan 全局框架（自动检测并清理 Cursor / Claude Code / OpenCode / Trae 所有已安装环境）：

```bash
bash /path/to/codeman/uninstall.sh
```

卸载后重启 Cursor 生效。各业务项目中的 `.codeman/` 目录不受影响。

### 项目反初始化

只清理当前项目的 CodeMan 配置（规范、模板），保留 `.codeman/docs/` 文档资产：

```bash
# 在项目根目录执行（清理所有 IDE 的项目级配置：Cursor / Claude Code / OpenCode / Trae）
bash /path/to/codeman/uninit.sh
```

## 常见问题

### 使用问题

**Q：我需要记住这些命令吗？**
A：不需要。日常只需记住两个：`CodeMan 初始化`（第一次）和 `CodeMan 继续`（后续）。每个阶段完成后 AI 都会提示"下一步：XXX，是否立即开始？"，你回复"是"/"继续"/"开始"即可。

**Q：一个阶段完成后，AI 没有提示下一步怎么办？**
A：说 `CodeMan 继续`，AI 会读取进度文件，自动从上次停的地方接着走。

**Q：确认的时候我可以说什么？**
A：AI 每次暂停确认时，你可以：
- `确认` / `是` / `继续` — 同意方案，进入下一步
- 提修改意见 — 如"把搜索功能改成可选的""模块 A 和 B 合并"
- `跳过` — 跳过当前阶段
- `切换到 [其他场景]` — 换一种执行路径

**Q：说了命令但 AI 没反应？**
A：确认已重启 Cursor。检查 `~/.cursor/rules/codeman-bootstrap.mdc` 文件是否存在。

**Q：换了一个 Cursor 会话，AI 还记得之前的进度吗？**
A：记得。进度保存在项目的 `.codeman/docs/STATUS.md` 中，新会话说 `CodeMan 继续` 即可恢复。

**Q：可以跳过某个阶段吗？**
A：可以。确认时告诉 AI 跳过即可。轻量修复场景（`CodeMan 修复`）会自动跳过不必要的阶段。

**Q：我用 Cursor Spec / 其他工具开发了一半，能让 CodeMan 接手吗？**
A：可以。说 `CodeMan 接手`，告诉 AI 你的 PRD 和技术方案文档路径，以及当前到哪个阶段了。AI 会读取你的文档和代码，从你指定的阶段继续（测试 / Review / 修 Bug 等）。迭代完成后自动沉淀为 CodeMan 原生文档，后续迭代走原生流程。

**Q：`CodeMan 接手` 和 `CodeMan 初始化` 有什么区别？**
A：`初始化` 是从零开始建立文档体系（全新项目或旧项目全量扫描）。`接手` 是你已经有文档和代码，只是想让 CodeMan 从某个中间阶段继续工作。接手不会重新生成你已有的文档，而是直接读取使用。

**Q：PRD-FULL.md 和碎片文件（feat-*.md）是什么关系？**
A：`feat-*.md` 是 AI 的工作文件，每个功能点一个。`PRD-FULL.md` 是所有功能点的累积完整版，测试通过后自动从碎片同步。你看 `PRD-FULL.md` 就行，不用翻碎片。

**Q：DESIGN-FULL.md 包含什么？**
A：四大章节：架构概述、模块设计、API 设计、数据库设计（表结构、字段、索引、表关系）。同样测试通过后自动同步，不用手动维护。

**Q：碎片文件越来越多怎么办？**
A：每次部署完成后，AI 会扫描过时的碎片文件并建议归档到 `archive/`。你确认后批量归档，工作目录保持干净。总文档（PRD-FULL.md、DESIGN-FULL.md）不受归档影响。

### 协作与项目管理

**Q：多人协作怎么用？**
A：将 `.codeman/` 目录提交到 Git，团队成员各自安装 CodeMan 框架后，打开项目说 `CodeMan 继续` 即可基于同一份文档协作。

**Q：新人接手项目，从哪里开始了解？**
A：打开 `.codeman/docs/PROJECT-OVERVIEW.md`，5 分钟即可了解项目全貌。说 `CodeMan 概览` 可刷新。

**Q：PROJECT-OVERVIEW.md、PRD-FULL.md、DESIGN-FULL.md 有什么区别？**
A：三份文档分工明确：`PRD-FULL.md` 看产品做了什么（所有功能点完整版），`DESIGN-FULL.md` 看技术怎么搭的（架构 + 模块 + API + 数据库），`PROJECT-OVERVIEW.md` 看项目概况（简介、进度、上手指南）。三份都自动维护，不需要手动更新。

**Q：支持哪些 IDE？**
A：支持 **Cursor**、**Claude Code**、**OpenCode** 和 **Trae** 四种 AI IDE。安装时自动检测环境，也可手动选择安装目标。四者可以同时安装，共享同一套 `.codeman/` 项目文档体系。

**Q：多个 IDE 可以在同一个项目里混用吗？**
A：可以。`.codeman/docs/` 文档是环境无关的，所有 IDE 都能读取。项目初始化时会同时生成各 IDE 的引导文件（`.cursor/rules/`、`.claude/rules/`、`.trae/rules/`、`AGENTS.md`），无论用哪个 IDE 开发，规范都会自动生效。

**Q：Claude Code 中如何触发 CodeMan？**
A：两种方式：
1. 直接说命令：`CodeMan 初始化`、`CodeMan 开始开发` 等（Claude Code 会读取 `~/.claude/CLAUDE.md` 中的引导）
2. 使用斜杠命令：`/codeman-orchestrator`（由 `~/.claude/skills/codeman-orchestrator/` 提供，安装脚本会创建指向框架目录的符号链接）

**Q：输入 `/codeman-orchestrator` 提示 Unknown skill？**
A：Claude Code 要求 skill 目录名与 `SKILL.md` 里 `name` 字段一致，且位于 `~/.claude/skills/<name>/`。请重新运行 `bash /path/to/codeman/install.sh`（或 `update.sh`），或手动执行：`bash /path/to/codeman/adapters/claude-code/link-skills.sh ~/.claude/skills/.codeman`。完成后可运行 `/skills` 确认已列出 `codeman-orchestrator`。

**Q：没有设计稿/截图怎么办？**
A：设计资产引用是可选的。需求分析阶段会询问你是否有设计截图，回复"无设计稿"即可跳过，后续基于文字描述开发。截图机制不会阻塞任何流程。

**Q：需求精粹卡是什么？会增加额外成本吗？**
A：精粹卡是需求确认后自动生成的 ≤500 tokens 精简摘要，嵌入在已有的 `DIRECTIVES.md` 中，不新增文件。每个 Skill 本来就会读取 `DIRECTIVES.md`，所以额外成本几乎为零。

**Q：AI 在长对话中还是会忘记之前的约束怎么办？**
A：周期性上下文回顾协议会在每个模块开始前输出结构化回顾摘要（精粹卡约束、领域规则、测试用例、偏离检查），如发现偏离自动暂停。不是敷衍"已回顾"，而是必须列出具体内容。

**Q：测试脚本本身有 bug 怎么办？不会影响测试结果的可信度吗？**
A：三项机制防止这个问题：① **UI 探索前置** — 写脚本前必须先打开页面截图观察实际 UI，严禁基于文档假设编写；② **脚本冒烟验证** — 正式执行前先验证选择器有效性；③ **失败归因流程** — 脚本失败时必须按决策树排查是脚本 bug 还是产品 bug，脚本 bug 修复后重跑不计入报告。

**Q：怎么把第三方 Skill（如 Superpowers）挂到 CodeMan 工作流里？**
A：直接对话即可，如"把 Superpowers 的 brainstorming 挂到需求分析之前"，AI 会引导你完成配置。也可以手动编辑 `.codeman/config.yaml` 的 `external_skills` 段。外部 Skill 保持独立安装和更新，CodeMan 只在运行时通过路径读取。详见「外部 Skill 注册」章节。

**Q：外部 Skill 没安装或路径错了会怎样？**
A：CodeMan 在调度时会检查文件是否存在，不存在则跳过并提示你，不会阻塞工作流。

**Q：用了第三方 API 但 AI 写的代码是旧版用法怎么办？**
A：在 `.codeman/config.yaml` 的 `api_references` 中添加该 API 的官方文档 URL。开发时 AI 会自动抓取最新文档并参照编码。如果忘了配置，编码前 AI 也会检测并提醒你补充。

**Q：同事不用 CodeMan，推了代码后我的文档不同步怎么办？**
A：两种方式：① 自动 — 你执行 `CodeMan 继续` 或 `CodeMan 新需求` 时，会自动检测同事的新提交并补全缺失文档；② 手动 — 如果代码不是通过 git 获得的（比如同事直接给你文件），说 `CodeMan 同步` 手动触发。只补缺失的文档，不会覆盖你已有的。

**Q：什么是领域规则？和编码规范有什么区别？**
A：领域规则记录的是"代码和测试无法表达的业务隐性知识"，如"VIP 等级只升不降"、"订单状态只能单向流转"。编码规范（`.codeman/rules/`）是通用的代码写法约束，如"函数 ≤50 行"。两者互补：领域规则告诉 AI "业务上不能怎样"，编码规范告诉 AI "代码上应该怎样"。

**Q：领域规则文件会不会越来越大？**
A：不会。全局规则文件限制 ≤1500 字，模块级规则跟着模块文件走（只在开发该模块时加载）。已被代码校验或测试覆盖的规则会在模块完成时提示"毕业"移除，反复出现的踩坑会提示"升级"为编码规范。规则总量是自然收敛的。

**Q：测试报告里的描述太笼统看不懂怎么办？**
A：强制要求所有描述包含「定位 + 实际值 + 期望值 + 来源」四要素，并附 10 类反面示例对照表（如 ❌"按钮样式不对" → ✅"提交按钮圆角 0px，PRD 要求 8px；背景色 #ccc，设计稿要求 #1890ff"）。不达标的描述视为不合格报告。
