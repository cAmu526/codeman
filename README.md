# CodeMan v0.7

> 全流程开发工作流框架，支持 **Cursor**、**Claude Code** 和 **OpenCode** 三种 AI IDE。用自然语言驱动 AI 完成从需求分析到部署的全链路开发。每个关键阶段含**多次交互确认**，确保需求完整、方案合理、测试充分。

---

## 它能帮你做什么

CodeMan 是一套安装在 Cursor / Claude Code 上的 AI 工作流框架。安装后，你只需在 AI IDE 的对话框里用自然语言下指令，AI 就会自动按照**需求分析 → 技术方案 → 编码实现 → 测试验证 → 部署清单**的完整流程帮你开发。

**核心价值：**

- **文档驱动**：每一步都产出结构化文档（PRD、技术方案、API 设计、测试用例、部署清单），AI 跨会话也不会丢失上下文
- **多次交互确认**：需求分析 3 次确认（功能点→PRD 审阅→精粹卡）、技术方案 3 次确认（模块划分→API 契约→方案细节）、测试用例先生成再确认后执行，每个关键决策都需要你确认
- **确认平衡**：明确的直接做，不确定的才确认。影响小且方案明确 → 直接执行；影响大或有歧义 → 暂停等你决策。不会事事都问你，也不会自作主张
- **质量内建**：编码前后自动对照需求，Review 门禁拦截问题，测试基准来源于 PRD 而非代码
- **需求完整性自检**：PRD 生成后自动输出完整性评分报告（用户故事、验收标准、边界场景、数据字典等 10 项检查）
- **UI 设计对齐**：PRD 中可嵌入设计截图引用，编码前强制读取截图、测试时视觉对照、Review 时截图比对，杜绝前端实现与设计稿不一致
- **需求不丢失**：需求确认后自动生成"需求精粹卡"（≤500 tokens），跨所有阶段携带，防止中间文档过多导致需求细节丢失
- **上下文不遗忘**：周期性上下文回顾协议，每个模块开发完成后和每次阶段转换时自动重读核心约束，防止长对话遗忘
- **不自作主张**：编码时遇到不确定即停，PRD 未提及的逻辑必须标注 `[UNCERTAIN]` 等待确认，复杂任务强制拆分子步骤逐步确认
- **断点续做**：关掉 IDE 再打开，说一句 `CodeMan 继续` 就能从上次的进度接着干
- **新旧项目通吃**：全新项目从零开始，已有代码库也能快速接入

## 快速开始

### 第一步：安装（一次性，约 10 秒）

```bash
bash /path/to/codeman/install.sh
```

> 将 `/path/to/codeman/` 替换为你实际存放 `install.sh` 的路径。

安装脚本会自动检测你已安装的 IDE，并按需安装：

| 环境 | 安装位置 | Bootstrap 方式 |
|------|---------|--------------|
| Cursor | `~/.cursor/skills/.codeman/` | `~/.cursor/rules/codeman-bootstrap.mdc`（alwaysApply） |
| Claude Code | `~/.claude/skills/.codeman/` | `~/.claude/CLAUDE.md`（自动注入片段） |
| OpenCode | `~/.claude/skills/.codeman/`（兼容层） | `~/.config/opencode/AGENTS.md`（自动注入片段） |

如果多个 IDE 都已安装，脚本会询问你安装到哪个（或全部安装）。

**Cursor 用户：安装完成后重启 Cursor。**

### 第二步：在项目中初始化

用 Cursor 或 Claude Code 打开你的项目，说：

```
CodeMan 初始化
```

AI 会自动运行初始化脚本，交互式询问你的项目信息（项目名、技术栈、前后端框架等），然后在项目根目录生成 `.codeman/` 文档体系。

**初始化会根据项目情况走不同路径：**

| 项目情况 | AI 的处理方式 |
|---------|-------------|
| **全新项目**（目录里没有业务代码） | 自动进入需求分析，引导你梳理功能点，生成 PRD |
| **已有代码库 — 完整接入** | 扫描全部代码，反向提炼 PRD + 技术方案 + API 文档（适合 ≤20 个模块的中小项目，耗时 1-3 小时） |
| **已有代码库 — 骨架接入** | 只读目录结构和配置文件，5-10 分钟生成骨架，后续开发时自动按需补全文档（推荐大型项目使用） |

### 第三步：开始开发

初始化完成后，根据你的需要说不同的命令：

```
CodeMan 开始开发
```

AI 就会自动走完整开发流程。关键节点会暂停等你确认，不会自作主张。

## 所有命令一览

| 命令 | 什么时候用 | AI 会做什么 |
|------|-----------|------------|
| `CodeMan 初始化` | 项目第一次使用 CodeMan | 生成 `.codeman/` 文档体系，检测技术栈，安装编码规范 |
| `CodeMan 开始开发` | 初始化后启动开发 | 按流程走完：需求 → 方案 → 编码 → 测试 → 部署 |
| `CodeMan 新需求：[描述]` | 需要加新功能或改需求 | 增量需求分析 + 影响评估，然后走开发流程 |
| `CodeMan 继续` | 上次没做完，接着干 | 读取进度文件，从断点处继续 |
| `CodeMan 修复：[描述]` | 修 Bug 或小改动 | 定位 → 修复 → 回归测试 → 文档同步（3-10 分钟搞定） |
| `CodeMan 状态` | 想看看当前进度 | 展示当前阶段、已完成/待完成的功能点、遗留问题 |
| `CodeMan 概览` | 查看/更新项目概览 | 汇总所有碎片文档，生成面向新成员的一站式项目概览 |

## 完整工作流详解

说完命令后，AI 自动识别场景并按以下流程执行：

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 需求分析 │ → │ 技术方案 │ → │ 开发实现 │ → │ 测试验证 │ → │ 部署清单 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
                    ↓               ↓              ↓
               Review 门禁     Review 门禁    失败 → 修复闭环
                                               (最多 3 轮自动修复)
```

### 1. 需求分析

AI 通过**八维**头脑风暴帮你拓展需求，然后收敛为结构化的 PRD 功能点碎片文件。含**三次交互确认**。

- **输入**：你的需求描述（一句话或一段话都行）
- **过程**：八维拓展 → 红/黄/绿分类 → **【确认①：功能点分类】** → 收集 UI 设计截图 → 生成 PRD → **【确认②：PRD 内容审阅 + 完整性校验】** → 生成需求精粹卡 → **【确认③：精粹卡确认】**
- **PRD 质量（行业标准）**：每个功能点必含用户故事、业务功能流程图、ASCII 界面原型、Given-When-Then 验收标准、数据字典、非功能需求、假设与约束
- **完整性自检**：PRD 生成后自动输出 10 项检查评分（用户故事、流程图覆盖、验收标准可量化性、边界场景、数据字典等）
- **产出**：`.codeman/docs/prd/feat-*.md`（每个功能点一个文件）+ 需求精粹卡（`DIRECTIVES.md`，≤500 tokens）

### 2. 技术方案

基于确认的 PRD，AI 做模块拆分和技术方案设计。含**三次交互确认**。

- **输入**：PRD 功能点 + 项目配置 + 需求精粹卡
- **过程**：模块边界划分 → **【确认①：模块划分】** → 每模块技术选型与数据模型 → API 接口设计 → **【确认②：API 契约审阅（逐接口）】** → 自动 Review 门禁 → **【确认③：方案细节审阅（数据模型/核心逻辑/决策）】**
- **精粹卡对照**：设计前加载需求精粹卡，确保方案覆盖所有核心功能和绝对约束
- **版本迭代影响面扫描**：新需求/修改时，Grep 扫描代码中所有引用点（数据模型、API、枚举、组件等），展示直接引用 + 间接影响 + 前端页面影响，用户确认哪些纳入本次迭代后再继续设计
- **产出**：`.codeman/docs/design/mod-*.md` + `.codeman/docs/api/api-*.md`

### 3. 开发实现

按技术方案逐模块编码，编码前后强制对照需求。每个模块开始前自动执行上下文回顾。

- **上下文回顾**：每个模块开始前重读 DIRECTIVES.md，输出回顾摘要，防止长对话遗忘核心约束
- **编码前**：从技术方案（`mod-*.md` + `api-*.md`）机械化提取细粒度 Checklist（每个 API 端点、每个数据模型字段、每个设计决策），连同 PRD + UI + 设计截图一起输出对照清单，你确认后才开始
- **编码中**：禁止凭推测补充 PRD 未提及的逻辑，遇到歧义暂停问你。如需添加 PRD 未提及的功能必须标注 `[UNCERTAIN: PRD 未提及]`
- **变更影响面扫描**：修改现有代码时，自动识别 10 类变更类型（枚举/API 契约/组件签名/Schema/权限/路由/配置/业务规则/状态机/错误码），全局搜索所有引用点，逐一同步修改，修改后执行一致性验证（类型检查 + exhaustive check + 单元测试）
- **编码后**：**机械化 Grep 比对**（技术方案 API 列表 vs 代码路由定义、数据模型 vs 类型定义、设计决策 vs 代码实现，产出结构化 diff 表）→ 需求实现对照表 + 跨功能一致性自检 → **浏览器冒烟验证**（有 UI 的模块用 Playwright 走核心路径截图）→ **自动 Review 门禁**（技术方案逐项对照输出实现率）→ Git commit
- **复杂任务**：满足复杂条件（3+ 模块联动、嵌套 ≥ 3 层等）时强制拆分子步骤，逐步确认后执行
- **同步产出**：单元测试（覆盖率 ≥ 80%）

### 4. 测试验证

**先根据 PRD 生成测试用例和自动化脚本，再逐层自动化执行。** 测试基准来源于 PRD（不是代码），能自动化的一律自动化，有 UI 的功能一律用 E2E 浏览器测试。

- **启动判断**：自动检测是否已有测试报告。全新测试走完整流程；续测模式先审计已有结果（虚假通过降级），再增量执行未验证的用例
- **阶段一：用例生成 + 脚本编写**：从**三类来源**推导测试用例 — ① PRD 验收标准和边界场景 ② 技术方案设计决策（缓存策略、错误码、中间件行为等）③ 代码扫描（Grep 发现实现了但文档未提到的 API）。隐性需求自动推导（8 类场景）+ 代码分支覆盖辅助。用户确认前自动执行**五项质量校验**：完整性校验、浅测试检测、覆盖率前置校验（含代码扫描交叉比对）、技术方案覆盖校验、代码分支覆盖分析。**验收标准理解确认**：逐条输出 AI 对每条验收标准的理解和验证方式，用户确认后才写入脚本
- **阶段二：逐层执行（门禁自检）**：代码审查预检 → L1 → L2a → L2b → L3 → L3 网络异常 → L4 → 兼容性 → L5，每层之间有强制门禁自检，防止跨层级标记通过

| 层级 | 测试类型 | 自动化方式 |
|------|---------|-----------|
| 代码审查 | 预检 | **全栈强制覆盖**（7 类文件缺一不可）+ 硬编码检查 + 前后端参数一致性 + **跨功能一致性检查**（枚举/API/Schema 等共享定义变更时自动检测所有引用方是否同步） |
| L1 | 单元测试 | 编码时已同步生成，此阶段自动验证并补充 |
| L2a | DB 直查 | 直连数据库验证表结构、迁移、初始数据（无需 API 认证） |
| L2b | API 集成 | API 测试脚本 + **响应数据格式全面校验**（字段类型、枚举值、嵌套结构、数组格式，基于 API/技术文档而非代码） |
| L3 | E2E 端到端 | **强制 UI 探索 → 脚本冒烟 → 正式执行**三步流程 + **Playwright 浏览器自动化** + **对照 PRD 原型图校验布局结构和交互逻辑**（含 8 类前端场景 + 4 类网络异常场景 + 3 类组件标准化验证模板：下拉框/列表页/表单完整流程） |
| L4 | UI 视觉测试 | Midscene.js 自然语言断言 + Playwright 截图对比 + **逐项对照设计稿校验样式** |
| 兼容性 | 多浏览器 + 多设备 | L3/L4 在 Chromium/Firefox/WebKit + iPhone/Android/iPad 上复跑（默认 Chromium，执行前确认范围，支持 BrowserStack 真机增强） |
| L5 | 非功能测试 | **性能**（Lighthouse 逐页面 + k6 三场景压测 + 内存泄漏检测）+ **安全**（OWASP 自动化检测 + 手动建议清单）+ **可访问性**（axe-core WCAG AA 逐页面扫描），各子项独立开关，执行前逐项确认 |

**防虚假通过机制：** 每个用例必须有实际执行证据（脚本输出 / API 响应体 / 截图）才能标 ✅，严禁凭代码审查推断通过。标记前强制自检 7 项 checklist（含描述质量校验和 UI 实际页面验证），续测模式自动审计降级无证据的 ✅。**描述质量最低标准**：所有报告描述必须包含「定位 + 实际值 + 期望值 + 来源」四要素，附 10 类笼统 vs 具体反面示例对照表，禁止"样式不对""接口异常"等模糊描述。**API ≠ 页面规则**：涉及 UI 的功能即使 API 测试通过，仍必须通过 Playwright 实际打开页面验证渲染结果，附 6 类"API 正确但页面出错"的典型踩坑场景。

**脚本可信性保障：** E2E 脚本编写前强制执行 UI 探索（先用 Playwright 打开页面截图观察实际 UI，严禁直接基于 PRD 假设写脚本）；脚本写完后先冒烟验证选择器有效性；脚本失败时必须执行归因流程（区分脚本 bug vs 产品 bug，脚本 bug 修复后重跑不计入报告）。

**测试校验来源原则：** API 数据格式的预期来自技术文档/API 文档（严禁从代码反推）；UI 布局和交互的预期来自 PRD ASCII 原型图、UI 设计要求、设计截图/Figma（多来源冲突时暂停确认）。

**回归测试策略：** 三级回归（Smoke 3-5 分钟 / Regression 10-20 分钟 / Full 30-60 分钟），每次修改后提醒触发。影响分析**三层机制**：路径规则匹配 + AI 补充判断 + **跨功能依赖追踪**（Grep 共享定义的引用方，自动追加引用方的测试用例）。可选 CI/CD 集成（GitHub Actions / GitLab CI）。

测试完成后额外执行精粹卡验收标准交叉校验，确保每条关键验收标准都有 E2E 覆盖。

测试全部通过 → 进入部署清单；有失败 → 自动进入修复闭环（最多 3 轮）。

### 5. 修复闭环

测试失败时自动触发，也可手动用 `CodeMan 修复：[描述]` 触发。

- 修复前自动执行上下文回顾，对照需求精粹卡确保修复方向不偏离核心约束
- 根因分析 → 影响评估 → 修复 → 回归验证（含 P0 功能点扩大回归）→ **文档同步**
- 最多自动修复 3 轮，修不好会停下来让你介入

**文档同步规则**（根据根因类型自动处理）：

| 根因类型 | 同步哪些文档 | 说明 |
|---------|------------|------|
| 实现 Bug（代码逻辑错误） | 测试报告 | PRD 和技术方案不用动 |
| 设计缺陷（技术方案有问题） | `.codeman/docs/design/mod-*.md` | 更新技术方案，记录变更原因 |
| PRD 歧义（需求不明确） | `.codeman/docs/prd/feat-*.md` + DIRECTIVES.md | 明确业务规则和约束 |

不会只改代码不更新文档。

**缺陷回溯学习**：每个产品 Bug 修复后强制回溯分析（需求→设计→开发→Review→测试，哪个环节应该拦住但没拦住），将改进建议沉淀为规则（如新增编码规则、Review 审查维度、测试推导场景），防止同类问题重复出现。

### 6. 部署清单

所有测试通过后自动生成，包含：环境变量、数据库迁移、依赖变更、部署步骤、API 变更说明、风险与回滚方案。

### 7. 自进化（自动触发）

Review 中发现同类问题出现 ≥ 2 次时，AI 会自动草拟新的编码规范（Rule）或工作流技能（Skill），经你确认后写入项目，后续开发自动遵守。

## 质量保障机制

v0.5 引入三大机制，v0.6 新增测试可信性保障，解决 AI 辅助开发中的常见痛点：

### UI 设计资产引用系统

**解决问题：** 前端实现与 UI 原型图/设计稿不一致。

PRD 模板中新增"设计资产引用"表格，支持在每个功能点中引用设计截图文件路径。引用会贯穿整个开发链路：

| 阶段 | 如何使用截图 |
|------|------------|
| 需求分析 | 收集设计截图，写入 PRD 的设计资产引用表 |
| 开发实现 | 编码前强制读取截图文件（利用 AI 多模态能力），理解视觉要求后再编码 |
| 测试验证 | L4 UI 视觉测试时，读取设计截图与实际页面截图逐项对比 |
| Review | 代码审查新增第⑥维度"UI 设计一致性"，对比截图与前端实现 |

截图文件建议存放在 `.codeman/assets/design/` 目录。如无设计稿，标注"待设计稿确认"即可，不阻塞流程。

### 需求精粹卡

**解决问题：** 中间文档过多（PRD → 设计 → 代码 → 测试），跨阶段传递时需求细节丢失。

需求确认后自动生成一张 ≤500 tokens 的"需求精粹卡"，存入 `DIRECTIVES.md`，内容包含：

- **产品目标**（一句话）
- **核心功能清单**（每个 P0/P1 功能点一行）
- **绝对约束**（不可违反的硬约束）
- **关键验收标准**（最关键的 3-5 条可量化条件）
- **设计资产摘要**（截图路径，如有）

精粹卡随 `DIRECTIVES.md` 被每个 Skill 自动读取，无需额外文件管理。版本迭代时同步更新。

### 周期性上下文回顾协议

**解决问题：** 长对话中 AI 遗忘早期的关键约束和指令。

在两个时机强制执行上下文回顾：

1. **每个模块开发完成后**（多模块迭代循环的第 0 步）
2. **每次阶段转换时**（Orchestrator 调度下一个 Skill 之前）

回顾动作：重新读取 `DIRECTIVES.md`（含需求精粹卡），输出一行回顾摘要。如发现当前实现方向与核心约束存在偏离，立即暂停并报告。

### 测试可信性保障（v0.6）

**解决问题：** 测试脚本本身有 bug 导致结果不可信；测试报告描述笼统无法定位问题；只跑 API 不验页面漏过前端缺陷。

通过四项机制形成闭环，从源头保障测试结果可信：

| 机制 | 解决什么问题 | 核心做法 |
|------|------------|---------|
| **UI 探索前置** | 脚本基于 PRD 假设编写，选择器/操作路径可能与实际 UI 不符 | 写脚本前必须先 Playwright 打开页面截图，观察实际 UI 结构后再编写 |
| **脚本冒烟验证** | 脚本写完直接批量跑，一旦有基础问题则大面积失败 | 正式执行前先验证选择器有效性、页面可加载、前置操作可达 |
| **失败归因流程** | 脚本 bug 被误报为产品 bug，浪费开发时间 | 失败时按决策树排查（选择器/断言/运行错误），区分脚本 bug 与产品 bug |
| **描述质量标准** | 报告写"样式不对""接口异常"无法定位问题 | 所有描述必须包含「定位 + 实际值 + 期望值 + 来源」四要素，附 10 类反面示例 |

### 机械化比对（v0.7）

**解决问题：** AI 自己写代码、自己检查，盲区一样 — 漏掉的编码时漏，自检时也漏。

用结构化 Grep diff 替代 AI 主观判断，在三个环节强制执行：

| 环节 | 比对内容 | 做法 |
|------|---------|------|
| **开发后置检查** | 技术方案 vs 实际代码 | Grep 代码路由定义、类型定义、关键实现，与技术方案逐项 diff |
| **测试覆盖校验** | 实际 API vs 测试覆盖 | Grep 代码中所有路由定义 → 与测试文件中的请求 URL 交叉比对 |
| **Review 审查** | 技术方案实现率 | 逐项 Grep 验证，输出结构化实现率报告（而非笼统"基本一致"） |

### 缺陷回溯学习（v0.7）

**解决问题：** 同类 Bug 反复出现，流程没有学习能力。

每个产品 Bug 修复后强制回溯到源头阶段（需求→设计→开发→Review→测试），分析哪个环节应该拦住但没拦住，将改进建议自动沉淀为对应 Skill 的规则或项目编码规范，堵住同类漏洞。

## 升级与卸载

### 升级框架

在 CodeMan **源码目录**下执行（不是安装目录）：

```bash
bash /path/to/codeman/update.sh
```

升级只更新框架文件，不会覆盖各项目的 `.codeman/docs/` 文档和项目级规范。脚本会自动检测已安装的环境并同时升级。Cursor 用户升级后需重启 Cursor。

### 全局卸载

彻底移除 CodeMan 全局框架（同时清理 Cursor 和 Claude Code 两个环境）：

```bash
bash /path/to/codeman/uninstall.sh
```

卸载后重启 Cursor 生效。各业务项目中的 `.codeman/` 目录不受影响。

### 项目反初始化

只清理当前项目的 CodeMan 配置（规范、模板），保留 `.codeman/docs/` 文档资产：

```bash
# 在项目根目录执行（同时清理 Cursor 和 Claude Code 两个环境的项目级配置）
bash /path/to/codeman/uninit.sh
```

## 常见问题

**Q：说了命令但 AI 没反应？**
A：确认已重启 Cursor。检查 `~/.cursor/rules/codeman-bootstrap.mdc` 文件是否存在。

**Q：换了一个 Cursor 会话，AI 还记得之前的进度吗？**
A：记得。所有进度保存在项目的 `.codeman/docs/STATUS.md` 中，新会话说 `CodeMan 继续` 即可恢复。

**Q：可以跳过某个阶段吗？**
A：流程中的关键节点都会暂停确认，你可以在确认时告诉 AI 跳过或调整。同时遵循「确认平衡」原则：明确的常规操作直接执行不打扰你，不确定的、有歧义的、高风险的才暂停确认。轻量修复场景（`CodeMan 修复`）会自动跳过不必要的阶段。

**Q：多人协作怎么用？**
A：将 `.codeman/` 目录提交到 Git，团队成员各自安装 CodeMan 框架后，打开项目说 `CodeMan 继续` 即可基于同一份文档协作。

**Q：新人接手项目，从哪里开始了解？**
A：打开 `.codeman/docs/PROJECT-OVERVIEW.md`，5 分钟即可了解项目全貌（业务目标、技术架构、功能地图、模块结构、上手指南）。如果文档不是最新的，在 Cursor 中说 `CodeMan 概览` 即可刷新。

**Q：PROJECT-OVERVIEW.md 和其他文档是什么关系？**
A：`PROJECT-OVERVIEW.md` 是面向人类的「内容聚合文档」，从各碎片文档（PRD、技术方案、API 等）中**内联核心摘要**（如功能需求、模块设计、API 字段、决策原因等），同时附有详情链接供深入阅读。它在每个开发阶段完成后自动更新，不需要手动维护。

**Q：支持哪些 IDE？**
A：支持 **Cursor**、**Claude Code** 和 **OpenCode** 三种 AI IDE。安装时自动检测环境，也可手动选择安装目标。三者可以同时安装，共享同一套 `.codeman/` 项目文档体系。

**Q：Cursor、Claude Code 和 OpenCode 可以在同一个项目里混用吗？**
A：可以。`.codeman/docs/` 文档是环境无关的，三个 IDE 都能读取。项目初始化时会同时生成 `.cursor/rules/`、`.claude/rules/` 和 `AGENTS.md` 三套引导，无论用哪个 IDE 开发，规范都会自动生效。

**Q：Claude Code 中如何触发 CodeMan？**
A：两种方式：
1. 直接说命令：`CodeMan 初始化`、`CodeMan 开始开发` 等（Claude Code 会读取 `~/.claude/CLAUDE.md` 中的引导）
2. 使用斜杠命令：`/codeman-orchestrator`（由 `~/.claude/skills/codeman-orchestrator/` 提供，安装脚本会创建指向框架目录的符号链接）

**Q：输入 `/codeman-orchestrator` 提示 Unknown skill？**
A：Claude Code 要求 skill 目录名与 `SKILL.md` 里 `name` 字段一致，且位于 `~/.claude/skills/<name>/`。请重新运行 `bash /path/to/codeman/install.sh`（或 `update.sh`），或手动执行：`bash /path/to/codeman/adapters/claude-code/link-skills.sh ~/.claude/skills/.codeman`。完成后可运行 `/skills` 确认已列出 `codeman-orchestrator`。

**Q：没有设计稿/截图怎么办？**
A：设计资产引用是可选的。需求分析阶段会询问你是否有设计截图，回复"无设计稿"即可跳过，后续基于文字描述开发。截图机制不会阻塞任何流程。

**Q：需求精粹卡是什么？会增加额外成本吗？**
A：精粹卡是需求确认后自动生成的 ≤500 tokens 精简摘要，嵌入在已有的 `DIRECTIVES.md` 中，不新增文件。每个 Skill 本来就会读取 `DIRECTIVES.md`，所以额外成本几乎为零。

**Q：AI 在长对话中还是会忘记之前的约束怎么办？**
A：周期性上下文回顾协议（v0.5 引入）会在每个模块边界和每次阶段转换时强制重读核心约束，输出回顾摘要。如果发现偏离会自动暂停。这比依赖 AI 自觉记忆更可靠。

**Q：测试脚本本身有 bug 怎么办？不会影响测试结果的可信度吗？**
A：v0.6 新增三项机制防止这个问题：① **UI 探索前置** — 写脚本前必须先打开页面截图观察实际 UI，严禁基于文档假设编写；② **脚本冒烟验证** — 正式执行前先验证选择器有效性；③ **失败归因流程** — 脚本失败时必须按决策树排查是脚本 bug 还是产品 bug，脚本 bug 修复后重跑不计入报告。

**Q：测试报告里的描述太笼统看不懂怎么办？**
A：v0.6 强制要求所有描述包含「定位 + 实际值 + 期望值 + 来源」四要素，并附 10 类反面示例对照表（如 ❌"按钮样式不对" → ✅"提交按钮圆角 0px，PRD 要求 8px；背景色 #ccc，设计稿要求 #1890ff"）。不达标的描述视为不合格报告。

## 版本历史

- **v0.7** — 2026-04-06：四层质量保障体系（解决 AI 实现偏差和漏测两大痛点）：
  - **机械化比对**：开发后置检查从 AI 自查改为 Grep diff（API 端点、数据模型字段、设计决策逐项比对），Review 技术方案一致性改为结构化逐项对照输出实现率
  - **技术方案细粒度 Checklist**：编码前从 mod-*.md 和 api-*.md 机械化提取每个 API 端点、每个字段、每个设计决策，生成可打勾的实现清单
  - **浏览器冒烟验证**：有 UI 的模块编码后用 Playwright 启动浏览器走核心路径截图，开发者视角确认"能跑通"
  - **测试三来源**：用例从 PRD + 技术方案 + 代码扫描三类来源推导；新增技术方案设计决策覆盖校验和代码分支覆盖分析
