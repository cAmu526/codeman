# CodeMan 项目开发规则

> 本文件是 CodeMan 框架项目自身的开发约束。AI 在修改本项目时必须遵守。

## 强制规则：每次修改必须更新 README

**每次对本项目的任何修改，都必须同步更新 README.md 中的相关内容。**

具体要求：

1. **新增文件/目录** → 更新 README 中的"框架目录结构"和"项目中会生成什么"章节
2. **新增/修改 Skill** → 更新 README 中的"完整工作流详解"和"所有命令一览"章节
3. **新增/修改规则** → 更新 README 中的"内置编码规范"章节
4. **新增/修改适配器** → 更新 README 中的"框架目录结构"、安装说明和 FAQ 章节
5. **新增/修改模板** → 更新 README 中相关章节的描述
6. **修改工作流逻辑** → 更新 README 中的"完整工作流详解"章节
7. **版本号变更** → 更新 README 顶部的版本号和"版本历史"章节
8. **新增配置项** → 更新 README 中相关配置说明

**检查清单（每次 commit 前必须自检）：**

- [ ] README.md 是否反映了本次修改的所有变更？
- [ ] 目录结构章节是否与实际文件一致？
- [ ] 版本号是否需要更新？
- [ ] FAQ 中是否有需要新增或更新的问题？

**违反后果：** 如果修改了代码/配置但未更新 README，视为不完整的提交。

## 交互规则

- **修改任何文件前，必须先展示修改方案让用户确认，确认后才执行编辑**
- 紧急修复（如 typo）也要先说一声

## 代码同步规则

- 修改 install.sh 中的 bootstrap 模板 → 必须同步修改 update.sh 中的对应模板
- 修改任何 Skill 文件 → 必须在源码目录（skills/）修改，不要改安装目录（~/.claude/skills/.codeman/）
- 修改完后通过 update.sh 同步到安装目录
- 新增 config.yaml 配置项 → 必须同时更新三处：`templates/config.yaml`、`init.sh` heredoc、`skills/orchestrator/SKILL.md` 迁移表（codeman_config_version +1）

## 反臃肿规则（修改 skills/rules/README 前必读）

CodeMan 的核心资产是 markdown skills，AI 是消费者。规则越多 → AI 注意力衰减越严重，最终所有规则都被忽略。
**加规则前先问：不加会怎样？如果答案是"只有理论风险、没有真实踩坑"，就不要加。**

修改前必须通过以下六道门禁：

### 1. 先复用，再新增

- 新增规则前必须 Grep 现有 skill 文件，确认无主题重叠
- 同主题已有规则 → 合并/改写，不另开段落
- 反例：在 `testing/SKILL.md` 里再加一段"防虚假通过" —— 应合并到已有那段

### 2. 行数软上限

| 文件类型 | 软上限 | 超过后的动作 |
|---------|-------|------------|
| `skills/*/SKILL.md` | 1000 行 | 新增前先砍同等行数的冗余 |
| `skills/*/phase*.md` | 1200 行 | 考虑拆子文件 |
| `README.md` | 1100 行 | 合并重复描述，压缩版本历史 |

### 3. 具体案例驱动

- 新增规则必须能指出**真实踩坑场景**（最好带一句反例引文）
- "理论上可能发生" / "未来可能出现" 不是触发条件
- 推荐格式：`> **反例（本规则诞生的源头）**：某次 XXX ——` ——已在 `skills/testing/SKILL.md` 使用

### 4. 优先删除/重写，而非新增

- 面对"规则是否覆盖 Y 场景"的问题，先问：**能否重写现有规则让它同时覆盖 Y？**
- 只有答案是"不能且强加会扭曲原义"时才新增
- 删除 10 行 + 改写 5 行 优于 新增 15 行

### 5. Checklist 克制

- 单个检查点的 checklist 不超过 7 项；超过 → AI 会跳着读，等于没加
- 必须全做的用枚举，可跳过的别进 checklist
- 反例：24 项的"编码前全栈检查清单" —— AI 只会认真看前 5 项

### 6. 可选 vs 强制

- 新机制默认标"可选"（通过 `config.yaml` 开关控制），验证有效且高频再升级为强制
- 小众场景的机制强上全员 = 对 80% 项目的噪音

### 自检清单（每次 commit 前必做）

- [ ] 本次新增的规则/段落，是否已在现有文件中有类似表达？
- [ ] 新增部分有对应的真实踩坑案例（而非理论风险）？
- [ ] 修改后的单文件行数未超过软上限？
- [ ] 新 checklist 项目数 ≤ 7？
- [ ] 如果"只在特定场景有用"，是否做成了可选而非强制？

**违反后果：** 加了只在理论上有用的规则 = 对后续所有 AI 执行造成持续噪音，视为负优化。

## 其他项目约束

- 本项目使用 Bash + Python3 编写，不引入额外依赖
- Shell 脚本必须兼容 macOS bash 3.x+ 和 Linux bash 4.x+
- Skills 的 frontmatter `name` 字段必须与目录名一致
- 规则文件使用 `.mdc` 作为源格式，各 IDE 适配器负责转换为各自的格式（Cursor 用 `.mdc`，Claude Code 用 `.md`，OpenCode 用 `instructions` 引用）
